Balance Automation

A closed loop that turns brute-force simulation into the thing that tunes the game — so balancing and feature-gating stop being a manual, play-to-verify grind. Built on Hearthlands' existing headless playtest harness.

What this is

Hearthlands is a deep game with a mature, production-grade headless simulator (tools/playtest/, src/playtest/) that plays the real reducer deterministically and already proposes balance changes. What it lacked was the loop around it: nothing applied the proposals, nothing searched for good values, and the sim's bot played nothing like a real player.

This system closes that loop. The bottleneck on finishing the game was never writing code — it was that balancing and feature-gating were manual: hand-edit a number, play to check, repeat; hand-wire an unlock gate, play to verify it's reachable. That is exactly the work brute-force simulation is built for.

TL;DR. run the sim → it proposes edits → the optimizer searches for values that hit a goal → a codemod writes them back into constants.ts → re-sim. Balance is evaluated across a floor↔ceiling bracket of how differently players play, so the numbers transfer to real humans — not to a naive bot. Every auto-tune is a reviewable git diff.

The validity principle

The load-bearing idea, and the reason this isn't just "auto-cut some numbers":

Sim-derived balance is only as trustworthy as the player policy the sim runs. A bot that always grabs the longest chain and never builds or crafts will make the optimizer tune the game for a player who doesn't exist.

So we don't chase one "true" player. We bracket play between a floor (naive: longest-chain, greedy tier-up, never builds or crafts) and a ceiling (near-optimal: need-aware chains + a build→craft→tier macro). Real players live between. Balance is healthy when both ends are acceptable — the floor isn't soft-locked and the ceiling can't trivialize the game. The spread between them is honest uncertainty, far better than one bot's point estimate.

Floor agent

greedy board policy + floor macro. Models a player who never builds or crafts. On home it permanently stalls at tier 0 on crafted bread.

Ceiling agent

climb board policy + climb macro. Builds the Bakery, crafts bread from farm flour+eggs, climbs to Hamlet (tier 1+). ~55% higher coins/run.

The closed loop

Five stages, one cycle. The catalog (constants.ts) is the single source of truth; everything reads it, the codemod writes it.

Why a loop, not a script. Each stage is independently usable (audit the economy, compare the bracket, just apply a hand-written patch). Chained, they let the game tune itself toward a goal you set, while you review the diff at the end.

M1 — Close the apply loop

The sim and the Dev Panel both emit the same machine-readable patch — a flat map of dotted constants paths to new numbers:

{ "ITEMS.pearls.value": 101, "RECIPES.rec_bread.inputs.flour": 2,
  "BUILDINGS.mill.cost.plank": 6 }

Until M1 that was a dead end: a human pasted it into an LLM to hand-edit the file. Now src/playtest/applyPatch.ts parses constants.ts with the TypeScript compiler API, walks each path down to its numeric literal, and rewrites only that literal by text-span splice. It does not re-print the AST — every other byte (formatting, comments, trailing commas) is preserved, so the only diff is the changed numbers.

  • Handles the real shapes: the ITEMS→ITEMS_DATA spread aliases, and array-by-id descent for BUILDINGS.
  • Fails loud: any path that doesn't resolve to a numeric literal aborts the whole apply — a silent half-write is the one outcome we never want.
  • Self-healing snapshot: a balance edit moves the fixed-seed metrics by design, so --apply regenerates the harness drift-guard snapshot — that snapshot diff is the reviewable record.

M2 — A faithful player bracket

Three pieces give the sim a real player model instead of a longest-chain bot.

M2a · State-aware board policies

The per-chain policy widened from "see only the chain list" to a full PolicyContext (live state, zone, board, RNG). Policies: greedy (longest — the floor), value (coin-chaser), and climb / needAwareClimb — the first policy that reads game state to steer chain choice toward the next tier rung's shortfalls.

M2b · The macro-decision layer

What actually paces progression isn't the per-chain choice — it's the between-run decisions: what you build, what you craft, when you tier up. src/playtest/macro.ts models them with real reducer actions (BUILD / CRAFTING/CRAFT_RECIPE / TIER_UP):

  • floorMacro — greedy tier-up only (the original behavior; default, so existing snapshots are byte-identical).
  • climbMacro — each pass: build the crafting stations the next rung needs, craft those goods up to the requirement, then tier up — to a fixpoint.
Result. The documented "home stalls at tier 0 on crafted bread" finding is a floor artifact: the ceiling macro builds the Bakery and crafts bread from farm-produced flour+eggs, clearing the stall and climbing to Hamlet. That's a real floor↔ceiling gap on progression, not just per-run economy.

M2d · The comparison harness

src/playtest/compare.ts runs the same seeds under each agent and tabulates the spread. npm run playtest -- --compare:

floor    [greedy/floor] tier 0 · net/run 37.6 · stall Hamlet: bread×6
ceiling  [climb/climb]  tier 1 · net/run 58.4 · stall Village
progression spread: tier 0 → 1 (Δ1)

M3 — The target-seeking optimizer

src/playtest/optimizer.ts is the framework that ties it together. Given a set of knobs (each a live get/set onto a constants value with bounds) and a loss to minimize, it runs coordinate descent — sample candidate values per knob, keep the one that most reduces the loss, repeat — then emits the winning values as a change-list for M1 to apply.

Non-destructive

It evaluates each candidate by mutating the in-memory catalog, then always restores the originals. Nothing changes on disk — it returns a proposal. Disk edits happen later, reviewably, via playtest:apply.

Goal-driven

The loss is supplied by the caller, never invented here. The bundled spreadObjective targets the objective family-value-spread audit; pacing/economy goals plug into the same optimize() once their target bands are set.

The worked example needs no design input: drive every high family-value outlier under the ceiling factor × median (no resource should pay more than 3× a typical tile-spend). Live run:

$ npm run playtest -- --optimize
loss 329.33 → 0 over 1 pass · ✓ in band
proposed 7 edit(s):  ITEMS.jade.value → 68 · ITEMS.pearls.value → 101 · …
Pacing targets are yours to set. An optimizer is only as good as what it optimizes toward, and a pacing goal ("reaching Hamlet should take ~N runs") is a design decision, not something the tool should invent. The current PROGRESSION_TARGETS bands are explicit placeholders. The framework is ready; point it at a real number when you have one. See Balancing Town 1 for the exact fields and the run-it-later loop.

The end-to-end loop, by hand

The whole point — tune toward a goal and review the diff:

# 1. See where the economy stands + get a proposal
npm run playtest -- --zones home --runs 5 --seed 1

# 2. (or) let the optimizer search for values that hit the goal
npm run playtest -- --optimize

# 3. apply the proposal back into constants.ts (+ snapshot regen)
npm run playtest:apply

# 4. review — the diff is just the changed numbers
git diff src/constants.ts

# 5. re-validate across the bracket
npm run playtest -- --compare --runs 60
Check the bracket, not one bot. Before accepting a change, confirm it keeps the floor un-stuck and the ceiling un-trivialized (--compare). A change that fixes one end by breaking the other is a regression the single-policy view would miss.

Balancing Town 1, concretely — what it tunes & where you set it

This is the walkthrough for the most common question: how do I use this to balance finishing Town 1, and where do the numbers live? "Completing Town 1" means climbing home's tier ladder — Outpost → Hamlet → Village → City (tiers 0→3). Each rung has a real cost in src/features/cartography/data.ts; that's what pacing is measured against:

Hamlet   { plank: 8,  hay_bundle: 10, bread: 6 }
Village  { plank: 12, bread: 14, pie: 6, eggs: 8 }
City     { plank: 20, bread: 24, pie: 12, soup: 8 }

How completion is measured

Campaign mode keeps one persistent state and plays many runs in a row; between runs the macro policy dispatches real BUILD / CRAFTING/CRAFT_RECIPE / TIER_UP actions. Its output is the Town-1 completion pacing — runs to reach each tier, plus where it stalls:

npm run playtest -- --campaign --zones home --runs 80 --macro climb

The macro you pick is the bracket end. --macro floor never crafts and stalls at tier 0 on bread×6; --macro climb builds the Bakery, crafts bread from farm flour+eggs, and climbs. Balance Town 1 so both ends land where you want — fast enough that the naive player isn't stuck, slow enough that a min-maxer can't skip it.

The metrics, and where each target lives

Every target is a hand-maintained literal in one file — src/playtest/targets.ts. That's where you "set" the goals:

MetricAnswersSet the target in targets.tsAuto-tuned?
Family-value spread"Does any resource pay ≫ a staple tile?"FAMILY_SPREAD_OUTLIER_FACTOR (e.g. 3)yes --optimize
Coins / run"Is one home run worth a sane amount?"ZONE_COIN_TARGETS[].coinsPerRunmeasured
Runs → coin milestone"How many runs to afford the next settlement?"PROGRESSION_TARGETS[].coinMilestonesmeasured
Runs → tier Town 1 completion"How many runs to reach Hamlet / Village / City?"PROGRESSION_TARGETS[].tierRuns (empty today)wire it

The tierRuns band is the Town-1-completion knob, and it was deliberately left empty in v1 with a note: "add tier bands once the campaign simulates crafting." The climb macro now does — so the band is ready to fill, e.g. tierRuns: [{ tier: 3, name: "City", runs: [30, 55] }].

Measured vs. auto-tuned — be clear-eyed. The campaign measures all four metrics and grades each against its band (within / above / below / unreached). But --optimize today only auto-tunes the spread objective, because that one needs no design call. Pacing — including runs-to-tier — is measured and reported, not yet auto-searched. Closing that is two steps: (1) set a tierRuns band for home (the design call — "how long should Town 1 take?"), and (2) wire a pacing objective (a sibling of spreadObjective) whose loss is the campaign's distance from that band.

Which numbers move Town-1 pacing

All of these are codemod-reachable (M1+M5), so the optimizer can propose them and playtest:apply can write them back:

  • Tier costs — MAP_NODES.home.tiers.<rung>.upgradeCost.resources.<k> (e.g. lower City's bread from 24).
  • Recipe inputs — RECIPES.rec_bread.inputs.flour (cheaper bread → faster every rung that needs it).
  • Tile / item values — ITEMS.<r>.value (raises coins/run, indirectly speeds the coin milestones).
  • Market prices & session length — MARKET_PRICES.<k>.*, board-template baseTurns.

Run it later — the Town-1 loop

# 1. Measure where Town 1 completion stands, at BOTH bracket ends
npm run playtest -- --campaign --zones home --runs 80 --macro floor
npm run playtest -- --campaign --zones home --runs 80 --macro climb

# 2. Decide the target — add a tierRuns band for home in src/playtest/targets.ts:
#      tierRuns: [{ tier: 3, name: "City", runs: [30, 55] }]

# 3. Adjust a cost (a tier upgradeCost, the bread recipe, or a tile value) —
#    by hand, or let --optimize propose it — then write it back to source:
npm run playtest:apply

# 4. Re-measure; the campaign verdict should move toward "within"
npm run playtest -- --campaign --zones home --runs 80 --macro climb
Next build (optional). Wire a pacing objective so --optimize searches the tier costs / bread recipe to land runs-to-City inside your band automatically — validated across floor and ceiling so it can't "fix" the average player by soft-locking the naive one. The optimizer framework already supports it; it needs the band (step 2) plus a ~30-line loss function reading the campaign's runs-to-tier.

Worked example — the bread stall

Why the bracket matters, concretely. Home's Hamlet rung needs bread (a crafted good). The two agents diverge:

FloorCeiling
Board playlongest chainsteer toward the rung's farm shortfalls
Macrotier-up onlybuild Bakery → craft bread → tier up
Outcome (home, 60 runs)stuck tier 0, blocked on bread×6reaches Hamlet (tier 1), +55% coins/run

A naive-only sim would conclude home is soft-locked and "fix" it by slashing the bread cost — over-correcting for a player who simply never crafts. The bracket shows the rung is reachable; the real question becomes pacing (how many runs should Hamlet take), which is a design target, not a softlock.

Key design decisions

constants.ts is the single source of truth

A committed overrides layer was deliberately removed previously. M1 honors that by writing the canonical literals directly via codemod — no split-brain between an overrides file and the catalog, and every tune is a normal git diff.

No fakes — real reducer actions

The macro layer dispatches genuine BUILD/CRAFT/TIER_UP through the same rootReducer the game and unit tests use. The sim measures real economy numbers, never re-implemented arithmetic.

Deterministic by construction

A seeded Math.random override wraps every run; macros and the optimizer use no RNG and fixed iteration order. Same seed → byte-identical metrics. A committed snapshot guards against silent drift.

The optimizer never touches disk

Candidates are evaluated by mutating the in-memory catalog and restoring it in a finally. The optimizer returns a proposal; the reviewable write is a separate, explicit step.

Verification

Each piece ships with unit + integration tests; the default-path snapshots stay byte-identical (proving the additions don't disturb existing behavior).

  • Codemod: single-literal rewrite, idempotent re-apply, unresolved-path failure, array-by-id descent, byte-clean neighbors.
  • Policies/macro/compare: selection logic, the floor-stalls-on-bread contract, the ceiling-clears-it result, determinism.
  • Optimizer: synthetic convergence + non-destructiveness, the real spread objective driving loss → 0, deterministic proposal.

Full suite at time of writing: 4073 pass; lint / typecheck / typecheck:tests clean.

What's shipped since, and what's left

The loop closed with M1–M3; M4, M5, and the M2 extensions then landed too.

shipped  M4 — Declarative gating + CI oracle

The read-only half of the progression engine's Phase 2: a pure factsFromGameState projector, so a feature gate is now data — { fact: "resource.flour.total", op: "gte", value: 50 } evaluates against real state (tested to flip at exactly 50; compound gates compose natively). A coherence test plus the existing softlock/reachability oracle and tier-unlock union invariant mean a bad gate fails the build. Deferred: the live Effect applier that drives unlocks to replace the scattered hardcoded reducer gates — those gates already work, so swapping them is a behavior-preserving reducer refactor with no player-facing change and shouldn't be rushed.

shipped  M5 — Reach more knobs

The codemod descent generalized to follow id-keyed arrays at any depth, so it now reaches MARKET_PRICES, nested MAP_NODES.<zone>.tiers.<id>.upgradeCost.*, and board-template baseTurns — across both constants.ts and cartography/data.ts in one patch.

shipped  M2 extensions — hiring + board search

Worker hiring (WORKERS/HIRE) in the ceiling macro (build a House → hire farm threshold-reducers, conservatively with surplus), and a board-level lookahead search policy that beats greedy on chain coins (~+13% on home) by rolling out candidate moves a few plies.

next  Cross-zone arc — the one substantial remainder

Sequencing the FULL progression (farm → mine → fish → next zone) so cross-zone pacing is measurable. This is genuinely a separate project, not a quick generalization: the mine and fish boards don't share the farm's FARM/ENTER season flow — the mine is an expedition system (EXPEDITION/DEPART + food rations) and fishing has its own slice — so it's a mechanic integration, and the only live cross-zone edge today is home→meadow. The campaign's board loop is parameterized and ready; wiring the non-farm mechanics is the work. The optional reducer-level search agent (macro-exploit finder) sits behind the same deferred state.ts RNG-injection.

Glossary

Knob — a tunable balance value (a live get/set onto a constants field) with bounds. Loss — a number the optimizer minimizes; 0 = goal met. Floor / ceiling — the naive and near-optimal ends of the player bracket. Macro policy — between-run build/craft/tier decisions. Family-value spread — the ratio of realized value-per-tile across resources; a premium resource paying ≫ a staple is a balance smell. Softlock oracle — a code-derived check that the progression spine is reachable end-to-end.
