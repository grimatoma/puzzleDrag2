The suite is not broken — it is green, and most of it is genuinely valid. The feeling that "things are going out of date" is real, but it is not a story of thousands of stale tests. It is three specific stories:
  1. A few guards have silently rotted into no-ops — tests that look like they protect a contract but, after the TS migration or a hand-copied list drifted, can no longer fail. These are worse than no test: they lie to the next reader.
  2. The gates went out of date, not the tests. Green ≠ protected. The e2e suite, the visual suite, and all UI/canvas code run but do not block a merge — so they can rot again undetected, exactly as e2e already did once (doc 09).
  3. Rapid growth left sediment — 237 commits in one week produced a copy-not-moved "phased" shadow suite, a coverage-padding family, and ~315 weak assertions. Individually minor; collectively they inflate the count and the maintenance surface without adding protection.

Are they all valid? No — but the invalid fraction is small and nameable. The adversarial pass refuted three of the auditors' own "delete this" claims, defending tests that turned out to be load-bearing (see Don't touch). The honest headline is: a mostly-healthy suite with a rotted gate layer and a dozen genuinely dead/misleading tests.

Methodology (so you can trust the numbers). Every one of the 369 vitest files was read by one of 16 auditor agents, each cross-referencing the source under test (not the reference docs — per CLAUDE.md, the code is truth). Every "this test is invalid" claim was then handed to an independent adversarial verifier told to refute it by reading the actual source; only claims that survived are reported as CONFIRMED. Eight further agents audited the infra (CI, coverage, visual, e2e, perf/type-safety, tooling). The headline findings below were additionally re-confirmed by hand.

What's healthy (credit where due)

Before the problems: the large majority of this suite earns its keep, and several parts are better than most repos ship.

  • The core behavior tests are real. 8 of 16 audited file-groups came back with zero flagged files. Reducer slices, chain/spawn/layout math, boss modifiers, market pricing, save migrations, and the feature slices assert exact numeric outputs and the CLAUDE.md referential-equality contract against the real reducer.
  • The live drift-guards are excellent. The wiki/concept guards (derivedFacts dangling-key check, staleContent forbidden-token scan, conceptEntities/conceptSchemas exhaustive round-trips, catalog-keys-invariants) iterate live data structures and fail the moment new content is added without wiring — exactly the drift this repo produces.
  • The reachability exhaustive-scope guard pins the entire unreachable-content catalog so scoped-out content can never silently leak to players — and it's the same report the live /b#unreachable wiki page renders. Deliberately un-driftable.
  • The playtest harness drives the real reducer deterministically (seeded RNG) and is pinned by value-sensitive snapshots in the gating test job — a balance-constant edit genuinely breaks it.
  • The infra design is thoughtful. The node/jsdom split (fast node for logic, per-file jsdom opt-in for components), the beforeEach localStorage reset guarding order-dependence, a three-layer typecheck strategy, and a documented console-error allowlist are all above-average rigor.
  • The e2e specs are genuinely de-rotted. Every stale assumption doc 09 flagged (starting coins, nav selectors, zoned inventory, biome gating, dialogs-disabled) now resolves correctly against current src/. The only residual staleness is a comment in doc 09 itself, not the tests.

The rest of this document is about the minority that doesn't work — and the gates that let the good tests go unprotected.

1 · Guards that have rotted into no-ops highest priority

These are the most dangerous finding class: tests that appear to protect a contract but can no longer fail. Each was confirmed by hand.

flag-reads-drift.test.ts — a drift guard that guards nothing CONFIRMED (by hand)

The file walks src/ to prove "every direct story-flag read is documented in FLAG_READS." But its walker filters /\.(js|jsx)$/ (line 15) — and after the TS migration exactly 4 .js files remain in src/ (all under textures/categories/), none of which read a flag. So directFlagReads() returns an empty map, missing is always [], and expect(missing).toEqual([]) is trivially true and can never fail. Real reads (story.ts:637 state.flags.festival_announced, plus hearth_lit, keeper_choice_made, …) are invisible. The FLAG_READS registry in src/flagReads.ts also still lists dead .js/.jsx paths.
Fix, don't delete: change the filter to /\.(ts|tsx)$/, then reconcile flagReads.ts (it will immediately surface the many undocumented reads the guard was meant to catch).

actionTypes.test.ts — the slice-routing footgun test has itself silently drifted CONFIRMED (by hand)

This test hand-copies a SLICE_PRIMARY list meant to mirror SLICE_PRIMARY_ACTIONS in state.ts:1760 — the #1 footgun called out in CLAUDE.md. The copy has drifted: it omits SETTINGS/SET_TILE_ART_MODE, CIVIC/CLAIM, CIVIC/OPEN_CARE_PACKAGE, TOASTS/DISMISS (all present in the real set). And it only asserts each entry is a member of the ACTION_TYPES catalog (subset check) — it never compares against the real state.ts set, so the drift produces no failure. The guard is decorative. Worse, the CI gate named for this — action-types:check — only de-dupes the generated list; it never reads state.ts and cannot detect an unregistered slice action at all.
Export SLICE_PRIMARY_ACTIONS/ALWAYS_RUN_SLICES from state.ts, delete the hand-copied list, and assert set-equality against the imported source. Then add the real check the codebase thinks it has: every slice-owned action string must appear in one of those sets (automate the check-slice-action skill).

2 · Vacuous, name-lie & tautological tests

Tests that pass no matter what the code does, or whose name promises a behavior the body never exercises. All CONFIRMED by the adversarial pass.

tests/phase-11-polish.test.ts — "CLOSE_SEASON increments shuffle count" is a name-lie over removed behavior CONFIRMED

The test named CLOSE_SEASON increments shuffle count never dispatches CLOSE_SEASON; it calls createInitialState() and asserts next.tools.shuffle is defined — satisfied by the shuffle: 0 initial value. The claimed behavior was deliberately removed: state.ts:1083 says verbatim "Spec §11: shuffles are earned via almanac/quests — not free per season." The file's other two tests are bare is-defined smoke. A test whose name asserts a behavior the game intentionally does not have is worse than none.
Delete the file (or rewrite the season-close test to dispatch CLOSE_SEASON and assert what it does: turnsUsed reset, bonus coins, worker/building season-end tools).

tests/phase-7-quests.test.ts — expect(true).toBe(true) in disguise + a tautological loop CONFIRMED

"fresh state quests is an array or has active/claimed" asserts expect(s.quests !== null && s.quests !== undefined).toBe(true) — the argument collapses to the boolean literal true before s.quests is even inspected. The achievements-unlocked test reconstructs Object.fromEntries(ACHIEVEMENT_LIST.map(a=>[a.id,false])) — the exact expression init.ts:170 uses, from the same imported list, so it cannot fail. (The almanac spot-checks in the same file are fine.)
Replace with a real shape assertion (Array.isArray(s.quests) && length === 6) and drop the self-referential loop.

tests/phase-8-boss-weather.test.ts — two assertions, one duplicated, "weather" is fiction CONFIRMED

The 24-line file has exactly two live assertions: boss is null (valid) and CLOSE_SEASON resets turnsUsed to 0 — which phase-38 already asserts as a strict superset (it also checks view==='town'). The filename and header promise "weather / portal / decorations / boss-8.x" cases; none exist (grep finds them only in the comment). On fresh state boss is null, so this never even exercises the boss slice's own CLOSE_SEASON branch.
Drop the duplicate assertion, fold the null-boss check into a real fresh-state or boss-slice test, and stop naming a file after content it doesn't contain.

Local-stub tautologies & empty bodies CONFIRMED

reshuffle-horn.test.ts: tests 5–6 build the test's own mockShuffleBoard()/invokeShuffleAnimation() and assert them against themselves — they import no app code (tests 1–4, which drive the real reshuffle_board reducer path, are valid). iconUsage.test.ts: the test "does NOT include obviously-never-referenced UI icons" (lines 54–59) has an empty body — no expect, always passes. coverage-misc.test.ts:34: expect(sellPriceFor("iron_frame")).toBeGreaterThanOrEqual(0) — a price is always ≥0; the name claims "returns 0 when no coins field," but iron_frame resolves to 28 and the "coins field" concept no longer exists (pricing reads item.value). resource-to-threshold.test.ts: guards RESOURCE_TO_THRESHOLD, an export no app module imports (grep-confirmed), and opens with a can't-be-empty length > 0 check.
Delete the tautological/empty cases; keep the real ones alongside them. For resource-to-threshold, delete the test and the dead export together (or wire a real consumer first).

A permanently-skipped "flagship" test CONFIRMED

qa-pass-3-hazard-smoke.test.ts (and farm-10.7) gate their fire-spawn test on it.skipIf(!FIRE_HAZARD_ENABLED), but featureFlags.ts:6 hardcodes FIRE_HAZARD_ENABLED = false with no override — so it is the suite's permanent skip and never asserts farm-fire spawning works. (These are the "2 skipped" in the baseline.)
Either delete the skipped case or replace the flag-gate with a direct unit test of the spawn function that runs regardless of the flag, so the file stops implying coverage it doesn't provide.

~315 weak assertions across the suite quantified

Suite-wide grep: 187 toBeDefined(), 110 toBeTruthy(), 18 toBeGreaterThanOrEqual(0). Not all are bad (a toBeDefined after a lookup can be meaningful), but the ≥0-on-a-count family is always-true by construction, and the volume signals a habit of shape-smoke over behavior-assertion.
No mass action needed; treat as a lint-style code-review nudge (prefer exact-value assertions). Add an ESLint rule flagging toBeGreaterThanOrEqual(0) on counts.

3 · The phased suites (tests/phase-*) — shadow copies & poor discoverability

The 29 tests/phase-*.test.ts files are named after historical dev sprints, not features. Roughly a third are copy-not-moved duplicates of live co-located tests, and the naming scheme actively obstructs finding "the test for X."

Copy-not-move duplication across ~9 low-phase files

Every low-phase file's header says "Migrated from <src file>" — but the migration was a copy: every source it claims to replace still exists. phase-1-chain is byte-identical in assertions to src/__tests__/upgrade-thresholds.test.ts; phase-0-bugs re-covers helpers already in utils-coverage.test.ts; phase-2/3/5/6/7/9/10 shadow story-slice, market, species-5.x, npc-6.x, quests-7.x, mine-9.x. Topical counts confirm the imbalance — story: 14 co-located vs 1 phase; farm 10 vs 2; mine/boss/species 7–8 vs 1. Net: ~9 files / ~90 tests of pure duplication doubling CI time and drift risk.
Finish the move: delete phase-0,1,2,3,5,6,7,9,10 after a diff-confirm that each assertion is covered by its still-present src/__tests__ counterpart.

The phase-12-* meta-tests test the toolchain, not the game

phase-12-ci.test.ts regex-asserts the text of ci.yml (job names, node-version: 20, timeout presence) — a duplicate of what CI proves by running, brittle to any benign YAML edit. phase-12-infra.test.ts asserts Vitest globals are functions (tautological — they're the globals running the test) and self-enforces the very phase-N naming convention this review recommends retiring. phase-12-build.test.ts runs a full npm run build inside the 128 s vitest loop to check chunk-splitting. (The one keeper: phase-12-infra's SMOKE_INVARIANTS shape check.)
Move CI/build meta-tests out of the unit loop (the build guard belongs in the build job); reduce phase-12-ci to at most "ci.yml exists and defines the gating jobs by name"; extract the SMOKE_INVARIANTS test to its own file.

Source-text-regex behavior guards (phase-40, phase-41) brittle, but keep

Both readFileSync a .tsx and regex the source: phase-40 asserts /windowEvents:\s*false/; phase-41 slices a fixed 900-char window and matches setLoading(false) / mountedRef.current. They pass on a comment and die on a benign reformat, and boot no Phaser. But the adversarial pass confirmed each guards a real, documented regression (leaked pointer events; stuck loading overlay). Brittle, not invalid.
Keep, but harden: assert against the actual Phaser boot config object (or a jsdom render of TownPhaserCanvas), or downgrade to an ESLint rule — not a passing-on-comment "test."

The CLOSE_SEASON reset is re-tested in 6 different phase files

The same "turnsUsed resets to 0" assertion appears in phase-4, phase-6, phase-8, phase-11, phase-38, and phase-13-qa-quests — a change to CLOSE_SEASON semantics forces edits in six unrelated-looking files.
Consolidate to one canonical closeSeason.test.ts after relocation.

The deeper problem is discoverability. "Where is the test for zones?" has no answer unless you know zones were built in "phase 30." The repo's own dominant convention is co-location (222 files already live under src/). Retire the phase-N scheme: relocate the surviving meaningful files (phase-30/32/33 → src/features/zones/, phase-34/36 → src/features/workers/, the QA batch → their features), keeping the test bodies. First delete the phase-12-infra assertions that mandate the old naming.

4 · The coverage gate — honest numbers, hollow scope

The 70% / 60% thresholds are not gamed (vitest's coverage.all keeps the denominator honest — untested included files count as 0%). The problem is the include list: the gate measures ~20% of the app and says nothing about whether the game renders, drags, or boots.
~18k
source lines the gate measures (features/game/state/config + utils/state)
~40k
lines of .tsx UI (189 components) excluded entirely by **/*.{jsx,tsx}
~20%
of non-generated source is measured (~8% counting textures)
0%
of rendering / interaction / data-table code is gated

The whole canvas / UI / data layer is invisible to the gate

Unmeasured because they fall outside every include glob: GameScene.ts (2,429 lines), TileObj.ts, phaserBridge.ts, constants.ts (1,226 lines — the central resource/zone data table), story.ts, router.ts, market.ts, townLayout.ts, all of textures/**, all of ui/** (5,109 lines), plus every .tsx component and prototype.tsx (the reducer↔Phaser bootstrap) via the blanket exclude. The gate can catch a regression in a reducer helper — not in board layout, drag/merge, spawn tables, routing, or the town.

Excluding the cartography slice contradicts the gate's stated purpose

The config comment says the gate protects "the riskiest reducer code," yet exclude lists features/cartography/** — a live, wired slice (slice.ts, 8 action handlers; data.ts, 630 lines) that biome-entry gating depends on. (The sibling townsfolk exclusion is harmless — it's a 20-line .tsx re-export already caught by the blanket exclude, so that entry is redundant.)

The coverage-padding family is mostly legitimate — with a few wasted/loose spots auditor over-claim refuted

Contrary to first suspicion, the 11 coverage-* / *-coverage files mostly assert concrete outputs (boss reward increments, portal turn-budget extension 10→15, castle contribution deductions, lava spread) and the referential-equality no-op contract. Do not mass-delete them. Real defects are narrow: story-engine-coverage.test.ts covers src/story.ts, which the gate doesn't even measure (its coverage is thrown away); and ~6 loose assertions (utils-coverage.test.ts:176 typeof === "boolean" after runSelfTests() passes whether self-tests pass or fail; story-engine wraps its only assertion in if (r) {}).
Keep the family; tighten the ~6 loose assertions; add story.ts to the include so its padding actually counts.

The compounding risk: the coverage gate excludes UI/canvas → visual regression is non-gating → e2e (the only thing that can see the canvas bridge) is not a required check. So the entire rendering/interaction layer of a drag-based game has no gating coverage from any layer. That is the single most important structural gap in this report.

6 · CI gating — the root of "things going out of date"

The tests aren't the thing that rotted first — the gates did. A job existing in ci.yml does not mean it blocks a merge; required checks live only in GitHub repo settings, which are not in the tree and cannot be verified from the code.

The sharded e2e check name can never match the documented required check

The e2e job runs a matrix: shard: [1,2,3], so GitHub reports the checks as e2e (1), e2e (2), e2e (3). Every doc and the ci.yml header tell a human to add a required check named e2e — which matches none of them. The yaml author noticed (ci.yml:135-136) but the required-checks list (ci.yml:3) was never updated. So even a diligent maintainer following the docs would create a gate that silently never applies. doc 09's "e2e is now gating — DONE" is only half true: continue-on-error was removed (the job can fail red), but nothing blocks a merge.
Require all three e2e (1..3) contexts, or add a fan-in e2e-required job (needs: [e2e], if: always()) that asserts all shards passed and require that one stable name.

visual-smoke is continue-on-error: true — it protects nothing

ci.yml:179. A continue-on-error job reports success to the checks API even when the step fails, so it cannot gate regardless of settings. (Full detail in §7.)

Gates encoded as comments/intentions, with nothing keeping settings in sync

The required-checks list at ci.yml:3 is a comment; ci.yml:179 "REMOVE after goldens…" and ci.yml:5-6 "A HUMAN must still add e2e…" are unenforced TODOs. No in-tree ruleset/branch-protection/CODEOWNERS exists (confirmed: .github/ holds only workflows/), so the authoritative gate list can drift from the comment forever, undetected.
Add a tiny gate-guard job that reads branch protection via the API and asserts the required-check set equals a checked-in .github/required-checks.json, failing on drift.

Doc 09 has itself gone stale vs the code

09-ci-e2e-visual-gate.md predates the sharding + de-rot work and contradicts current playwright.config.js (retries, reporter) and ci.yml (sharding). Per CLAUDE.md, code is truth — a future agent trusting the brief would mis-audit the gate.
Banner the brief as historical, pointing at 09-ci-e2e-gate-findings.md and this doc as current.

7 · Visual regression — running, but zero pixel protection

The Playwright visual system gates nothing today

The only visual CI job, visual-smoke, is continue-on-error, non-required, and runs only the 3 *.desktop-smoke specs on --project=desktop. By the file's own admission the goldens "were not generated on this Linux runner, so the first run is expected to diff" (ci.yml:172-175). It is expected to fail, its failure is swallowed, and nothing gates. The only gating visual check is a vitest manifest test — and it validates file presence and bookkeeping, not pixels.

The committed goldens have local-host provenance — declared invalid for the runner that compares them

Git history shows the 110 goldens were committed by normal author PRs, not by the visual-rebaseline bot (whose commit test(visual): re-baseline… never appears). The config itself warns off-runner captures drift 3–5% (DOM) to ~38% (Phaser WebGL). So the baselines are the very captures the system says are wrong for CI — which is exactly why the gate can't be turned on. There is no hash/platform/commit provenance recorded, so staleness is undetectable by any automated check.

The bulk of the goldens (80 mobile) never run in CI; the diff-detector is never even proven

Of 110 goldens: 80 are mobile (visual.spec.ts/iphone-portrait), 16 desktop-smoke, the rest small balance/story sets. No CI job invokes the mobile project — those 80 are compared only when a developer runs npm run test:visual:mobile locally. The negative-control spec (which proves the harness can catch a diff) is test.skip unless VISUAL_NEGATIVE is set, and that env is set nowhere in CI. For a mobile-first game, the largest golden asset is dormant and the safety check unproven.

The rebaseline recovery path is fragile and partial

The visual-rebaseline workflow force-commits to the branch (fails on a protected main), has no concurrency guard, and regenerates only the desktop-smoke set — it can never re-baseline the 80 mobile or the balance/story goldens. It has never actually been exercised.

Path forward: run the rebaseline on Linux CI, wire the negative control into CI, record golden provenance (hash + platform), then remove continue-on-error and require the check — or explicitly decide the WebGL-golden maintenance cost isn't worth it, retire the board pixel-goldens to structural/DOM smoke, and drop the "REMOVE after…" TODOs so they stop reading as unfinished.

8 · E2E — load-bearing, de-rotted, but non-gating

The e2e suite is the only coverage of the React→registry→GameScene→TileObj bridge that node/vitest structurally cannot see (getContext is unimplemented under jsdom). It reads as genuinely current. But at the gate level its protective value is near-zero, and it can rot again exactly as it did before.

Non-required + mis-named gate (see §6) means the canvas layer can silently rot again

The suite bit-rotted 32→38 failures once while ungated (doc 09). The moment it's non-required, that failure mode recurs. The name mismatch (e2e vs e2e (1..3)) means even the intended fix wouldn't work.
Make it required (fan-in job), and add a nightly main run so rot is caught within a day even between PRs.

full-year.spec.ts is a timeout cliff against the shard cap

full-year.spec.ts:21 sets a 360 s (6 min) budget for one test (4 seasons + a mid-run reload; already overran a 180 s budget). With CI retries: 2, a slow/failing run can consume 3 × 360 s = 18 min inside a shard capped at timeout-minutes: 20 that also runs ~20 other boot-heavy tests serially. Playwright shards round-robin, so it isn't isolated. One slowdown can fail the whole shard for an unrelated reason.
Pin full-year to 0 retries and/or trim to 2 seasons, or move it to a nightly non-PR run.

Interactive e2e is mobile-landscape-only — desktop layout is behaviorally untested

playwright.config.js declares exactly one project, iphone-landscape (844×390). Desktop (1280×1024) exists only in the visual config, which is non-gating and screenshot-only. So the state-bridge is verified at two mobile viewports and never at a desktop breakpoint — any desktop DPR/hit-testing regression (the hidpi.spec concern) is invisible.
Add a desktop project to the interactive config, even scoped to a smoke subset.

The fiber-walking bridge + broad console allowlist can swallow the errors it should catch

helpers.ts hand-walks the React fiber tree for a hook with a "view" key — brittle to any component refactor (the exact break doc 09 recorded). And /Cannot read properties of null/i is on the ignored-console list, so a bridge returning null degrades to a slow timeout, and a genuinely new null-deref console.error is swallowed by the guard meant to catch it.
Expose a dev/e2e-only window.__hearthStore = { getState, dispatch } hook; scope the null-deref allowlist to the specific known Phaser-tween message.

A few specs assert nothing (boot-only smoke mislabeled as behavior)

chain.spec.ts "chain via touch" ends in a console.log with no assertion on the drag outcome (synthetic pointer events are unreliable); tile-collection accepts "either outcome"; town-buildings decoration asserts only toBeTruthy(). Fine as no-crash checks, but their titles overclaim.
Assert a real outcome or rename to mark them boot-only.

9 · Test-infra health — perf & type-safety

The typecheck-test-files gate is O(n) and already eats most of its 20-min budget

tools/scan-test-tsc.mjs spawns a fresh npx tsc per file over 222 files, each rebuilding the entire ~349-file program (~51k lib lines) at ~0.9–1.3 s/file → an estimated 200–290 s serial, trending toward the cap as the suite grows. The design intent (isolate per-file leakage) is valid; the implementation pays full program-build cost 222 times instead of once.
Replace the spawn loop with a single tsc program over a test-only tsconfig (keep isolatedModules). ~200 s → ~5–15 s.

142 test files get no strict typecheck, and 160 as Action casts defeat the TypedAction guard

tsconfig.json excludes all *.test.*; tsconfig.tests.json covers only e2e/visual/setup (and runs strict:false); the per-file scanner is src/__tests__/*.test.ts-only and non-recursive. So 142 files (109 co-located src tests incl. every feature slice.test.ts, 4 .test.tsx, 29 tests/) escape strict checking entirely. Compounding it, 160 as Action / as unknown as …Action casts on real dispatches bypass the TypedAction discriminated union — renaming a slice payload field would not fail these tests at compile time, and with loose assertions, often not at runtime. Green here does not mean "verifies current behavior."
Point one strict tsc program at all test files (fold into the fix above), and add a lint rule banning as Action on dispatch objects (steer to un-cast typed dispatches or the testAction() helper).

The getContext() canvas flood drowns the console-warning channel

Icon.tsx calls canvas.getContext("2d"); under jsdom that returns null (handled gracefully) but jsdom routes "Not implemented: HTMLCanvasElement's getContext()" to console.error — hundreds-to-thousands of lines across the ~52 jsdom files, with no stub anywhere. It masks nothing today, but it buries any real React act()/key/prop-type warning a future jsdom test would emit.
Add a jsdom-guarded getContext stub in setup.ts (return null; Icon already tolerates it). One-line, reopens the warning channel.

Default forks+isolate:true re-imports the heavy graph per file

No pool/isolate config; state.ts (1,908 lines / 68 imports) + constants.ts re-import into a fresh registry for each of 133+ files — the dominant chunk of the reported "import 152.91 s." Not incorrect, just untuned.
Measure pool:'threads' and/or isolate:false, scoping isolation to the ~7 files that use resetModules/module-global caches.

10 · Non-vitest validation tooling

action-types:check doesn't guard the footgun it's named for

The CI gate (list-action-types.mjs --check) only regex-parses ACTION_TYPES and de-dupes. It never reads state.ts, never sees SLICE_PRIMARY_ACTIONS, and cannot detect an unregistered pure-slice action — the CLAUDE.md footgun is not gated by the check named for action types. (Pairs with the drifted actionTypes.test.ts in §1.)
Add a real completeness check: every slice-owned action string must be in SLICE_PRIMARY_ACTIONS/ALWAYS_RUN_SLICES, driven off the exported sets + a scan of features/*/slice.ts.

build:docs is coupled to the production build and can hard-fail it

package.json build: vite build && npm run build:docs. build-docs.mjs calls execFileSync('git', ['ls-files'…]) with no try/catch and imports marked at top level. If git is unavailable (shallow/exported checkout) or a doc is malformed, npm run build exits non-zero even though dist/ was already emitted. Per CLAUDE.md, reference/ is NOT the game — reference material breaking the shippable build is a category error.
Split build (vite only) from build:site (vite + docs); guard the git/marked calls to warn-and-exit-0.

The playtest CLI/apply glue is unguarded, and --apply auto-updates its own snapshot

The harness math is well-tested and gated — but the 344-line CLI (SSR loading, HTML marker splicing, baseline promotion) and applyChangeList.mjs have no CI coverage. And runApply() runs vitest -u to update the drift-guard snapshot on every apply, so a bad balance edit can update its own tripwire; the only signal left is a git diff a human must read.
Add a --no-write smoke run of the CLI to CI; gate --apply's snapshot regen behind a tolerance band or explicit --accept-snapshot.

Don't touch — tests the audit tried to kill and couldn't

The adversarial pass refuted these "delete/merge" claims by reading the source. A cleanup that lumps them in as "meta tests" would delete real protection. Named here so they're spared.

  • reachability.test.ts exhaustive-scope guard — the ~90-id pinned catalog is the point: it enforces the SCOPED_OUT manifest so scoped-out content can't silently reach players. Same report as the live /b#unreachable page.
  • upgradeThresholds.test.ts "no imports" regex guard — a genuine architectural cycle-guard; the file is deliberately import-free so constants.ts/tileCatalog.ts/productionLines can share it without a circular dep. Regex-on-source is the only way to assert "zero imports."
  • mine-hazards-coverage.test.ts — not a mergeable duplicate; it lands unique branches (boss guard, base-rate gate boundary, null/undefined tile, no-grid bails, referential-identity no-op, floater text) that the mine-9.x phase files don't.

  • iconUsage.test.ts prefix block — a two-sided registry+classifier contract (breaks if an icon is deregistered or a prefix is dropped), not a self-referential tautology. Only the one empty test goes.
  • The playtest harness snapshot tests and the e2e specs — de-rotted and load-bearing; the fix is to gate them, not touch them.

How this review was produced

16 auditor agents read all 369 vitest files against their source-under-test; a per-group adversarial verifier refuted or confirmed each "invalid" claim by re-reading the code (3 auditor claims were refuted). 8 infra specialists audited CI, coverage, visual, e2e, perf/type-safety, and non-vitest tooling; 3 of those (CI, visual, e2e) were re-run individually after the first pass returned malformed output. The two headline rotted-guard findings and the coverage/CI facts were additionally re-verified by hand. Total: ~40 agents, ~2.7M tokens of analysis. Nothing here was taken from a reference/ doc as fact — every claim is anchored to src/, per the CLAUDE.md "code is truth" rule.
