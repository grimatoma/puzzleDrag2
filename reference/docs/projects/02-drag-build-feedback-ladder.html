Drag-Build Feedback Ladder

A per-tile audio + haptic ladder during the drag-build chain — so building feels as alive as the release payoff. Pure presentation: no economy, no state shape, no persisted-data changes.

Overview

Drag-to-build a chain was almost silent during the drag. The only escalating audio (chainCollect, pitch rising with chain length) fired after release, and the only haptic during the drag was a single navigator.vibrate(10) at drag-start. The drag was heavily juiced visually (growing line segments, ring bursts, star previews, validity colour-tween) — but with no per-tile sound and no escalating haptic, building felt flat next to the satisfying release.

This change adds a per-tile feedback ladder during the drag at the single chokepoint, addToPath: a rising-pitch tick on each successful tile add, a distinct confirm cue the moment the chain first crosses the collectable threshold, and an escalating haptic — for maximum game-feel ROI per line of code.

TL;DR. Two new synth sounds + one tiny pure helper module + a wired-up addToPath. Zero economy / state-shape / persisted-data changes — no SAVE_SCHEMA_VERSION bump, no new actions, no slice touched. Fully revertible.

What shipped

🔊 Per-tile tick

A chainTick bleep on every successful add, pitched up by chain length so a long chain climbs audibly (a 7-chain's last tick is clearly higher than its first).

🔔 Threshold confirm

A distinct chainReady two-note cue the instant the chain first reaches _effectiveMinChain() (base 3, raised by an active boss). Fired once per drag.

📳 Escalating haptic

A per-add navigator.vibrate that grows with length, with a stronger bump at the crossing — gated on hapticsOn, the same gate startPath already uses.

Where the ladder lives

Both startPath (drag start) and tryAddToPath (pointer-move, extend) route a successful add through the single funnel addToPath. That's the one place the tick + confirm + haptic belong — fired between the path.push and redrawPath so sound lands on the same frame as the visual ring-burst. The release path (endPath → collectPath) and its post-release chainCollect payoff are untouched.

Step 1 — two new sounds

Added to the SOUNDS record in src/audio/index.ts, mirroring the existing step shape and the low ~0.05 gain range. play() is unchanged — it already accepts { pitch }, clamps it to [0.5, 2.5], and self-gates on enabled.sfx.

Step 2 — pure helpers (no Phaser, unit-testable)

In a new src/game/dragFeedback.ts — kept free of window / AudioContext so they unit-test in node and co-locate a .test.ts like the other src/game/*.ts pure helpers.

export function tickPitch(len: number): number {
  // 1.0 at len 1, +~6%/tile, clamped to 2.2 so it never tops play()'s 2.5 ceiling.
  return Math.min(2.2, 1 + Math.max(0, len - 1) * 0.06);
}

export function crossesThreshold(prevLen, newLen, minChain): boolean {
  // True only on the single add where the chain first becomes collectable.
  return prevLen < minChain && newLen >= minChain;
}

export function tickHapticMs(len: number): number {
  // 0 below len 2 (startPath already buzzed len 1); escalating, clamped 35ms.
  if (len < 2) return 0;
  return Math.min(35, 6 + len * 3);
}

Step 3 — wire into the funnel

addToPath reads the length before and after the push, fires the tick, fires the confirm once (latched), and the haptic — all gated correctly — then proceeds to the existing redraw/emit.

addToPath(tile: TileObj): void {
  const prevLen = this.path.length;          // length BEFORE this add
  tile.setSelected(true);
  tile.pulse();
  this.path.push(tile);
  const len = this.path.length;              // length AFTER this add
  const minChain = this._effectiveMinChain();
  const justBecameReady =
    crossesThreshold(prevLen, len, minChain) && !this._readyFiredThisDrag;

  play("chainTick", { pitch: tickPitch(len) });   // per-tile rising tick
  if (justBecameReady) {
    play("chainReady");
    this._readyFiredThisDrag = true;
  }

  if (getRegistry(this.registry, "hapticsOn") && navigator.vibrate) {
    const ms = justBecameReady ? 50 : tickHapticMs(len); // stronger at the crossing
    if (ms > 0) { try { navigator.vibrate(ms); } catch { /* unsupported */ } }
  }

  this.redrawPath();
  this.updateGrassHover();
  this._emitChainUpdate();
}

A one-shot latch _readyFiredThisDrag (a new class field) is reset in clearPath — which every drag-start routes through via startPath → clearPath(false) — so the confirm fires at most once per drag even across a backtrack re-cross.

Design decisions

Latch, not replay

On a backtrack below the threshold then re-cross, the confirm is suppressed by the latch. A skeptic could argue a re-cross is a real re-confirmation; the latch was chosen so the cue reads as a single deterministic "now collectable" moment.

Scene-side, not React-side

Firing inside addToPath (option A) lands the audio on the same frame as the ring-burst and reuses the hapticsOn registry read startPath already does. The CHAIN_UPDATE-delta alternative (option B) is one event-hop later and fiddlier (it also emits on backtrack and drag-end).

Cross-collect cue deferred

The optional "partner became eligible" blip runs a grid scan on every add and is the least certain to feel good — deferred to keep this tight and CPU-cheap.

Static behaviour trace

Traced through the funnel (the canvas layer has zero unit coverage, so behaviour is reasoned + the pure helpers are tested directly):

  • Straight drag to length 5 (minChain 3): 5 chainTicks with rising pitch (1.0 → ~1.24), chainReady once at the 3rd add, crossing haptic 50 ms.
  • Backtrack 5 → 2 → 4: re-cross at len 3 sees _readyFiredThisDrag === true → no confirm replay.
  • sfx OFF: play() short-circuits at if (!enabled.sfx) return — scene-side calls cannot bypass it.
  • haptics OFF: the getRegistry(registry,"hapticsOn") gate skips the whole vibrate block.
  • Boss minChain 5: chainReady fires at length 5, not 3, via the live _effectiveMinChain().

New tests

src/game/dragFeedback.test.ts — the pure helpers
  • tickPitch: 1.0 at len 1; monotonic increasing; clamped ≤ 2.2 (and < play()'s 2.5 ceiling); floors at 1.0 for degenerate lengths.
  • crossesThreshold: fires at the crossing (incl. jump-past), not when already at/above, not on backtrack-down; honours a raised boss min-chain.
  • tickHapticMs: 0 below len 2; positive and non-decreasing from 2 up; clamped ≤ 35.
src/__tests__/chain-feedback-audio.test.ts — sound registration

Asserts SOUNDS.chainTick and SOUNDS.chainReady exist with non-empty steps whose every step has positive freq/dur/gain, and that the confirm climbs above the tick's range. Does not call play() (it would construct an unavailable AudioContext in node).

Adversarial double-check

How to prove the cues fired (without ears)

Spy on play / add a console.count('chainTick') at the call site, drag a 5-chain: expect 5 ticks (incl. the startPath first add, which routes through addToPath) and chainReady logged once at the 3rd add. For haptics on desktop (where navigator.vibrate is a no-op), stub it: navigator.vibrate = (...a)=>{console.log('VIBE',a);return true;} and toggle Haptics off to confirm zero VIBE logs.

Rollback safety

Entirely additive presentation. Reverting the addToPath edits + the latch field + removing the two SOUNDS entries + the helper file fully restores prior behaviour. No data, schema, or actions touched — a revert cannot corrupt saves or no-op a slice.

Risks & landmines

Wrong setting names (top landmine). There is no volume and no reduceMotion in this project. The only gates are sfxOn (→ enabled.sfx, auto via play()) and hapticsOn (registry, read with getRegistry(this.registry,"hapticsOn")). Do not invent settings or gate on reduceMotion.
  • sfxOn is not in the Phaser registry. Scene-side code relies on play() self-gating, not a registry read — and does not need one.
  • Length timing. Read prevLen before the push and len after, or crossing detection is off by one.
  • AudioContext gesture unlock. Sound is silent until the first pointerdown unlocks the context; click the canvas once in manual/console testing.
  • Canvas has zero unit coverage. Don't import GameScene in a Vitest test (needs window/Phaser); keep tests on the pure helpers + SOUNDS shape.
