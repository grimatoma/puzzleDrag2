The Quest Reward Display

Quest rewards used to be just coins and ✦ XP. They are now a small bundle — tools, resources, runes, and (newly) a building or tile unlock. The current card silently drops everything past coins+XP. This doc settles on a design: a headline chip + an expanded reward list, capped at five, with worked example quests in real game content.

Scope: src/features/quests/ · src/features/almanac/ · reward grants in src/state.ts. Decision recorded June 2026.

1The problem

Today the daily quest card renders its reward as one gold token with two text glyphs — coins (◉) and almanac XP (✦):

// src/features/quests/index.tsx — QuestCard, ~line 186
<span className="quest-reward" title="Reward">
  <span>+{coins}◉</span>
  {xp > 0 && <span className="quest-reward-xp">+{xp}✦</span>}
</span>

But a QuestReward can already carry tools and items, the almanac track adds runes and structural perks, and we now want unlock-a-building and unlock-a-tile rewards too. grantQuestRewardExtras() hands the existing extras out on claim — they're just never drawn.

This is a live bug, not just a polish gap. A daily quest authored with rewardItems: { plank: 5 } grants 5 planks on claim, yet its card shows only the coin/✦ chip. The two [TEST] templates in templates.ts already exercise this — they're invisible in the UI right now. The almanac is the only place extras surface, via rewardLabel(), as an icon-less text string like +1000◉ · +5 Seedpack · +3 Lockbox · 1 rune.

2Reward taxonomy

The design must hold every kind below, in any combination, up to a hard cap of five rewards per quest. The two unlock kinds are the reason we moved to a roomy list — an unlock deserves a name, not a bare icon.

Rewards are not flat: baseline rewards (coins/XP) appear on nearly every quest and stay quiet; a rune, a perk, or an unlock is the reason a player cares and must lead. A single text string flattens that hierarchy — which is what makes the almanac string read like a receipt.

3The chosen design

Headline chip + expanded reward list. Because a bundle is small (≤5), we don't need to cram chips into the header. Instead:

  • One headline chip sits top-right where the old token was — the single most valuable reward (an unlock if present, else a rune, else coins). It pops in its kind's premium palette.
  • An expanded list sits under the progress bar: one row per reward, icon + name + an optional badge. The badge is a ×N quantity for counted goods (only when N>1), or the plain value for currencies. Unlock rows wear a framed, tinted treatment and an UNLOCKS / NEW TILE tag.

The five-reward cap is a design rule, not just a UI fallback. Authoring guidance: a quest grants at most five rewards, at most one headline (unlock/rune/perk). The list never scrolls, never collapses, never needs a +N popover. If a bundle ever exceeds five, that's a content bug to fix at authoring time, not a layout to design around.

5Directions considered

Two alternatives were on the table before settling on the list. Both lose to it specifically because bundles are small and one reward is often a named unlock.

A · Horizontal icon-chip cluster

A wrapping row of icon+count chips where the token sat.

  • Most compact; smallest change.
  • No room for a reward name — an unlocked building reads as a bare icon, which is exactly the case we most want to spell out.
  • Wrapping/overflow logic is wasted effort when the cap is five.

C · Compact summary + popover

Coins + a +N pill; tap to expand.

  • Tiniest footprint.
  • Hides the prize behind a tap — buries the unlock, the whole reason the quest is exciting.
  • Extra interaction for a bundle small enough to just show.

The list keeps A's "show everything" and C's calm footprint, and adds the one thing both lack: room to name a reward. With a five-item cap there's no penalty for spelling each one out.

7The claim moment

There's already a floating reward-chip layer (src/ui/RewardChipsLayer.tsx + rewardEvents.ts) that pops “+N coins” on claim. Extend it so claiming a bundle fans out one chip per reward row, each flying to its destination — coins to the coin counter, tools to the tools tray, items to inventory. The unlock lands last and biggest: the building/tile chip flies to the town/almanac tab with a brief hold, so the headline reward is the climax of the animation, not buried in a stack of coins.

Sequencing: stagger ~60ms apart in taxonomy order — coins, XP, tools, items — then the headline (rune/perk/unlock) last. For an unlock, follow with a one-time toast: “Mill unlocked — build it in town.”

8Implementation sketch

The rendering is cheap and presentational. The grants for the two unlock kinds are the real work — and they differ sharply in cost.

Rendering — one component, one resolver

// src/features/quests/RewardBundle.tsx (new) — replaces the inline token AND rewardLabel()
export function RewardBundle({ reward, mode }: { reward: DisplayReward; mode: "full" | "summary" }) {
  const rows = toRewardRows(reward);        // flatten → ordered Row[] (coins,xp,tools,items,runes,structural,unlock*)
  const headline = pickHeadline(rows);       // unlock > structural > rune > coins
  // "full": headline chip + <ul class="manifest"> of rows. "summary": headline chip + "+N".
}
  • Icon resolution — extend dailyRewardIconKey() (concepts.ts:306) into rewardRowIcon(kind,key): coin→tile_coin_golden, tool→bare id (or player_<id>), item→its key, rune→rune_stone, building→bld_<id>, tile→the tile key itself. All keys already exist in the registry.
  • Names — buildings from BUILDINGS[].name; tiles/items/tools from the ITEMS label field. No new strings to author.
  • Swap-in sites: daily token (index.tsx:186); delete rewardLabel() and use the list at the almanac rows (index.tsx:246/263); the QuestsPanel rows (mode="summary").
  • Styling: add .quest-manifest + .qm-row (and .qm-row--unlock-*) to components.css beside the existing quest classes; colours all come from tokens.css.

9Open questions

  • Does a tile-unlock quest make the tile spawnable, or only discovered? Affects whether Example ② actually changes the board or just the almanac. (Recommend: discovered-only for v1, matching the daily-streak behaviour.)
  • Where does an unlocked building live in state? A dedicated questUnlockedBuildings set vs. forcing it into the cartography roster. Needs the small spec noted above.
  • Headline tie-break when a bundle has two headline-class rewards (e.g. a rune and a perk): proposed order unlock > structural > rune > coins — one constant.
  • Cap enforcement. Should a guard test fail CI if any template/almanac tier authors >5 rewards, to keep the "no overflow" promise true?
  • Resource icon coverage. Every items key needs a registry icon or a generic-crate fallback; audit the keys reachable from templates.
