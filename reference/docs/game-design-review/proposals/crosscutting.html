Proposals · Cross-cutting lenses 07

Cross-cutting — the six whole-game programs

These are not single screens; they are systems that touch every surface. Each lens below proposes the program — the one shared mechanism to build — and cross-links to the per-surface pages for the concrete spots it lands. The pattern repeats: the game already computes the value, defines the token, or ships the primitive; the work is wiring the last mile so the player sees it.

Cross-cutting through-line. Almost every gap here is a last-mile gap, not a missing feature. XP is granted (1/chain) but never floated; a 44px hit-target token exists but is applied only in the Dev-Panel wiki; motion-duration tokens are defined but consumed by nothing; an announce() screen-reader bridge exists but the board never calls it; the ProgressTrack primitive ships but fractional harvest progress is shown nowhere. Two lenses carry the most player-felt value — reward legibility (#2) and accessibility — and one carries the most brand value: iconography unification (#12).

X1S1Effort M–L Top priority #2
Build the reward ladder — make every gain the game already computes actually show

The economy is generous under the hood but silent on the surface. Every chain grants +1 XP (state.ts:573 applyAlmanacXp(state, 1)) yet the collect float shows only the resource gain (GameScene.ts:1881); the run summary tallies chains, coins, upgrades, bonds and beats but has no XP/level row (features/runSummary/index.tsx:183-207); level-up celebrates a bare numeral even though each level has a named tier + concrete reward sitting unused (ui/LevelUpCinematic.tsx:44-49 vs features/almanac/data.ts ALMANAC_TIERS); a craft produces an item worth its sell value (a Bread Loaf sells 125◉, ITEMS.bread.value) but the recipe pane shows inputs only, never the output or its worth (crafting/index.tsx:199-217); fractional harvest progress accrues in state.resourceProgress toward each threshold but is surfaced in no UI component; and the board states only a decrementing turn budget, never a season goal.

X2S2Effort L Top priority #12
Split the icon system by size/role, not by key — one flat line register for metadata, painterly for subjects

Two icon families render side by side and the choice between them is made by key identity alone: Icon asks svgRenderFor(iconKey) and, if the key is a registered design.* vector, draws a flat inline SVG (Icon.tsx:109, :246); otherwise it bakes an iconRegistry entry to a painterly PNG (Icon.tsx:154). Nothing keys off render size, so a 14px inline metadata icon and a 40px subject thumbnail can each land in either family — a flat vector chip beside a glossy painted one at the same scale. Add the board's saturated candy squares (Board · B1), the canvas-baked ui_* nav register (ui.tsx:69-74), heavy-gloss inline thumbnails and the season strip's fourth vignette register, and the chrome reads as four illustration languages at once.

X3S2Effort M
Adopt one duration/easing scale across CSS and canvas, then escalate the reward-motion ladder

The motion-duration tokens --motion-fast/med/slow are defined (tokens.css:190-192) but consumed nowhere — a grep for var(--motion-*) returns zero hits, and the canvas _dur() that every board tween routes through is a pure passthrough no-op (GameScene.ts:577). Meanwhile routine chains get lavish juice (shake, radial flash, particles) while the run summary has zero motion, the board hard-gates all idle motion for a flat 1500ms warm-up (ANIM_STARTUP_DELAY_MS, GameScene.ts:59), and cinematic holds are unrelated magic numbers (LevelUpCinematic.tsx:5 HOLD_MS=1600). The tokens are a scale waiting to be plugged in.

  • Actually consume var(--motion-*) in the CSS transitions/animations (they are currently orphaned) so DOM and canvas share one vocabulary; fix the stale tokens.css:188 comment that claims motion is intentionally ungated.

  • Gate the collect juice behind reduced-motion (see X5) while keeping the audio/haptic ladder, so "reward" still lands without vestibular motion.

X5S1Effort M–L Top priority #5 (canvas motion)
Make the board reachable and legible without a pointer or full color vision

Verified against code: the DOM/CSS layer is reduced-motion-safe — a global prefers-reduced-motion block collapses transitions/animations (index.css:533). The gaps are canvas-side and input-side. The board host is a bare <div> with no role/aria-label/tabIndex (prototype.tsx:391); it is pointer-only. An announce() aria-live bridge already exists and works (a11y.ts:20, wired to the toast notifier) — but the only caller is the storage-full toast; the canvas never announces a chain result. Chain validity is signalled by color alone: warm orange 0xff6d00 (valid) vs desaturated brown 0x9a4630 (too short) (GameScene.ts:84-85) — the worst pairing for red-green CVD. And chain.ts already exports a pure isAdjacent() 8-way check (game/chain.ts:47) that a keyboard path mode could reuse verbatim.

X6S2Effort M Theme T1
One nav hub + a consistent taxonomy — the umbrella fix for feature sprawl

Feature views have accreted well past the six-slot bottom nav (ui.tsx:69-74): Town, Inventory, Craft, Map, Townsfolk, Quests — while KNOWN_VIEWS also carries almanac, chronicle, charter, boons, market, decorations, tileCollection, festivals and more (router.ts:34 + FEATURE_VIEW_KEYS). Labels mix the world's language with the code's: the nav says "Map" but the route and view key are cartography; the tile catalog is tileCollection in code but "Tiles"/"Almanac" to the player. The umbrella fix is a single navigation hub plus a taxonomy that reads one way everywhere. The hub design lives on Shell · nav hub; this lens is the summary + the rename list.

X7NEWfrom this pass
New findings surfaced while grounding these programs

Items not in the original report, found reading the cross-cutting code for this pass — each re-verified against the live source (one initial reading did not survive that check; see the note).

  • The recipe coins field is a dead slot; the value to surface is the item's sell value. grantCraftOutput does coins += (recipe.coins ?? 0) (crafting/slice.ts:86) and the type declares an optional coins? (crafting/index.tsx:37), but no recipe in RECIPES ever sets it (verified across constants.ts:934–1000), so the branch always adds 0. The recipe detail pane also renders inputs only (crafting/index.tsx:199-217): the player is told the cost but never the output or its worth. Fix by showing getItem(recipe.item).value (a loaf sells 125◉), and either wire or delete the dead coins field (feeds X1). (An earlier draft of this finding claimed the reducer "pays out" a hidden coin reward; that's false — recipe.coins is never populated. Corrected here.)
  • The icon-family split keys off identity, never render size. ui/Icon.tsx:246 chooses SVG-vs-painterly purely by svgRenderFor(iconKey). That is the mechanical root cause of "flat next to painterly at the same size" — the fix is to make size a first-class input to the decision (feeds X2).
  • The 44px hit-target token is not "unapplied" — it is applied in the wrong bundle. --tap-min is consumed only in wikiTheme.css:1678-1836 (the Dev-Panel wiki), so the mechanism is proven; it simply never reached the game UI (corrects the report's "defined but never applied"; feeds X4).
  • Level-up feedback is inconsistent, not just bare. The reducer computes leveledUp and fires a congratulatory bubble only at level 2 (state.ts:587-590), while LevelUpCinematic shows a bare numeral at every level. Two different level-up responses coexist; X1 Option A unifies them.
