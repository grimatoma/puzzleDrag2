Proposals · Additional findings 08

Additional findings — a fresh hunt through the code

This page is not a proposal against the original report — it is a second pass through the areas that review scoped out: the Settings screen, cold-start/loading, error and failure paths, audio, the returning-player flow, and small-viewport behaviour. Everything below is read straight from src/, with the file:line and a fully-defined fix. Where an area turned out to be genuinely solid, it is listed under Checked and OK rather than dressed up as a problem.

Through-line. The engine-robustness story is good — a first-byte boot splash, a root error boundary with a Reload screen, a wired daily-streak return, and crafting that shows exactly which input you're short. The gaps are all in what the player is (not) told when something goes wrong or when they want out: save writes fail silently (AF2), a developer cheat menu and internal design-docs link ship inside the player menu (AF1), the entire reset-save pipeline exists but no button triggers it (AF3), a failed town-art load paints a blank scene with no retry (AF4), and the Settings tab is missing the three controls players reach for first — volume, reduce-motion, tutorial (AF5).

AF1S2Effort S new
Gate the Debug cheat panel and the internal Docs link out of the player-facing menu

The main menu renders a 🛠 Debug button unconditionally — in every build — and it opens a panel of dev cheats (DEV/ADD_GOLD +10,000, DEV/ADD_LEVEL, jump-to shortcuts for in-development views). Right below it, a 📄 Docs link opens the internal design-docs site. Neither is behind any dev flag, so a shipped player can grant themselves infinite gold and browse the team's design notes. Debug button: features/settings/index.tsx:180; Docs link: settings/index.tsx:196; cheat actions: features/debug/index.tsx:102.

Reuse the existing appQueryParams + sessionStorage pattern to expose a dev flag; render Debug/Docs only when set. Pro: still reachable in a deployed build for QA without a rebuild. Con: discoverable by anyone who reads the URL scheme — fine for a cheat, not a secret.
B · Build-time (import.meta.env.DEV)

Implementation
  • Add a isDevFlagEnabled() next to isConceptTilesFlagEnabled() (src/appQueryParams.ts:36), keyed on dev=1.
  • In src/features/settings/index.tsx:180 wrap the 🛠 Debug button and the 📄 Docs anchor (settings/index.tsx:196) in that check; leave the player-facing 📖 Game Wiki / update buttons untouched.
  • Belt-and-braces: no-op the DEV/* cheat actions unless the flag is set, so the buttons can't be replayed via a saved action log (src/features/debug/index.tsx:102).
  • Confirm the 📖 Game Wiki link (settings/index.tsx:186 → /b/, the Dev Panel) is the intended player build; if the balance-tuning tools ride along, gate it too.
AF2S2Effort S–M new
Make save failures visible — a full or blocked storage loses a whole session silently

Two failure paths in the save layer report only to console.warn and never to the player. Writes: persistStateNow swallows a quota-exceeded / private-mode error and warns exactly once (state/persistence.ts:57) — a player in a private tab or with full storage plays for an hour and loses everything with no signal. Loads: a corrupt or unmigratable save is removeItem'd and discarded, again with only a console line (persistence.ts:28, :42) — the next launch is a silent fresh start.

NOWfailure only reaches the console
PROPOSEDone honest, non-blocking notice
Implementation
  • Have persistStateNow return a boolean/status and, on the first failure, dispatch a toast (the src/features/toasts slice already exists) instead of only warning (src/state/persistence.ts:57). Keep the once-guard so it never spams.
  • Distinguish quota vs unavailable (the code already sniffs /quota/i at persistence.ts:63) to pick the copy ("storage full" vs "private mode").
  • On load-discard, set a transient flag in the returned state (or a sessionStorage marker) that the shell reads once to show the "started fresh" notice (persistence.ts:28).
  • Add a reducer/unit test that a stubbed setItem throw surfaces exactly one toast.
AF3S3Effort S new
Add a "Start Over" control — the entire reset-save pipeline is wired but nothing dispatches it

There is a complete, working reset path: the reducer handles SETTINGS/RESET_SAVE (state.ts:1873), its effect clears every hearth.* key (clearAllHearthStorage, features/settings/slice.ts:27), and prototype.tsx:647 reloads on pendingReload. But no component ever dispatches SETTINGS/RESET_SAVE (grep across src/ finds only the type, reducer, and tests). A player with a broken or unwanted save has no way to start over short of clearing site data by hand.

NOWSettings tab · no way out

Implementation
  • Add a "Data" section to SettingsTab with an ActionBtn variant="danger" (src/features/settings/index.tsx:262) that opens a confirm.
  • Reuse the existing inline confirm markup from the leaveBoard branch (settings/index.tsx:359); on confirm dispatch { type: 'SETTINGS/RESET_SAVE' }.
  • Nothing else to build: state.ts:1876 clears storage and prototype.tsx:647 performs the reload via pendingReload.
AF4S3Effort S–M new
Give the town a failed-asset fallback — a load error renders a silent blank scene

The board bakes its art procedurally to canvas, so it has no network-asset risk. The town does: TownScene.preload network-loads town/tileset.png and the character-atlas (ui/town/TownScene.ts:145) with no loaderror handler anywhere in the codebase. If the tileset fails (flaky first load, 404), create() hits addTilesetImage(…) → null and early-returns (TownScene.ts:181), leaving an empty scene — no ground, no message, no retry.

NOWtileset fails → blank
PROPOSEDhonest error + retry
Implementation
  • In preload, add this.load.on('loaderror', (f) => …) to set a this.assetsFailed flag and emit a registry/bridge event (src/ui/town/TownScene.ts:142).
  • Replace the silent if (!tilesetImg) return; at TownScene.ts:181 with a flat-colour fallback ground so the scene still renders.
  • Have src/ui/TownPhaserCanvas.tsx subscribe to the failure event and render a Retry card that calls scene.restart(); the map already models a boot-failure overlay (src/features/cartography/PhaserMap.tsx:217) — mirror it.
AF5S3Effort S–M new
Round out a thin Settings tab — master volume, an in-app reduce-motion toggle, tutorial off-switch

Settings currently offers exactly two audio toggles and a tile-art picker (AUDIO_ROWS = sfxOn, hapticsOn, features/settings/index.tsx:211). Three controls players reach for are missing: (1) volume — audio is binary on/off with no level (audio/index.ts plays at fixed per-sound gains); (2) an in-app reduce-motion toggle — motion is only honoured from the OS media query (GameScene.ts:2163), so a player who wants calm without changing system settings can't; (3) a tutorial off-switch — DEFAULT_SETTINGS.tutorialDisabled exists (settings/slice.ts:46) but no UI ever sets it (only a one-shot "Show Tutorial" replay in the main menu).

NOWtwo toggles + tile art
PROPOSEDthe three expected controls
Implementation
  • Add a volume (0–1) setting; multiply each step's gain by it in src/audio/index.ts:127 (read via setEnabled-style sync in src/audio/useAudio.tsx:26). Render a range row in features/settings/index.tsx:262.
  • Add a reduceMotion setting and OR it into _motionEnabled() beside the media-query check (src/GameScene.ts:2163); mirror into the CSS-motion gate so both layers agree.
  • Expose the existing tutorialDisabled flag as a toggle (features/settings/slice.ts:46) so tips can be turned off, not just replayed.

Also folded in (polish)

  • No ambient music track. SOUNDS is SFX-only synthesis (src/audio/index.ts:48) — a cozy seasons game reads as unusually quiet at rest. If music is ever added, it wants its own volume row (the reason the slider above is "Volume", not "SFX Volume").
  • Haptics is binary only. The vibrate calls are hard-coded durations in src/audio/useAudio.tsx:43; fine, but a "subtle / strong" choice is cheap once the toggle grows a slider neighbour.

Checked and OK — areas that turned out solid

These were on the hunt list and hold up in the code; noting them so they aren't re-litigated.

  • Cold-start / loading. A hand-crafted boot splash is painted on the first byte in index.html (animated hearth, honours prefers-reduced-motion), dismissed by dismissBootSplash() once the board engine is up (prototype.tsx:440), with a 12 s safety-net so a boot failure can't trap the player. Strong.
  • Render-crash recovery. A root error boundary shows a "Something went wrong · Reload" screen and reassures the player their save is intact (main.tsx:36); per-feature boundaries wrap every modal/screen (src/ui.tsx:21). The map also models a Phaser/WebGL boot-failure overlay (features/cartography/PhaserMap.tsx:217).
  • Returning-player flow. LOGIN_TICK is dispatched on mount (prototype.tsx:637), advances/breaks the streak by date diff, credits the DAILY_REWARDS ladder, and opens the daily-reward modal — idempotent per local day. (This game has no idle/offline production, so a "while you were away" economy summary would be inventing a system that doesn't exist; the daily reward is the correct return moment.)
  • Insufficient-resource feedback (crafting). The recipe detail renders a per-input have-vs-need grid with an OK check and a "Missing inputs" status, and disables the Craft button — so a failed craft explains itself (src/features/crafting/index.tsx:187).
  • Save migration. An out-of-version save is run through a migration ladder rather than blindly wiped; only truly forward/corrupt saves are discarded (src/state/persistence.ts:24). The gap is only that the discard is silent — see AF2.
  • Landscape / small-viewport. The board has a purpose-built landscape composition at src/ui/puzzleBoard.tsx:1749 (rail beside a full-height board), the town scene reflows its zoom for portrait phones, and cartography swaps rail↔bottom-card by orientation. Desktop/tablet board width is covered separately by the Board page (B3).
