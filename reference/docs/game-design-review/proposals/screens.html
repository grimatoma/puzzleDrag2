Proposals · Core surface 04

Screens & panels — fully-defined proposals

The tabbed screens — Quests, Achievements, the Tile Almanac, Crafting, Inventory, and the Townsfolk hub (Workers · Castle · Bosses · Boons · Charter) — are where the game explains itself. They're structurally sound, but a scatter of mislabeled states, buried actions, and hidden values makes the finished systems read as a demo. Each proposal below pairs the panel as it is now with a rendered build target and a concrete file:line spec.

Screens through-line. Two priorities carry the page. (SC1, priority #6) Repair the leaks — a cluster of near-one-line bugs where a shipped feature renders as broken or contradictory (a researched tile that says "Locked", blank mining tabs, snake_case boss tributes, a backwards buy-cost line, "✓ Done" over a 12/100 bar). (SC5, priority #8) Show what actions produce and are worth — crafting rows hide the Craft button, the inputs, and the fact that a bake yields both the loaf and its coin value. The rest reclaim buried CTAs on mobile and stop empty/zero rows from out-massing real content.

SC4Tile Collection & research

SC4S2Effort M
Turn the Almanac from a mislabeled catalog into a legible discovery checklist

The tile wiki is where the game teaches its deepest system — tile variants, research, and the active-tile slot. But its collapsed rows say only "Locked / Unlocked / Active", the Mining tabs are blank (see SC1), and a locked tile's whole story ("how do I get it?") lives one tap deep in the detail pane. The catalog data already carries the answer for every method — chain N, research N/M, buy N◉, build X, reach zone Y (effects.ts statusFor) — it's just not surfaced on the row.

NOWstate word only; how-to hidden
PROPOSEDthe discovery verb on the row
Implementation
  • Extend CategoryRowViewModel with a short rowStatus string + a progress pair, computed from the same discovery fields statusFor already reads (effects.ts:203).
  • Render it as the subtitle (and an optional inline bar for research) in TileListItem (tileCollection/index.tsx:189), replacing the flat Active/Locked/Unlocked string at tileCollection/index.tsx:197.
  • Fold in the SC1 "???" hint and category-label re-key here so the whole Almanac lands in one pass.

SC5Crafting

SC5S1Effort M Top priority #8
Surface the Craft action, the inputs, and the output value on the recipe row itself

On a phone every recipe is a bare row — icon, name, "Ready" — and the actual craft (inputs, have/need, the Craft button) is hidden behind a tap that expands the accordion body (ExpandRow renders no body while collapsed, ExpandList.tsx:168; the body is RecipeCraftBody, crafting/index.tsx:224). Worse, no row — collapsed or expanded — ever shows what the craft produces or its coin worth: the CostGrid is fed inputs only (crafting/index.tsx:243), and no row shows the produced item or what it's worth — a Bread Loaf sells for 125◉ (ITEMS.bread.value, constants.ts:410) and also doubles as a build ingredient. (The recipe grants the item only; the schema's coins field is dead — coins += (recipe.coins ?? 0) at crafting/slice.ts:86, but no recipe sets coins, verified across RECIPES — so surface the item's sell value, not that unused field.) The player commits blind and never learns the payoff.

NOW — real rendername + "Ready", nothing else
PROPOSEDinputs + Craft + payoff on the row
Implementation
  • Add the inputs + a Craft button to the collapsed header. Either pass a trailing/headerExtra slot into ExpandRow (ExpandList.tsx:152) that survives the collapsed branch at ExpandList.tsx:168, or render RecipeExpandRow (crafting/index.tsx:265) as a custom row rather than a bare ExpandRow.
  • Add an explicit outputs line: feed CostGrid (or a sibling "Yields" grid) the produced item plus its sell value getItem(recipe.item).value alongside inputs in RecipeCraftBody (crafting/index.tsx:229) and RecipeDetail (crafting/index.tsx:187). Do not read recipe.coins — it's unused (see the new finding below); either wire it as a deliberate craft-for-coins bonus or delete the dead field.
  • Fix the locked-station dead end: at crafting/index.tsx:541 the "Build the {station} in town" copy is plain text — add a button that routes to the Town build flow for that station.

Also folded in (S3 / polish)

  • No batch / repeat craft. Crafting 12 bread is 12 taps + 12 "Crafted!" bubbles. Add a quantity stepper or a "Craft ×N / Craft max" affordance on the row.
  • Ambiguous status vocabulary. "Ready", "Missing inputs", "No station", and the header pill "2/2 ready" / "Locked" (crafting/index.tsx:160, :521) overlap in meaning; standardize one status lexicon shared with the tile wiki.
  • Empty canvas under a short list. A two-recipe bakery leaves ~60% of the phone blank (visible in the NOW shot). Center a short list or add a "what unlocks more recipes" hint so the space reads as intentional.

SC6Inventory

SC6S1Effort S
Sort the storehouse by what you actually have, and stop empties out-massing real stock

The Storehouse hardcodes sort="alpha" (inventory/index.tsx:209), overriding the grid's own count default — so a stack of 40 wheat sorts below a resource you own zero of, purely by name. And every biome resource is listed whether or not you own any (the resource filter never checks count>0, ui/Inventory.tsx:553), so the list is padded with empties while the header pill counts only non-empty kinds — "N kinds" over a visibly longer list.

NOWalpha; zeros interleaved
PROPOSEDcount-desc; empties tucked
Implementation
  • Drop the sort="alpha" override at inventory/index.tsx:209 so the grid uses its sort="count" default (ui/Inventory.tsx:426), or expose a small sort control.
  • Filter zero-count biome resources out of the default view in visibleResourceKeys (ui/Inventory.tsx:553), behind a "show not-yet-gathered" toggle; items and tools already require count>0 (ui/Inventory.tsx:447).
  • Confirm the header pill {distinctItems} kinds (inventory/index.tsx:175) now equals the visible row count.

Also folded in (S3 / polish)

  • The grid/list toggle is pixel-identical clutter. The list and grid modes (inventory/index.tsx:192) render nearly the same on a phone; either differentiate them meaningfully (dense chips vs. detailed rows) or drop the toggle.
  • Fractional order progress never surfaces. A shared ProgressTrack primitive exists (ui/primitives/ProgressTrack.tsx) but the inventory order state renders as a bare wrapping word; show the "3/5 filled" bar the component already supports.
  • Filters are by catalog kind, not intent. "Resources / Tools / Items" (inventory/index.tsx:12) don't answer the real questions ("what can I sell?", "what does an order need?"); add intent filters (sellable, order-needed) alongside kind.

SC8New findings

SC8NEWfrom this pass
Surfaced while grounding these mockups in the panel code

Not in the original report; found reading the screen code for this pass. Each was re-verified against the live source (including one initial reading that did not hold up — see the note).

  • The recipe coins field is dead code. grantCraftOutput adds the crafted item (crafting/slice.ts:80) and does coins += (recipe.coins ?? 0) (crafting/slice.ts:86), and the RecipeDefinition type declares an optional coins? (crafting/index.tsx:37) — but no recipe in RECIPES ever sets coins (verified across constants.ts:934–1000; rec_bread is just {item, station, tier, inputs}). So the branch always adds 0: a harmless-but-misleading dead slot. Decide its fate — either wire it as a deliberate craft-for-coins bonus (and show it on the row), or delete the field so nobody mistakes it for a live reward. (An earlier reading of this pass claimed crafting "double-grants" the item plus its 125◉ sell value; on inspection that's false — recipe.coins is never populated, and the loaf's 125 is its sell value realized only when you sell it. Corrected here and in the SC5 proposal, which now sources the shown value from getItem(recipe.item).value.)
  • The Almanac claim button can enable an unclaimable tier. Because the UI's ×100 unlock math (quests/index.tsx:212) is looser than the reducer's ×150 level gate (almanac/data.ts:120), for higher tiers the CLAIM button lights up at an XP the reducer rejects, so the tap no-ops with no feedback (fixed by SC3, called out here as its own defect).
  • Mining wiki tabs also drop treasure entirely. Beyond the tile_-prefix key drift (SC1), the coin-tile category treasure (tileCatalog.ts:64) has no entry in CATEGORY_LABELS at all, so its tab is blank even after the prefix fix — the re-key must add it.
  • The "recent" inventory sort is dead on the Storehouse. useRecentOrder tracks deltas and is threaded through as recentOrder (inventory/index.tsx:139), but with sort="alpha" hardcoded the recency data is computed and never used — SC6's sort fix also reclaims this.
