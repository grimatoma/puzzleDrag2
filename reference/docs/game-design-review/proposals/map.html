Proposals · Core surface 03

World Map — fully-defined proposals

Say it plainly first: the map is the best-realized screen in the game. A cohesive parchment atlas — watercolor terrain, themed medallions, solid-vs-dashed roads, drifting smoke over lit hearths, a real compass rose. It's the one surface that already fills a widescreen and reads as a place. None of these proposals is a redesign. They're about the two spots where the craft doesn't reach the edge: the mobile detail panel (where every decision is actually made) and node legibility (so the right two-thirds of the atlas stops reading as interchangeable grey discs).

Keep, don't touch. The desktop composition, the terrain/region painting, the ember-smoke language, the road states, and the panel's storybook voice all land. MapScene.ts is a model for how the rest of the game should feel. Everything below layers onto that, changing behavior in a handful of functions.

BENCHMARK — real renderdesktop, the surface to protect

Map through-line. Two S1 fixes carry the value: (M1) un-cap the mobile panel and pin the load-bearing CTA to a non-scrolling footer so "Found this hearth" / "Walk the road" is never below the fold; and (M2) give the tapped node a persistent selection ring so the panel and the map stop being two disconnected regions. Then (M3–M6) make undiscovered nodes read as places, label the mystery-meat "i", true up the stale legend, and give the permanent keeper choice the ceremony it deserves.

M1S1Effort M Top priority #9
Mobile: un-cap the detail panel and pin the primary CTA to a non-scrolling footer

Every map decision — found, walk, fast-travel, face keeper — happens in the detail panel, but on portrait that panel is capped at 38% of the map area. Lore + subtitle + zone-info + status chip + activities + the settlement block all live in an overflow-y-auto region, and the load-bearing ActionButton is pinned mt-auto at the very bottom — so on a 390×844 phone the button that the whole screen exists to press is below the scroll fold. An always-on legend eats another ~90px underneath. Portrait rail maxHeight:"38%" (index.tsx:805); NodePanel is overflow-y-auto with the CTA at mt-auto (index.tsx:572); LegendBar renders always (index.tsx:813).

NOW — real renderkeeper choice, 38% panel
PROPOSED — Option Apanel ~55%, CTA pinned
Options — how to split the portrait screen
A · Fixed 55/40 + pinned CTA footerRecommended
Give the panel ~55%, the map ~40%; move the CTA into a non-scrolling footer; legend → a "Key" chip that expands on tap. Pro: smallest change, CTA is guaranteed visible, the map still shows the tapped node in context. Con: a fixed split isn't optimal for the lore-heavy nodes (they scroll).
B · Draggable bottom sheet
Panel becomes a sheet with peek / half / full snap points the player drags; CTA pinned to the sheet header. Pro: the map can go near-full when browsing, the panel near-full when deciding. Con: a real gesture component to build and test; more than the finding needs.
Implementation
  • Raise the portrait rail from maxHeight:"38%" to ~55% and let the map area take the rest (src/features/cartography/index.tsx:803).
  • In NodePanel split the body: keep overflow-y-auto on a lore/activities/status wrapper, then render the <FoundSettlementBlock/> + <ActionButton/> in a sibling footer outside the scroll region — drop the mt-auto (src/features/cartography/index.tsx:510–579).
  • Replace the always-on <LegendBar/> with a collapsible "Key" disclosure (src/features/cartography/index.tsx:681,813); default collapsed on portrait, open on landscape/desktop where there's room.
  • Add an e2e assertion that on 390×844 the primary ActionButton rect is fully within the viewport for a tapped node (tests/e2e).
M2S1Effort M
Give the tapped node a persistent selection ring so the panel and the map connect

Tapping a node opens its panel, but the only on-map response is a transient scale bounce that yoyos straight back to 1.0 — gone before your eye reaches the panel. Nothing on the map stays marked, so with ~13 near-identical medallions the player can't map "Greenmeadow" in the panel back to a location. That's the one thing a map+detail pattern must do. applyTappedHighlight is a scale:1.08, yoyo, repeat:1 tween back to 1.0 (MapScene.ts:1003); the selected id already flows through the carto.tapped registry key.

NOWbounce, then nothing
PROPOSEDsticky ring + pin, others dim
Implementation
  • Thread the current carto.tapped id into paintNodeStatus (or a dedicated paintSelection pass) so it draws a persistent dashed selection ring + a caret sprite on the selected node (src/features/cartography/MapScene.ts:887).
  • In applyTappedHighlight, keep the bounce but stop treating it as the only signal — clear/redraw the selection ring on the previous and new node, and drop a container alpha (~0.55) on every non-selected node group while a selection exists (src/features/cartography/MapScene.ts:1003).
  • Optional: on select, ease the camera so the node is comfortably in view (ties to M7 · N1) so on a zoomed/portrait map the selection is never off-screen.
M3S2Effort M
Undiscovered nodes: region-tinted silhouettes instead of identical grey "???" discs

Half the atlas is undiscovered, and every hidden node collapses to the same look: the baked icon forced to alpha 0.10, a faint lantern-mark, and a "? ? ?" label under a low-contrast brown ring. So the right two-thirds of this beautiful map reads as "locked slot #4, #5, #6" — interchangeable circles — instead of a quarry, a harbor, a capital worth walking toward. That kills the aspirational pull that justifies the whole surface. Hidden branch forces icon.setAlpha(0.10) + label.setText("? ? ?") (MapScene.ts:929); hidden ring is 0x8a6a50 for all of them (MapScene.ts:924).

NOWfour identical grey discs
PROPOSEDregion silhouette + varied frame
Options — how much to reveal before discovery
A · Region silhouette + tinted frameRecommended
A generic per-region glyph (pickaxe/anchor/antlers/crown) at low alpha on the region's terrain tint; keep a "???"-flavored half-name. Pro: preserves the sense of the unknown while making regions read as territories; one glyph per region, cheap. Con: not the node's exact identity (that's the point — it's still undiscovered).
B · Dim the real baked icon
Show each node's actual cartoIcon_* at ~0.35 alpha instead of 0.10 + "???". Pro: zero new art — the icons already exist. Con: spoils the exact destination before you've earned it; loses the "fog of war" tease.
Implementation
  • In the hidden branch of paintNodeStatus, stop forcing alpha 0.10 + "? ? ?" for regioned nodes; instead swap the medallion icon to a per-region silhouette texture and tint the disc from NODE_TERRAIN[node.id].palette (src/features/cartography/MapScene.ts:929, MapScene.ts:124).
  • Bake one silhouette texture per MapRegionId in bakeNodeTextures (reuse the existing REGION_ICON mapping in src/features/cartography/index.tsx:38), alongside the current lantern-mark (MapScene.ts:229).
  • Vary the hidden ring color by region rather than the single 0x8a6a50 (MapScene.ts:924); keep the flat grey path only for kind:"event" / truly-unknown nodes.
M4S2Effort S
Label the zone-info "i" — the map's most information-dense action is mystery meat

The single richest action in the panel — spawn rates, seasonal drops, hazards, and entry cost — is a bare 32px white circle with a lowercase "i", floating with only an aria-label to say what it does. On touch there's no hover, so a player has no reason to believe that unlabeled dot hides the whole zone briefing. The affordance is a w-8 h-8 rounded-full button whose only visible content is "i" (ZoneEntryCostInfo.tsx:33).

NOWunlabeled 32px circle
PROPOSEDlabeled row + at-a-glance strip
Implementation
  • Replace the bare circle button in ZoneEntryCostInfo with a labeled full-width disclosure row ("ⓘ Spawn rates & hazards") that opens the same ZoneInfoModal (src/features/zones/ZoneEntryCostInfo.tsx:31–43).
  • Add a compact stat strip above it in the panel — dominant seasonal drop, hazard names, entry cost — sourced from the zone's boards/dangers/entryCost (src/features/cartography/index.tsx:528–532, src/features/cartography/data.ts:62).
  • Keep aria-label but back it with a visible label so it works on touch and passes the "unlabeled control" a11y lens.
M5S2Effort S
Rewrite the legend to match what the map actually paints

The legend has drifted from the renderer. It says "Level-locked" — but gating moved to the zone tier ladder ("Grow Hearthwood Vale to City"). It reuses the same red dot (#bb3b2f) for two different entries. And it omits every state the map does draw: the settled wreath, the moss-green coexist tint, the iron driveout tint, and the Hearth-Token seals on the Old Capital. A key that misdescribes the map is worse than no key. Entries hard-coded in LegendBar — "Level-locked" and the duplicate #bb3b2f dots at index.tsx:687–689; the states it omits are painted in MapScene.ts:987–1000.

PROPOSEDtrue to the renderer, as a "Key"
Implementation
  • Rewrite the <Legenda/> entries in LegendBar: rename "Level-locked" → "Tier-locked", give it its own swatch, and add Settled / Coexisting / Driven-out / Hearth-Token rows (src/features/cartography/index.tsx:681–704).

M6S2Effort M
Give the permanent keeper choice real ceremony — and echo the consequence back

Each settled hold hosts a keeper (the Deer-Spirit, the Stone-Knocker, the Tidesinger) who offers an irreversible branch: Coexist (+5 Embers, the land stays rich) or Trial / drive-out (+5 Core Ingots, the hold is yours alone). Today it's entered from a small low-ceremony purple button and framed as two reward chips — a vending machine for a decision that permanently shapes a settlement. Nothing signals "this is permanent," and once chosen the lasting difference barely reads. Entry button + reward-chip framing in FoundSettlementBlock (index.tsx:280); the resolved state is a small chip at index.tsx:267.

NOWa purple button + two reward chips
PROPOSEDencounter with weight + a lasting echo
Options — how much ceremony
A · Enriched dialog + permanence framingRecommended
Keep the existing KeeperEncounterModal but add the portrait header, the "permanent" banner, and a world-effect line under each path. Pro: reuses the built dialog and the keeper data (intro/pitch already exist); ships fast. Con: still a modal, not a set-piece.
B · Full-screen story takeover
Route the encounter through the story-beat system as a full-screen moment (the intro lines already read like a beat). Pro: maximum weight for a signature, irreversible choice. Con: more wiring (story slice + effects); sequence it after A if the encounter tests well.
Implementation
  • Enrich KeeperEncounterModal with a portrait header + a "This choice is permanent" banner, and add the lasting-effect line to each path button (src/features/cartography/index.tsx:108–191); source copy from the keeper's coexist.pitch/driveout.pitch (src/keepers.ts:25–40).
  • Elevate the entry: replace the small purple "Face" button with a bordered, portrait-bearing prompt card (src/features/cartography/index.tsx:280–294).
  • Strengthen the resolved echo chip so "Coexisting / Driven out" reads as permanent and its swatch matches the map tint from M5 (src/features/cartography/index.tsx:267–279).

M7NEWfrom this pass
New findings surfaced while grounding these mockups

Items not in the original report, found reading MapScene.ts and index.tsx for this proposal pass. Verify against the live build before scheduling.

  • Tapping a node never moves the camera to it. applyTappedHighlight only runs a scale bounce — it never pans/centers the tapped node (MapScene.ts:1003), and there's no recenter or zoom-to-fit control (camera goes up to maxZoom:8). On a pinch-zoomed or portrait map you can tap a node from the panel/URL and be reading about a place that's entirely off-screen. Pair with M2: on select, ease the camera so the node is comfortably in view, and add a small "recenter" affordance.
  • No keyboard / assistive-tech path to select a node. Selection fires only from Phaser pointer zones (gameobjectup → carto.nodetap, MapScene.ts:209); the canvas exposes no focusable, labeled node list, so the entire map is unreachable without a pointer. The panel is React (partly accessible) but you can't get to a node's panel via keyboard. A parallel focusable node list (buttons that dispatch the same SET_VIEW_PARAMS) fixes it and also helps M4's labeling story.
  • discovered-unreachable has no distinct medallion. In paintNodeStatus the ring-color chain covers here / visited / ready / locked / capital / hidden but not discovered-unreachable, so it falls through to the default brown ring and a 0.78 icon — visually a plain discovered node (MapScene.ts:918–924). Only the panel disambiguates "↯ No road from here". Give it its own subdued ring (e.g. a dashed cool-grey) so "you can see it but there's no road" reads on the map itself.
