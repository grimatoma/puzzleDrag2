Proposals · Core surface 06

Shell & onboarding — fully-defined proposals

The board is the game; the shell is how a player finds the game. Today the shell hides most of what has already been built: seven full-screen systems have no door except the Debug menu, the modal chrome speaks two dialects, and the first-run tour points its spotlight at things that aren't on screen. Each proposal pairs the game as it is now with a rendered mock of what to build, and a concrete file:line spec.

Shell through-line. One change carries the page: (SH1) the navigation hub — give every built system a real door and stop the 6-tab nav from capping the game's future. Around it: (SH2) turn Townsfolk & Quests into real sub-hubs and fix the taxonomy; (SH3) give the main menu weight and thumb-reach; (SH4) one modal shell + one status vocabulary; (SH5) give NPCs a face and the daily reward its ladder back; and (SH6) aim the tutorial spotlight at elements that actually exist. None is a rewrite.

SH1S1Effort M Top priority #1
Build the navigation hub — give every finished system a door players can reach

Seven full-screen systems — Orders, Castle, Foes (bosses), Portal, Decorations, Tiles, Chronicle — are reachable only from the Debug menu's ten shortcut buttons (features/debug/index.tsx:31; a repo-wide grep for each view:"…" outside debug/visual-tests finds no other dispatch). Charter loops off Chronicle (both debug-only); Boons hides behind a story-gated corner button that only appears after you've faced a keeper (Town.tsx:376). Meanwhile the bottom nav is a fixed 6 tabs — Town, Inventory, Craft, Map, Townsfolk, Quests — with no overflow strategy (ui.tsx:69–74) and the ☰ menu is a flat 10-button stack. There is nowhere to put a new system, so finished content ships invisible.

NOW6 fixed tabs + a flat menu
PROPOSED — Option A5 core tabs + a structured "More" sheet
Options — where the overflow lives
A · 5 tabs + a "More" sheetRecommended
Keep 5 core-loop tabs; a "More" tab opens a grouped bottom sheet (shown). Pro: the core loop stays one thumb-tap away; the sheet is a scalable, sectioned home with room for progressive disclosure and unlock chips. Con: secondary systems are two taps deep — acceptable for systems you visit occasionally, not every turn.
B · A full "Systems" hub tab
Replace one tab with a full-screen hub view (a grid of system cards with art + status), not a sheet. Pro: more room per system, richer cards, a real destination. Con: a whole new screen to build and keep in sync; heavier than a sheet for what is mostly a launcher, and it spends a scarce tab slot on navigation rather than a system.
Implementation
  • Cap src/ui.tsx:64 BottomNav at 5 <Tab>s for the core loop and add a 6th More tab that dispatches OPEN_MODAL {modal:"more"} instead of a SET_VIEW.
  • Add a src/features/more/ feature (modalKey:"more") whose sheet reads a static HUB_SECTIONS map (Systems / Settlement / App → the real viewKeys currently hard-coded in features/debug/index.tsx:31), and hides any row whose unlock predicate is false (progressive disclosure).
  • Move the ☰ menu's non-system entries into the sheet's App section so there is one launcher, not two (features/settings/index.tsx:148 — see SH3).
  • Guard-rail: add a test/CI check asserting every viewKey in the feature glob has at least one non-debug dispatch that reaches it (walk HUB_SECTIONS + nav + known links). This is what prevents a future system from silently shipping Debug-only again.

The sub-navigation (Townsfolk & Quests becoming real hubs) and the taxonomy fixes are SH2; the menu-weight pass is SH3.

SH2S2Effort M
Make Townsfolk & Quests real hubs — and fix the names that mislead

The Townsfolk tab promises the whole social system but delivers only the Workers roster, while NPCs, Foes and the Castle are homeless. The goal systems are scattered: Daily quests, Orders (half of which live in Inventory), and Trophies (buried in ☰) never sit together. And the words collide: "Almanac" names three different things (the Lv pill, a Quests XP ladder, and an empty almanac feature), while the menu title says "Hearthlands" and everything else says "Hearthwood Vale". Nav labels at ui.tsx:69–74; menu title settings/index.tsx:151 vs HUD label Hud.tsx:142.

NOWone tab = one sub-feature
PROPOSEDa hub with a sub-tab bar
Implementation
  • Give src/features/townsfolk/index.tsx an internal sub-tab bar (Workers | Folk | Foes | Castle) that mounts the existing bosses and castle view bodies as panels instead of separate viewKeys.
  • Do the same in src/features/quests/index.tsx (Daily | Orders | Trophies | Almanac); fold the Inventory-side order list and the Debug-only orders / achievements views into it.
  • Taxonomy: rename the nav Tab label "Townsfolk" (ui.tsx:73); make the ☰ title read the same product name as the HUD (settings/index.tsx:151); reserve "Almanac" for the compendium and rename the XP ladder "Levels", deep-linking the Lv pill (Hud.tsx:248) to it.
SH3S2Effort S
Give the main menu weight — and put its trigger within thumb reach

The ☰ menu is an undifferentiated vertical stack of ten equal-weight buttons — Show Tutorial, Fullscreen, Settings, About, Trophies, Town Hall, Debug, Update, Wiki, Docs — so a destructive dev tool sits the same size and shade as "About". The trigger is pinned to the top-left corner, the hardest reach on a phone held right-handed. Stack at settings/index.tsx:155–205; ☰ button is the first HUD child, top-left, at Hud.tsx:173–179.

NOW — real rendermobile ☰ menu
PROPOSEDgrouped by weight, thumb-anchored
Implementation
  • Restructure MainTab (settings/index.tsx:148): a header row, one variant="ember" Settings button, a 2-column grid of quiet utilities, and a footer version line.
  • Gate the Debug entry behind a long-press (onPointerDown + timer) on the version line instead of a first-class ActionBtn (settings/index.tsx:180); keep the tutorialDisabled flag reachable there.
  • Move/mirror the trigger for thumb reach: keep ☰ top-left for AT/landscape but add a bottom-anchored affordance (or fold it into the SH1 "More" tab) so the menu isn't a top-corner-only reach on tall phones (Hud.tsx:173).
SH4S2Effort M Top priority #7
Adopt one modal shell + one status vocabulary — and bake the close X into it

Two modal dialects live in the same app. Debug and Menu render a plain ParchmentDialog.Body with their own centered heading and a bolt-on top-right × (settings/index.tsx:349–356, debug/index.tsx:60–71). Season and Daily use ParchmentDialog.Title — a heading with a hairline divider — and no close X at all, dismissing only by backdrop/ESC (Modals.tsx:43, dailyStreak/index.tsx:15–16). The shell itself provides no built-in close affordance (Dialog.tsx:120–165), so every modal improvises. Status words are just as loose: "Idle", "Assigned", "need", "locked" appear ad hoc with no shared chip.

NOWtwo shells, two dismiss rules
PROPOSEDone shell · one X rule · one chip set
Options — the close-affordance rule
A · X for exploratory, CTA for transactionalRecommended
Panels you browse (Debug, Menu, Town Hall) get the dismissable X; modals that resolve an action (Daily "Collect", Season "Return to Town") lead with the CTA and also accept backdrop/ESC. Pro: the affordance matches intent; no modal is a dead-end. Con: a per-modal judgement call — encode it as the default of the shell prop.
B · X on every modal, always
Ship the X on all of them unconditionally. Pro: zero ambiguity, one habit. Con: a redundant X above a single "Collect" button adds a second, competing exit and clutters small reward modals.
Implementation
  • Add a dismissable prop to src/ui/primitives/Dialog.tsx:120 that renders a shared 44×44 close button in the panel corner (reusing FeaturePanel.CloseButton), so no modal hand-rolls one.
  • Delete the bespoke CloseButton + centered text-[18px] headings in settings/index.tsx:352 and debug/index.tsx:62; route both through <ParchmentDialog.Title> with dismissable.
  • Promote the four .gm-status chips (ready / missing / locked / done) to a shared <StatusChip> and replace ad-hoc words ("Idle", "Assigned", "need") across features.
  • Apply the 44px tap-target token to the shell's close button and all modal footer CTAs.
SH5S2Effort M
Give NPCs a face, the news a home, and the daily reward its ladder back

NPC dialogue and Market News both surface as portrait-less top toasts. NpcBubble hands the notifier a label string ("Mira · Cook") where a portrait key should go, so the speaker never gets a face (Modals.tsx:616–621), and Market News is a 4000ms bubble that self-destructs (state.ts:1133) — a market you can't consult. And the Daily Reward modal renders only today's line, throwing away a rich 30-day ladder with real milestones at days 3/5/7/14/30 (dailyStreak/index.tsx:16–24 renders one day; DAILY_REWARDS is keyed 1–30 at constants.ts:1142).

NOWfaceless toast · today-only reward
PROPOSEDspeaker chip · ticker pill · day strip
Implementation
  • In src/ui/Modals.tsx:618 pass npcKey: bubble.npc (the real key) and move the name · role string into a separate label field so the toast can render the portrait via NPCS[key].look.
  • Split Market News out of the ephemeral bubble path (src/state.ts:1133) into a persistent marketNews state field; render it as a dismissible ticker pill in the HUD / above the nav rather than a 4000ms toast.

SH6Onboarding — aim the spotlight at things that exist

The nine-step first-run tour (features/tutorial/steps.ts) walks the core loop, but two of its spotlight steps target elements the player can't see, so the tour's most confident moments — a glowing ring that says "look here" — point at nothing or at the wrong control.

SH6aS1Effort S
"Seasons & turns" spotlights an off-screen element — anchor it to the visible strip

The turns step targets [data-testid="turns-left"] (steps.ts:78). That testid exists only as two hidden numeral mirrors positioned at left:-9999px (puzzleBoard.tsx:133 and seasonStripPhaser.tsx:158) — the real season strip is a Phaser canvas with no DOM anchor. readRect rejects only zero-size nodes, not negative coordinates (tutorial/index.tsx:284), so a −9999px span passes the guard and the ring is drawn far off-screen, leaving a fully-dimmed board with a coach card pointing at emptiness.

NOW — real renderring off-screen, board fully dimmed
PROPOSEDring on the real HUD strip
Implementation
  • Add data-tour="season-strip" to the visible SeasonIndicator in the HUD (Hud.tsx:205) and retarget the step (steps.ts:78) to it.
  • Harden readRect (tutorial/index.tsx:284) to also return null when r.left < 0 or r.top < 0 (or right/bottom fall off the viewport) so an off-screen node degrades to the centered card instead of an invisible ring.
SH6bS1Effort S
"Build your first building" points at the nav tab, not a plot — retarget & rewrite

The build step spotlights [data-tour="nav-town"] — the bottom-nav Town tab (steps.ts:98; anchor set in primitives/TabBar.tsx:83), while its copy says "Tap an empty plot, choose a building like the Mill, and build it." The ring sits on a nav icon; the instruction describes a plot the player must find on their own. The step auto-advances on BUILD, so a confused player is stuck staring at the wrong thing.

NOWring on the nav tab
PROPOSEDring on a real plot's Build CTA
Implementation
  • Add data-tour="build-cta" to the first empty/buildable plot (or its Build affordance) in the Town view (src/ui/Town.tsx) and point the step at it (steps.ts:98).
  • Rewrite the step body to reference the highlighted plot directly ("Tap this plot to build the Mill"), and keep the advanceOn:"BUILD" gate.
  • If no plot is buildable yet (can't afford it), fall back to the centered card so the ring never dangles — this pairs with the SH6a readRect guard.

Also folded in (S3 / polish)

  • Narrator is a bare lettered circle — NpcAvatar draws a colored disc with npc.name[0], not the real portrait (tutorial/index.tsx:26–49). Render the actual NPC art (same fix family as SH5's speaker chip) so Wren/Mira/Bram have faces.
  • No "Step 3 of 9" — the tour shows only dot pips (tutorial/index.tsx:52–68); add a numeric "Step N of 9" so players gauge how long the tour is.
  • Puzzle step card can cover the tool rack — the top-preferred placement on a near-full-height board can land the coach card over the tools; bias to a side gap on the board step.
  • Fresh town leads with an un-actionable "Upgrade to Hamlet" sign — the first thing a new caretaker reads is a locked upgrade they can't afford; suppress it until the first building stands.
  • Unexplained 50-coin first entry — the tutorial hands 50 coins with no line about where it came from or what it's for; add a one-line "Here's 50 coins to start" beat.
  • Auto-start fires on the first arbitrary action — the tour can kick off mid-gesture; gate its auto-start on a deliberate entry (first Town view) rather than any first dispatch.
SH7NEWfrom this pass
New findings surfaced while grounding these mockups

Items not in the original report, found reading the shell / tutorial code for this proposal pass. Verify against the live build before scheduling.

  • The visible season strip has no DOM anchor at all — the only elements carrying data-testid="turns-left" are the two hidden left:-9999px numeral mirrors (puzzleBoard.tsx:133, seasonStripPhaser.tsx:158); the real readout is painted on the Phaser canvas. Any tour/AT/analytics hook that wants "the season strip" currently has nothing on-screen to bind to — SH6a's anchor is the general fix.
  • readRect's guard is a latent trap for every spotlight — because it accepts any node with non-zero size regardless of position (tutorial/index.tsx:284), any off-screen-but-sized element (a parked Popover at −9999, a collapsed drawer) would be "spotlighted" invisibly. The turns step is just the one that trips it today.
  • The whole tour silently vanishes if a non-tutorial modal opens — Tutorial returns null whenever state.modal is set to anything but "tutorial" (tutorial/index.tsx:366). A spotlight step sets no modal, so if the player taps ☰ mid-tour the coach card disappears; it only returns when they close the menu. A resume affordance (or pausing rather than unmounting) would keep first-run guidance from feeling broken.
  • Boons can be permanently unreachable in normal play — its only non-debug door is a Town corner button gated on a keeper_*_coexist / _driveout story flag (Town.tsx:376). A player who hasn't faced a keeper has no route to a built meta-progression system except Debug — reinforcing SH1's "every system a door" rule (give Boons an unlock-chipped row in the hub).
