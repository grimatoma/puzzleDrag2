Codebase Health Review — puzzleDrag2

A deep, adversarially-verified pass over the React + Phaser tile-drag match/merge game: dead code, system duplication & drift, wiring footguns, type-safety, doc drift, test health, texture/art bloat, performance, config/balance, and architecture.

Scope reviewed: src/, prototype.tsx, the three Vite entries, tests, config & textures (~260k lines / 973 source files; ~136k of that is hand-authored art). reference/ is supporting material, not reviewed as game code.
Method: 21-agent verified review workflow (10 dimensions, each finder + an independent skeptic) + a focused perf pass + a completeness critic. Ground truth: typecheck ✅ and lint ✅ both pass with zero warnings.

Executive summary

Overall verdict: mostly-healthy. puzzleDrag2 is a well-built codebase. The two hardest architectural boundaries — the React-reducer ↔ Phaser-canvas split (via a typed registry bridge) and the three isolated Vite entries (game / Dev Panel / Story Editor, with Phaser deliberately kept out of the latter two) — are genuinely sound. Slice/action wiring discipline is strong, TypeScript is strict with zero : any, and CI's static gates are green.

The real problems are structural, not surface-level, and cluster into four themes:

  1. Resource-economy single-source-of-truth violations. The same fact (a tile's product, an item's sell price, a category's tier-0 divisor) is hand-encoded in two places with no guard — and several already disagree, producing a real gameplay bug (a tile yields the wrong resource and a quest is uncompletable) and mis-priced goods.
  2. A dead run-summary UI path. The "Story beats" recap can never populate because its action is dispatched only into the story slice, never the top-level reducer chain.
  3. Test-gate blind spots. One test passes only via never-reset shared localStorage pollution (it asserts a reducer path that never runs), and the coverage gate excludes the riskiest logic directories (game rules, save migrations, config).
  4. Module-graph debt. The core tile catalog is misfiled inside a feature folder yet imported by 27 modules (anchoring a 62-file import cycle), alongside a ~1455-line god reducer.

Everything else is low-severity hygiene: a handful of dead modules/handlers, copy-pasted helpers (PRNG, clamp, draw kernels), ~34.5k lines of Dev-Panel-only icon art shipping in the game bundle, and some stale docs. A small set of guard tests (tile-product ↔ produced-resource, value ↔ market sell, lineBase ↔ thresholds, uiMeta ↔ boss targets) would convert a whole class of silent drift into CI failures.

Scope of this document. Originally a report only. A first remediation pass (“wave one”) has since landed the safe, no-gameplay-change subset — see the box below. Each remaining finding still carries its own evidence and a recommended fix so any subset can be actioned later. Where a fix needs a design call (the blackberry product; whether crafted-good prices should rise ~10×), both options are presented rather than chosen.

What's genuinely healthy

Worth stating explicitly so these aren't mistaken for unreviewed — each was checked against the code.

React ↔ Phaser boundary

GameScene never imports the reducer; it reads a fully-typed registry contract (src/types/phaserRegistry.ts) and emits back via SCENE_EVENTS. Async boot/teardown guards against the unmount-mid-load race.

Three isolated Vite entries

vite.config.js manualChunks deliberately keeps Phaser out of the /b/ (Dev Panel) and /story/ bundles; both pull zero Phaser.

Slice/action wiring

All 18 feature slices are imported and registered in the slices array; every slice-exclusive action is in SLICE_PRIMARY_ACTIONS; all 18 slices preserve referential equality for no-ops; the action-type catalog is in sync (93 types).

Type safety & static gates

Strict tsconfig, zero : any annotations, no @ts-ignore outside tests, all 16 eslint-disable justified. typecheck and lint pass clean.

PWA / service worker

autoUpdate + skip-waiting, a singleton update poller with HMR-disposed listeners, a controllerchange double-reload guard, and a navigate-fallback denylist that correctly excludes /b/, /story/, /docs/.

Accessibility & secrets

Dedicated a11y.ts + a11y.test.tsx, aria across ~94 components, a feature error boundary, prefers-reduced-motion honored. No credentials/API keys in shipped src/.

Optimization findings

The runtime is broadly well-optimized: Phaser and GameScene are lazy-imported (import("phaser")), the town canvas is kept mounted/hidden to avoid cold reboots, the per-frame GameScene.update() loop is throttled (≤33ms seasonal / ≤50ms concept re-bakes) with distinct-key dedup and reduced-motion gating, and persistState is rAF-coalesced. The real wins are React-side.

Operational gaps (outside the 10 dimensions)

Surfaced by a completeness critic — categories the dimensional review didn't target. Each was spot-checked against the code.

Dependency advisories low (dev-only)

npm audit reports 3 — two HIGH on vite <=6.4.2 + an undici cluster, all dev/build-time transitive deps (not shipped to players). React 18 / Phaser 3 / Tailwind 3 / Vite 6 / TS 5 are each a major version behind. A genuine blind spot, but player-facing risk is low.

Silent save-quota failure medium

persistStateNow serializes the entire state every frame; on QuotaExceededError it logs one console line and silently stops saving — no user-facing signal, no size budget. A save-size guard test belongs here (same class as the economy guards).

PWA precache size low

globPatterns precaches all PNG/GIF/WebP plus the ~34.5k lines of Dev-Panel-only icon art; only a per-file cap exists, no total precache budget. Worth a one-time check of the generated sw.js manifest size.

Checked & healthy

PWA/service-worker correctness, async Phaser teardown, memory/listener cleanup, accessibility, and secret-hygiene were all spot-checked and are in good shape. i18n is a deliberate non-goal for a single-developer English game.

How to read this & suggested order of operations

Highest value for least risk: the four guard test opportunities (tile-product ↔ produced-resource, value ↔ market-sell, lineBase ↔ thresholds, uiMeta ↔ boss targets) plus the single global localStorage.clear() in test setup. These are small, safe, and convert silent drift into CI failures — do them first, even before fixing the bugs they guard.
  1. Decide the two economy design calls (#1 blackberry product, #2 crafted-good prices), then fix + add their guards.
  2. Fix the dead run-summary path (#3) and the test pollution (#4) — both correctness/clarity, low effort.
  3. Safe deletions (#11, #16, #17, #18, plus the dead abilities/handlers) — zero behavior change, shrinks surface area.
  4. Doc fixes (#14, #15) — the feature README is actively misleading for the task it governs.
  5. Widen the coverage gate (#6) and add tests/**/*.test.tsx to the vitest include (#20).
  6. Quick perf wins (OPT-2, OPT-4) are small; OPT-1 / OPT-3 / OPT-5 + #5 / #9 / #10 are larger refactors to schedule deliberately (run visual goldens for any UI-touching change).

Verification discipline: every finding was produced by a finder and then independently re-checked by a skeptic that re-grepped/re-read the cited code and refuted false positives (dead-code claims were cross-checked against import.meta.glob feature auto-discovery, string-keyed dispatch, the Phaser registry bridge, router keys, and texture keys). The two headline economy/wiring bugs were additionally confirmed first-hand.
