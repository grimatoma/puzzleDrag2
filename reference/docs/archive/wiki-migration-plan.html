Overview

The question that started this: “Have all the docs been migrated into the wiki, backed by their relevant configs?” The honest answer is partially. The game's story and design content was already migrated — it lives as authored narrative pages in the wiki and as concept pages generated live from config. What remains on disk is a small set of contributor / code-reference docs that describe how the codebase works, not what the game is.

TL;DR. The wiki is the source of truth for game knowledge; the markdown under docs/engineering/ and src/ is the source of truth for code knowledge. The work here is to state that boundary explicitly, delete the one place the two duplicate each other, fix stale links, and lock in the Overview page as the wiki's front door.

Two sources of truth, one boundary

The ambiguity behind “how much should move to the wiki?” dissolves once you name what each surface is for. They are not competitors; they answer different questions.

The test. Ask of any doc: “Would a player or a designer care about this, or only someone editing the source?” Player/designer → wiki. Source-editor only → code docs. Wiki content is generated from config wherever the data exists, and authored as narrative only for the prose the config can't hold (the why, the world, the plan).

Why the code docs do not move into the wiki

It is tempting to read “make the wiki the source” as “put everything in the wiki.” That would re-introduce exactly the drift the wiki was built to kill. The wiki's whole value is that its catalog is generated — a hand-written “how to add an enum” page pasted into it would be unmaintained prose with no config behind it, indistinguishable from the markdown it replaced. The engineering docs document code mechanics that have no config representation and that no player/designer reads. They stay code-side, and the wiki links to them from its developer-facing notes rather than absorbing them.

Update — catalog enums folded into CLAUDE.md. The former docs/engineering/catalog-enums.md (enum/attribute split, add-an-item workflow, parity tests, and the concept → enum → source map) was base knowledge agents need, so it now lives in the “Catalog enums” section of CLAUDE.md and the standalone doc was removed. That's a third surface — agent base knowledge — distinct from the wiki (game) and the longer-form contributor docs. The concept map is the contributor counterpart of the wiki inventory (which TS enum / source map backs each concept), not a duplicate of it.

The PROGRESS.md contradiction (read before deleting)

The orphan-cleanup instinct was to delete or trim docs/iso-buildings/PROGRESS.md. On inspection that is wrong: it is referenced — and actively written to — by the iso-building skill workflow:

.claude/skills/iso-building/SKILL.md         (×4: tracker, plot-tier, PR step, sub-agent loop)
.claude/skills/iso-building/builder-prompt.md   "Add/update this building's row in …PROGRESS.md"
.claude/skills/iso-building/critique-prompt.md  "update the PROGRESS.md row …"

Deleting or trimming it would silently break the skill's documented build→critique→approve loop. So this plan keeps PROGRESS.md as-is and surfaces the finding rather than honoring the delete. If you still want it gone, that's a coupled change to the skill — call it out and it becomes its own task.

Execution — five small, safe changes

No new wiki rendering engine is needed: the wiki already generates the concept catalog, so “migration” here means removing duplication and stating the boundary, not building pages. Each phase is independently committable.

P1 · De-stale CLAUDE.md fix

Remove/repoint the two references to the missing docs/engineering/ts-migration-completion.md (line 22) and docs/engineering/typed-tests.md (line 37). Point line 37 at the live npm run typecheck:tests script.

P2 · Write the boundary down edit

Add a short “Docs map” note to CLAUDE.md and a sentence to the wiki Overview's “How to use this wiki” section: game knowledge → wiki; agent base knowledge → CLAUDE.md; longer-form contributor reference → markdown by the code + the published docs site. This is the durable answer to “what's the purpose of these?”.

P3 · Consolidate the enum reference into CLAUDE.md moved

The catalog-enum reference is base knowledge agents need, so its whole body — enum/attribute split, add-an-item workflow, Zod layer, parity tests, inventory typing, the concept → enum → source map, board-only keys, runtime guards/imports — folds into the “Catalog enums” section of CLAUDE.md; the standalone docs/engineering/catalog-enums.md is deleted and its 3 inbound references (CLAUDE.md ×2, src/types/items.ts) repointed. Separately, balance.schema.md gains a cross-link pointing per-field semantics at the generated wiki concept pages.

P4 · Harden Overview-as-intro test

Add a unit test asserting that an empty hash resolves to the overview narrative page, so the front-door default can't silently regress.

P5 · Keep what should be kept keep

features/README.md, balance.schema.md body, iso-buildings/PROGRESS.md, and the Puzzle Craft 2 reference all stay. PROGRESS.md is explicitly not deleted (see the contradiction note).

Non-goals (named so they don't creep in)

  • Authoring per-entity “About” prose for every concept. The wiki is structurally complete — every entity has a generated lede + infobox + sections; only ~8 have hand-written bodies. Filling the rest is a separate content effort, one content/<concept>/<key>.html at a time, not part of this reconciliation.
  • Folding code-architecture docs into the wiki. Rejected on purpose — see The Boundary.
  • Touching docs/superpowers/ plans/specs. Out of scope by request.
Verification. npm run typecheck + the new Overview test + npm test for the wiki suites. Doc-only edits don't affect game rendering, so the visual golden suite is not triggered by P1–P3.
