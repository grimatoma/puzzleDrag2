Hearth · Procedural Canvas Icons

Icon Style Guide

The shared quality bar every icon must meet. All icons are drawn procedurally with Canvas-2D draw(ctx) functions on a fixed design box — no image assets. This guide defines the geometry, lighting, stroke, and palette conventions that make ~315 hand-coded icons look like one cohesive set, plus the rubric the review pass applies to each one.

The design box

Every draw(ctx) emits paths around the origin (0,0). The shared transform (src/textures/paintIcon.ts, ICON_DESIGN_BOX = 64) translates to the canvas centre and scales size / 64, so the coordinate system each function draws in is a 64×64 box centred on origin: roughly −32…+32 on each axis.

Safe area
Art lives inside a ±26 radius. The only thing allowed to approach the 64-box edge is the ground-contact shadow. Anything that crosses the box edge is clipped or collides with the circular tile background — that is the #1 "LLM-made" artifact this review removes.

Anatomy of a good icon

Icons read at 56px in the game and the wiki. The recurring recipe across the icons that already look right (e.g. tile_fruit_apple):

  1. Ground shadow first — a flat dark ellipse near y≈22 so the object sits on a surface.
  2. Body fill via gradient — a radial gradient with the light source upper-left gives volume.
  3. Outline — a single dark contour stroke defines the silhouette.
  4. Interior detail — thinner strokes/fills for features (stem, leaf, facets).
  5. Specular highlight last — a soft white/translucent blob upper-left.

Geometry & composition

  • Centred & balanced. The visual mass sits on the origin; don't let an icon drift into one quadrant.
  • One clear silhouette. The shape must be recognizable as a solid blob at 56px before any detail is added.
  • No floating parts. Stems, leaves, handles, sparkles must connect to or clearly belong to the body — nothing hovering with a visible gap unless it's an intentional, grounded accent (e.g. a contained sparkle).
  • Close your paths. Every filled shape calls closePath(); stray open sub-paths leave hairline tails.

Stroke widths (design-box units)

RoleWidthNotes
Primary outline / contour2.0 – 2.4One unifying dark contour per object
Interior detail1.2 – 1.6Veins, seams, facets, small features
Hairline accents0.8 – 1.2Use sparingly; can vanish at 56px

Always rely on the shared lineCap:"round" / lineJoin:"round" set by paintIcon — never reset them to butt/miter (spiky corners read as "broken").

Lighting & depth

  • Light from upper-left, consistently. Highlight upper-left, core shadow lower-right.
  • Prefer radial/linear gradients over flat fills for any rounded body.
  • One soft specular highlight, translucent white — not a hard opaque dot.
  • A ground shadow grounds free-standing objects (skip for flat UI glyphs).

✓ Do

  • Art within ±26, shadow only near edge
  • Single dark contour, rounded joins
  • Upper-left highlight, gradient body
  • Reads at 56px as one shape

✗ Avoid

  • Strokes crossing the 64-box edge
  • Scattered radiating lines (the "rambutan hairs" bug)
  • Disconnected floating elements
  • Flat fills with no light direction
  • Busy detail that muddies at 56px
Out-of-bounds = bug
Use the diagnostic render (split light/dark background + magenta 64-box). Any stroke touching or crossing the magenta square is a defect — pull it inside ±26.

Palette

Colours are defined per-icon (there is no central palette module), but they should stay within the game's warm, earthy world and group by biome. Keep saturation moderate — fully-saturated primaries look like clip-art against the muted board.

Contrast
Each icon sits on a faint tinted circle (its color field at ~16% alpha) and on the board. Make the silhouette read on both light and dark — that's exactly what the dual-background diagnostic render checks.

For each icon, open its diagnostic render and answer:

Outcomes recorded in the tracker (icon-review.html): OK (passes as-is), Fixed (redrawn), or Removed (unused & verified — see Workflow).

Review workflow

Driven per category module, with a render → review/fix → re-render → critique loop. The working renders live in the gitignored icon-review/ dir; the tracker embeds thumbnails so it survives their deletion.

  1. Render baseline. node tools/render-icons.mjs --mode before --module <mod> → diagnostic PNGs + contact sheet.
  2. Review & fix. Judge each icon against the rubric; edit the draw() in src/textures/categories/<mod>.ts. Never rename keys or remap item→icon — redraw only.
  3. Re-render. --mode after --module <mod>.
  4. Critique. Confirm each fix is a real improvement, matches siblings, no regression.
  5. Update tracker. Append notes to icon-review/notes.json, then node tools/build-icon-tracker.mjs.
  6. Guard & ship. npm run lint && npm run typecheck && npm test (icon-audit green) + npm run test:visual with justified golden updates. One PR per category.

Removing unused icons

The manifest flags removal-candidate keys (not used by getUsedIconKeys() and not legacy). These are candidates only — the usage scan misses JSX/template-literal references.

Verify before deleting
Grep the key across src/ (incl. [icon:KEY] rich-text and template literals). Confirm it isn't an alias target (a shared draw reachable via another in-use key — the manifest marks these alias-target, never delete). Only then remove the draw + registry entry and mark it Removed. Legacy/archived icons are never touched.
