Board Kinds — Wiki Concept

A new top-level Game Wiki concept giving Farm, Mine, and Harbor their own conceptual pages — tile roster, dangers, seasons & turns, and the zones that use them — over the existing BIOMES config.

Design spec · Status: Approved · 2026-06-02 · scope: Dev Panel Wiki (/b/) only

wiki dev-panel documentation-surface no-new-mechanics

Overview

The Game Wiki documents 23 concepts, but none of them is the page a player or designer actually reaches for when asking "how does the Farm work?" — the tiles it spawns, the dangers it throws at you, how seasons and turns play out, and which map zones use it. This spec adds a new boardKinds concept with three articles — Farm, Mine, Harbor — built live over the existing BIOMES config plus a short authored narrative per kind.

TL;DR. Add a boardKinds wiki concept (3 entries, sourced from BIOMES), one BoardKindDetail section rendering tile roster · dangers · seasons & turns · zones using it, and a per-kind authored HTML body for prose like fish tides and the Mysterious-Ore countdown. Also fold the structured farm hazards (Fire / Wolves / Rats) into the existing hazards concept so every Dangers link resolves to a real page.

Why this gap exists

The wiki already has two biome-adjacent concepts, but neither is the board-kind concept:

  • Zones (home, meadow, quarry, …) — rich, but they're individual map nodes, not "the Farm board concept."
  • Settlement Biomes (prairie, mountain, …) — marked PARTIAL; render only 4 fields, no rules.
  • BIOMES (farm / mine / fish) — the three real board kinds with their tile pools, hazards, and per-board mechanics — are not wiki entities at all.

This spec turns that third row into first-class wiki articles without touching the first two concepts.

Goals & non-goals

Goals

  • Three articles (Farm/Mine/Harbor) reachable like any other concept.
  • Config-derived: tile roster, dangers, seasons & turns, zones-using-it.
  • Authored prose for board-specific mechanics that aren't structured.
  • Farm hazards become real, linkable wiki pages.

Non-goals

  • No Zod schema for BIOMES (live-config-only, like hazards).
  • No editing board kinds from the Dev Panel.
  • No change to Zones or Settlement Biomes concepts.
  • No new game mechanics — pure documentation surface.

The concept & its three entries

One concept boardKinds, three entries sourced from BIOMES in src/constants.ts. Entry keys mirror the BIOMES keys (so fish, not harbor) — this keeps getEntity a direct BIOMES[key] lookup and matches the tile.biome field used to filter the roster.

How an article assembles

The article reuses the unified WikiArticle.tsx template. The only new rendering unit is BoardKindDetail (one well-bounded section, modeled on ZoneDetail.tsx); everything else is the standard lede / infobox / properties / relations / backlinks pipeline plus the optional authored body.

BoardKindDetail — the four blocks

A single new component src/balanceManager/wiki/sections/BoardKindDetail.tsx with a hasBoardKindDetail(entity) guard, rendering these blocks in order (each contributing a TOC entry):

  1. Tile roster. BIOMES[key].tiles as icon chips — label + what each produces (tile.next) — each linking to its tile article via wikiNavTarget("tiles", tileKey).
  2. Dangers. Hazard chips (icon + name + clear-instruction) linking to hazards articles. Mine → cave_in/gas_vent/lava/mole; Farm → fire/wolf/rats; Harbor → none (tides covered in prose).
  3. Seasons & turns. The four SEASONS badges + the global turn rules (turns-per-season, 4 seasons) read from constants, not hardcoded.
  4. Zones using it. ZONES filtered by hasFarm / hasMine / hasWater for this kind, each linking to its zone article.
Danger→kind mapping. There is no single config field "this board kind's hazards." The component maps it explicitly: mine→HAZARDS (mine module), farm→FARM_HAZARD_META keys, fish→none. This mapping lives in the component (small, board-kind-specific) rather than being forced into BIOMES.

Extending the hazards concept

So Farm's Dangers links resolve, the existing hazards concept grows from mine-only to include farm hazards. FARM_HAZARD_META already carries name / description / clearInstruction, and textures hazard_fire / hazard_wolf / hazard_rats already exist.

Authored bodies

Three fragments under src/balanceManager/content/boardKinds/, rendered in the existing About section. They carry conceptual prose that isn't structured config — allowed to drift like all authored docs:

  • farm.html — seasons / harvest rhythm; Fire, Wolves & Rats in narrative form.

  • fish.html — fish tides & pearl capture (the Harbor's signature mechanics).

Patterns to mirror

ZoneDetail.tsx — the model for BoardKindDetail

Zone articles render a dedicated ZoneDetail section gated by hasZoneDetail(entity), wired in WikiArticle.tsx via a showZoneDetail flag + TOC entry + render slot. BoardKindDetail follows the same three-point wiring exactly, so the change to WikiArticle.tsx is small and local.

Live-config-only concepts (no Zod schema)

categories, hazards, views, modals, tileDiscoveryMethods all return null from schemaForConcept and rely on LiveConfigFallback to render their Properties. boardKinds joins them — no schema invented for BIOMES.

Authored HTML body resolution

htmlContent.ts eagerly globs ../content/**/*.html and keys by <conceptId>/<key>. Dropping content/boardKinds/farm.html makes bodyFor("boardKinds","farm") return it automatically — zero loader changes.

Risks & edge cases

  • Harbor has no structured hazards. The Dangers block must render gracefully (omit, or a one-line "no board hazards — see Tides below") rather than an empty heading.
  • Key collision. Bare-key resolution (conceptForKey) iterates CONCEPTS in order; farm/mine/fish are unlikely to collide, but prefer the prefixed boardKinds:farm focus format for nav targets.
  • Visual goldens are platform-sensitive. Only refresh if the diffs are exactly the new pages.
  • Fire status. Don't mislabel a flag-off hazard as WIRED.
