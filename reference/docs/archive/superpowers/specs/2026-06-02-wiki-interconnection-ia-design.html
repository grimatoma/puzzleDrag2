Wiki Interconnection & Information Architecture

Make the Dev Panel (/b/) wiki more interconnected and better organized: a redesigned two-level navigation tree, explicit page-kind labelling, and bidirectional cross-links so concepts, categories, and instances all reference each other.

Design spec · Status: Approved · 2026-06-02 · scope: src/balanceManager/wiki/

Dev Panel /b/ Wiki Information architecture No game/runtime impact

Overview

The /b/ wiki already generates Wikipedia-style pages live from config and Zod schemas, with [[wikilinks]], forward "Relations," and "What links here" backlinks. But the structure reads flat and the cross-linking is applied unevenly: helper concepts like Tile Discovery Methods and Categories sit as top-level peers of Tiles, a concept's definition is buried inside its index page, and a tile doesn't reliably link to its category or the method that discovers it.

This spec does three things: (1) formalizes three page kinds with a visible badge, (2) replaces the left nav with a two-level primary→secondary tree, and (3) makes the relations graph bidirectional and consistent so declaring an edge once surfaces it from both ends.

TL;DR. One page per concept (unchanged), but each page is badged by kind and leads with a definition; the nav nests helper concepts under their parent; and tile↔category / tile↔discovery-method edges are added so every page links to its neighbours.

Goals & non-goals

Goals

  • Reader always knows what kind of page they're on (Concept, Category, or Instance).
  • A concept page leads with a clear "what is a Tile" definition before the schema reference and instance grid.
  • Helper concepts (discovery methods, categories, tool powers, keepers) are reachable under the concept they serve, not as flat top-level peers.
  • From a tile you can reach its category and its discovery method; from a category you can reach its member tiles and the Categories concept; from a discovery method you can reach the tiles that use it.
  • This bidirectional consistency is a general rule across concepts, not a tiles-only special case.

Non-goals

  • Splitting a concept's definition onto a separate page from its instance index (decided: keep on one page, label it).
  • Hand-authoring prose for every concept now (we keep authored-HTML hooks and fill them later).
  • Any change to the game (/) or Story Editor (/story/) entries, or to runtime/reducer behaviour.

Page-kind model

We name three kinds and render a small labelled badge in every page header, driven by a pageKind attribute on the route/concept model — never hand-set per page.

Concept

The canonical page for a concept (e.g. Tiles). Order: definition → schema field reference → instance grid. Rendered by CategoryPage.tsx.

Category / Index

A grouping that lists members (e.g. the Vegetables tile category, or Discovery Methods as a set). Body is the member list + an up-link to the concept it groups.

Instance

One entity (e.g. the Wheat tile). Structurally unchanged; gains the cross-links described in the Cross-linking tab.

The Concept page leads with an auto-generated definition: a titled "What is a Tile" section derived from the concept blurb + the Zod schema (kind, key namespace, and the attributes every member shares). The schema field reference and the browsable instance grid follow, as today. Definition and index stay on the same page — we make the definition an explicit, titled lead block rather than burying it.

Badge source. Add a pageKind field to the concept descriptor in concepts.ts (default "concept"; category-like concepts tag the kind of their instance pages). A new PageKindBadge.tsx reads it and renders in both WikiArticle.tsx and CategoryPage.tsx headers.

Navigation IA — two-level tree

The left nav becomes a two-level tree: primary concepts at the top level, with secondary/helper concepts nested beneath the primary they serve, collapsible. Section groupings (Board / Economy / World / Progression / Screens) are retained as headers. The mapping below is the approved structure.

Behaviour. Each primary with children renders a collapsible disclosure (caret). Expanding it lists the secondary concepts; both primary and secondary remain directly clickable to their own pages. The currently-active page auto-expands its parent so the user always sees where they are. Collapse state persists in component state for the session.
Routing unchanged. Each concept keeps its own tab id and hash route (/#/<concept>, /#/<concept>/<key>). Nesting is an IA/presentation change in wikiNav.ts + WikiShell.tsx only — no concept is removed, so existing deep links and [[wikilinks]] keep resolving.

Cross-linking principle

Every relationship is declared once as a forward edge in relations.ts and surfaced from both ends: forward via "Relations," reverse via "What links here" (an inversion of all forward edges in backlinks.ts). So adding tile → category and tile → discovery-method gives us category → its tiles and method → its tiles for free.

Member lists. Category and discovery-method instance pages render their member tiles directly in the body (derived from TILE_CATEGORIES / tileGrouping.ts and each tile's discovery config), in addition to the automatic "What links here." This is the literal "go to a category, see its tiles" request.

Consistency audit

To avoid a tiles-only special case, the implementation includes a pass over the other concepts to confirm each declared relationship is symmetric (forward edge present ⇒ backlink resolves, and vice-versa). Where a concept has an obvious "grouped by" or "unlocked by" relationship that's currently one-directional, add the missing forward edge so the backlink inversion fills in. Concrete candidates: Tools↔Tool Powers, Zones↔Keepers, Zones↔Zone Categories. The audit is encoded as a unit test (see Implementation), not just a manual check.

Testing

Unit tests (src/__tests__/)

  • Nav tree shape: every secondary concept is nested under exactly one primary; every concept id appears exactly once; all ids resolve to real concepts.
  • Page-kind assignment: every concept has a valid pageKind; category-like concepts mark instances correctly.
  • Relation edges + inversion: assert tile→category and tile→discovery-method resolve forward, and that the category→tile / method→tile backlinks resolve in reverse (round-trip).
  • Consistency invariant: a test that walks all forward edges and asserts each target's backlink set contains the source — encodes the audit so regressions fail CI.

Gates

Run npm run lint, npm run typecheck, npm test. /b/ is Phaser-free; the wiki is the rendering surface, so this is a UI change — also run npm run test:visual and refresh goldens with npm run test:visual:update if (and only if) the diffs are intentional, committing them in the same PR.

Definition of done. Nav renders the two-level tree with working collapse + active auto-expand; every page shows its kind badge; concept pages lead with a definition; a tile links to its category and discovery method; a category and a discovery method each list their member tiles; unit tests + the consistency invariant pass; visual goldens are clean or intentionally refreshed.

Appendix

Current architecture (reference)

Entry: balanceEntry.tsx → balanceManager/index.tsx → wiki/WikiShell.tsx (hash routing via router.ts, sidebar from wikiNav.ts, command palette). Concept inventory in concepts.ts; entity resolution in conceptEntities.ts; schemas in conceptSchemas.ts. Pages: CategoryPage.tsx (concept landing) and WikiArticle.tsx (instance: lede, infobox, relations, backlinks). Links: wikilink.ts parser, WikiLinkButton.tsx, forward relations.ts, reverse backlinks.ts. Discovery methods: src/config/tileDiscoveryMethods.ts + TileUnlock.tsx. Categories: features/tileCollection/data.ts (TILE_CATEGORIES) + tileGrouping.ts. Authored HTML via htmlContent.ts (bodyFor/pageFor).

Why "label, don't split" over separate definition/index pages

Splitting would double the page count for every concept and force a navigation hop between "what is a Tile" and "which tiles exist," for marginal benefit. A titled definition lead section plus a kind badge communicates the distinction without fragmenting the page or breaking existing routes/links.
