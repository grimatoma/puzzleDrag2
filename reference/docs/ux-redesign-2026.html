UX Redesign

A cohesive, polished visual language for every screen in the game — collapsing ~25 bespoke layouts into four archetypes and one living chrome, governed by a semantic status palette and a shared spatial-motion model. Built for touch & mouse, portrait & landscape, phone & desktop.

v1.0 · 2026-06-13 Theme: Illuminated Ledger Fraunces · Work Sans · JetBrains Mono 25 views → 4 archetypes
The diagnosis

The theme is good. The system isn't applied.

Hearthlands already has a warm, characterful identity — parchment & ink, ember/gold/moss accents, a serif display voice — and a real token layer (tokens.css, components.css). The problem is not the look. It's that ~25 screens each re-implement that look slightly differently. The result reads as a talented team that never agreed on the rules.

🧩

3 nav systems

A bottom-nav tab bar, a HUD menu pill, and FeaturePanel.Tabs — three affordances for the same job. The bottom nav also vanishes entirely on the board.

🎨

Hex, not tokens

"Active orange" is #ff7a00 in Bosses but #d6612a in Portal. Keeper-path & season colors are component constants. State color is reinvented per file.

📑

4 "list" layouts

Crafting uses Browser-Detail, Inventory uses inv-grid, Orders uses ad-hoc ActionCard rows, Decorations a plain card grid. One mental model, four executions.

🪟

Modals opt out

Settings, Tutorial, Festivals bypass the panel system with inline styles, so the most-seen surfaces look least like the game.

The cost

Every new screen starts from a blank file because there's no archetype to drop into. Cohesion decays with each feature, and a global restyle means editing dozens of inline style={{…}} objects by hand. The board literally "feels like two apps" because its portrait/landscape grid flips wholesale at 500px.

The thesis

Four archetypes, one living chrome

Almost every screen in the game is one of four shapes. Name them, build them once, and a "new page" becomes "an archetype with its slots filled" — not a blank canvas. The chrome around them stops disappearing and instead morphs to context.

🗺️

Atlas

Canvas + rail. A living scene (Phaser) with an inspector rail that docks or rises. Board, Town/Village, Cartography.

📖

Ledger

List ⇄ detail. Browse a collection, inspect one. Crafting, Inventory, Workers, Tile Collection, Wiki, Decorations.

🧾

Stack

Card column. A scroll of self-contained, actionable cards. Orders, Quests, Achievements, Castle, Boons, Bosses, Charter, Chronicle.

🛎️

Sheet

Focused interrupt. Rises over context, dismiss to return. Menu, Settings, Tutorial, Festivals, Run Summary, Boss intro.

The wins compound: a single Ledger component means Inventory's search, Crafting's detail pane, and Workers' hire flow all behave identically. A single status palette means "locked / available / active / owned / missing / in-progress" looks the same in the map, the shop, and the quest log. And one chrome means players learn navigation exactly once.

Five principles guiding every decision

1 · Depth earns focus

The current UI is parchment edge-to-edge — flat, no hierarchy. We float panels on a hearth-walnut atmosphere layer with vignette + grain. The thing you act on is always the brightest, highest object on screen.

2 · State has one color

A semantic ramp — locked available active owned missing — used everywhere. No screen invents its own "active."

3 · Chrome morphs, never vanishes

One Hearth Bar that becomes town navigation in the world and the tool tray on the board. Continuity over context-switching whiplash.

4 · Space is continuous

Three fluid tiers (Compact · Cozy · Hearth) instead of one hard 500px flip. Rails slide from bottom-sheet to side-dock; nothing teleports.

5 · Motion is spatial

Views live on a ribbon. Lateral nav slides sideways; drill-in pushes forward; sheets rise. Direction always encodes "where did I go."

The bar

Every screen should feel like a page torn from the same illuminated almanac. If a reviewer can tell which engineer built a screen, we've failed.

Foundations · Color

One palette, with state as a first-class citizen

The surface palette stays faithful to the game's warmth. The new idea is a semantic status ramp: six tokens that mean the same thing on every screen, replacing the dozens of one-off hexes. A tile that's locked on the map, a recipe you can't afford, and a quest not yet started all speak the same color.

Rule

No component literal ever names a hex for a state. If you're typing #ff7a00 to mean "this is active," you want var(--st-active). Brand hexes (ember/gold/moss) live in tokens too; the only raw hex left is inside canvas/SVG artwork.

Foundations · Type

Typography with a strong voice

Fraunces carries every title — it's a high-contrast "old-style" serif that reads as hand-set and illuminated, matching the game's existing --font-display intent (Cinzel/Fraunces). Work Sans handles body and UI for clean legibility at small sizes. JetBrains Mono locks every number to tabular figures so counters never jitter.

Foundations · Space & Elevation

A 4px rhythm and a real elevation ladder

Spacing stays on the existing 4px base. Elevation is where we add the depth the flat UI is missing — three rungs, each pairing a soft cream top-light with a warm drop shadow so panels feel like paper laid on a table, not painted on glass.

Elev 1 · Card

Rows, tiles, list items resting in a panel.

Elev 2 · Panel

A screen floating on the atmosphere layer.

Elev 3 · Sheet

A modal or bottom-sheet, above everything.

Radii & touch

Radii nest: --r-sm 7 (chips/buttons) → --r-md 11 (cards) → --r-lg 15 (panels) → --r-xl 20 (sheets). Every interactive target keeps the existing 44px hit floor; on the board, tools get a 52px floor because they're used under time pressure.

Motion · Spatial model

Every view has a place on the ribbon

The shell is one horizontal ribbon of peer views with the board/town as home. Transitions encode geography, so the player always feels where they went, not just that something changed. Three gestures, three motions — that's the whole vocabulary.

Lateral

Switching peer views (nav). The outgoing view slides off toward its side, the incoming slides in from the opposite edge. Direction matches the nav order.

Forward / Back

Drilling into a detail (zone, recipe, NPC). New layer pushes up 1 z-step, scales from 0.96→1 + fades; back reverses. The parent dims, never unmounts.

Rise

Sheets & modals. Translate from the bottom with a spring, scrim fades in. Dismiss drops back down. On phones this is a draggable bottom-sheet.

Motion tokens & the live feel

Durations and easings are tokenized so motion is consistent and tunable in one place. Tap the demo to feel each transition.

Responsive · Touch & mouse, portrait & landscape

Three fluid tiers, not one hard flip

The board's wholesale 500px reshuffle is the worst offender — it reads as two different apps. The redesign replaces every binary breakpoint with a continuous reflow across three tiers. Rails migrate; they never teleport.

Compact < 600px

Phone portrait. Single column. Inspector rails become draggable bottom-sheets. Hearth Bar pinned to the bottom. One primary action always thumb-reachable.

Cozy 600–1024px

Phone landscape · small tablet. Content + a docked side rail appear together. Hearth Bar can sit on the leading edge. Ledger shows list + detail side by side.

Hearth > 1024px

Tablet landscape · desktop. Full three-zone composition: persistent rail, generous canvas, contextual inspector. Hover affordances unlock; nothing is hidden behind a tap.

Implementation note

Affordances key off @media (hover:hover) and (pointer:fine), never off viewport width. A touchscreen laptop and a phone in a stand both get touch-first behavior at any size — the existing hl-btn hover rule already does this; we extend the pattern system-wide.

Living Chrome

One HUD, one Hearth Bar — and it never disappears

Today the bottom nav exists in town and vanishes on the board, while a hamburger pill and a separate tool grid try to fill the gap. Players relearn navigation every time they enter the field. The redesign uses a single bar that morphs: in the world it's navigation; on the board it's the tool tray and a Leave control. Same position, same physics, continuous identity.

Why this beats three separate systems

Before

  • Bottom nav (town) · hamburger pill (everywhere) · tool grid (board landscape only) · tool modal (board portrait only)
  • Nav disappears on the board → dead space + relearning
  • Only the Inventory tab can show a badge
  • Portrait & landscape board duplicate ~90% of tool code with different UX

After

  • One Hearth Bar with two modes (nav ⇄ tray) sharing physics, position, and styling
  • Persistent → muscle memory holds across contexts
  • Any destination can carry a status badge (orders ready, quest claimable, castle need)
  • One tool component; the tray simply docks to the side at Cozy+ tiers instead of forking the layout
Migration

This maps cleanly onto the existing code: the bottom-nav TabBar becomes the nav mode, the board's PuzzleHotbar/PuzzleToolGrid/PuzzleToolModal trio collapses into the tray mode, and the HUD's OPEN_MODAL menu/leaveBoard dispatch already does the right thing from the leading button.

Deep dive · The Board (core loop)

The most-played screen earns the most scrutiny. Today it's an Atlas in spirit but forks its entire grid at 500px and scatters tools across three components. The redesign makes it one Atlas whose rail is the Hearth Bar tray.

Layout & reflow

  • Compact: board fills the canvas; tray is the bottom Hearth Bar (scrollable); the action panel (chain progress / tool inspect) is a thin band above the bar.
  • Cozy: tray docks to the leading edge as a vertical rail; board grows into the freed width — continuously, no snap.
  • Hearth: board centers with the tray rail left and a live chain/score inspector right; hover reveals tool tooltips.

Animation spec

  • Chain build: matched tiles pulse (--m-fast), the chain meter fills (--m-med), milestone crossings flash gold.
  • Tool arm: armed tool gets the gold ring + a 1px lift; the board edge breathes a faint ember pulse (state, not hardcoded red).
  • Harvest / season turn: field gradient cross-fades between seasons over --m-celebrate; reward chips fly to the HUD coin pill.

Implementation roadmap

The redesign is mostly a consolidation, not a rewrite — most pieces already exist; they need to be unified and tokenized. Sequenced so value lands early and risk stays contained.

PhaseWorkWhy first / risk
0 · TokensAdd the status ramp + motion tokens to tokens.css; add hl-seg, hl-btn2, hl-acard, sheet primitives to components.css.Pure addition, zero behavior change. Unblocks everything.
1 · ChromeBuild the Hearth Bar (nav ⇄ tray morph) + unified HUD; retire the 3 nav patterns.Highest visibility, touches navigation — ship behind a flag, QA on board ↔ town transitions.
2 · StackMigrate Orders, Quests, Boons, Bosses, Castle to the action card. Kills the most inline hex.Self-contained screens; easy to verify with visual goldens.
3 · LedgerConverge Inventory + Crafting + Decorations + Tile Collection + Wiki on one Ledger.Inventory's grid mode is the trickiest; do it last in this phase.
4 · AtlasUnify Board reflow + Town + Cartography on the docking rail; remove the 500px fork.Phaser-adjacent; needs the scene-bridge regression suite green.
5 · SheetMove Settings, Tutorial, Festivals, Run Summary, Boss onto the Sheet.Cleanup + the celebration polish. Low risk, high delight.
6 · MotionWire the spatial ribbon transitions into the view router.Final layer; gate on prefers-reduced-motion.
Verification

Each phase ends with npm run test:visual goldens re-baselined and justified, per the repo's pre-PR rule. The token phase should produce no visual diffs; if it does, a screen was reading a hardcoded value it shouldn't have — that's a find, not a regression.

The one-sentence version

Keep the soul — parchment, ember, the illuminated-almanac voice — but give it structure: four archetypes, one living chrome, a status palette, and a spatial motion model, so every screen finally feels like a page from the same book, on any device, under any thumb.

Each interactive mock in this document is real HTML/CSS using the proposed token system and the actual container-query reflow — open it on a phone and resize to feel it. From here the natural next step is a clickable prototype branch implementing Phase 0–1.
