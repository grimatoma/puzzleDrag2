A first-pass, zone-by-zone progression design for Hearthwood Vale's kingdom of holds. The maintainer is building the game one town at a time — each zone fleshed out as a self-contained unit before reasoning about how they interlock — and this page captures the approved meta-model and the populated first-pass roster. It is a companion to the Zone flow page (which describes today's linear map order) and the Progression page (the build roadmap): this one answers what each zone becomes, and how finished zones keep paying out without piling up into an unplayable always-active game.

FIRST-PASS DESIGN — DRAFT PLANNED Design doc, dated 2026-06-04. Deliberately open to changing the roster, specializations, boards, and teaching of any town. Like everything under the wiki, this is allowed to drift from code — trust code where they disagree.

The central worry this design answers. Zone order is player-chosen with no cap. Left naive, an "always active" game becomes unplayable as completed zones pile up, each still demanding attention. The fix below: a completed zone retires into a passive hinterland that you glance at and collect from, while your active surface stays bounded to the 1–2 frontier zones you are currently exploring. That promise — finished zones stop being chores — is the spine of everything here.

The problem today

The game has ~11 map nodes, but per-zone identity is thin. Every building is buildable everywhere; zones don't differ in what they teach; there is no reason to revisit a finished zone; and nothing automates an old zone once you move on. Per-zone whitelists exist in data (MAP_NODES[*].buildings) but they don't gate, specialize, or idle. This doc gives each town a distinct fantasy, a specialization, a teaching job, a lifecycle, and a bounded reason to return.

Part 1 — The meta-model (cross-cutting)

The rules every zone obeys. Read this first: it is the load-bearing design. The per-zone roster in Part 3 is just this template, filled in.

1.1 Active frontier + idle hinterland

Every founded zone sits in one of three lifecycle states. The player's active surface is always the 1–2 frontier zones; the hinterland is glance-and-collect, not grind.

StateWhat it isPlayer attention
Frontiernewest zone(s) being explored / runfull, hand-played
Settlingfounded, mid build-out (Camp → City tier ladder)active
Hinterlandcompleted — core supported buildings built + keeper resolved + Hearth-Token mintedpassive — glance & collect
  • A completed zone flips to Hinterland and starts an idle production loop: a slow trickle of its specialization goods + coin, scaled by the buildings/workers placed there, accumulating as a local cache.
  • That cache is periodically ferried by caravans to the shared Kingdom Stockpile at the [[zones:home|home]] hub (see 1.5).
  • [[zones:home|home]] is the permanent hub: it never fully idles — it holds the Kingdom Stockpile, is where civic unlocks first land, and is where caravans deliver.

Why you return to a hinterland zone (bounded, never forced)

Three pull-backs, all optional or one-tap — none reopen an active grind:

#Pull-backWhat it does
1Civic upgradesWhen a global civic unlock fires, hinterland zones that support the new building get a one-tap upgrade that raises idle yield and/or opens a beat. You pop in to place it.
2Narrative beatsZone-scoped story hooks (build chapel → nun) play on revisit.
3Episodic demand (optional)The [[zones:fairground|Fairground]] runs seasonal contests asking for a specific old zone's specialty good (honey, pearls, gems, steel) for big rewards — the optional pull-back that complements idle without forcing active replay.

1.2 Cross-zone civic unlocks via support-lists

Each zone gets a supports list: the buildings it can ever host. This is separate from what is currently buildable. Per building in the list, an unlock gate:

  • from_founding — buildable as soon as the zone is founded.
  • event:<id> — buildable only after a trigger fires (local, or global/civic).
  • Buildings not in the list are never available there — a landlocked farm never supports harbor_dock.

Worked example

Zone 1 supports A, B, C but never D. A and B are available from founding; C unlocks after an event. D is simply absent from the list:

supports: {
  A: from_founding,
  B: from_founding,
  C: event:found_chapel_order,
}
// D absent → never supported here

Civic unlocks are global events that flip a gated building's availability across all founded zones that list it in supports. Example: a "religion" milestone reached anywhere flips chapel from event-locked → available in every zone whose support-list includes chapel. A mine town that doesn't list chapel is unaffected.

Maps onto existing data: MAP_NODES[*].buildings is already a per-zone building whitelist (src/features/cartography/data.ts); the change is to turn each entry from a bare id into { id, gate }.

1.3 Progressive boards

A zone opens with its primary board (its biome). As it tiers up, it can unlock additional boards, including cross-biome ones — each tied to a building/event and listed in the zone's support set. This makes "multiple boards per zone" the vehicle for cross-biome depth in late zones.

Host zone builds……opens this board
farm town builds [[buildings:harbor_dock|Harbor Dock]]a small fish board alongside the farm board
mine town builds [[buildings:forge|Forge]] at City tiera hazard-free Forge (smithing) board
mine town builds [[buildings:observatory|Observatory]]a deep-shaft board

The board-instance schema already supports boards: { farm?, mine?, fish? } per zone (src/config/schemas/boardInstance.ts) — today each zone fills only one slot.

1.4 Zone-relative progression (player-chosen order)

Because order is player-chosen after the first settlements, progression is zone-relative, not globally linear:

  • A shared settlement template runs per zone, keyed on event.zoneId: found → tier-ups (Camp → Hamlet → Village → Town → City) → keeper decision → complete → Hearth-Token.
  • A thin layer of zone-specific flavor triggers sits on top (the per-zone "key triggers" in Part 3).
  • The only global spine is: found a complete zone of each biome (farm / mine / fish) → unlock the [[zones:oldcapital|Old Capital]] finale. Everything else is local to a zone and reorderable.

1.5 Idle collection — caravan delivery

Chosen hand-off. Hinterland zones don't auto-teleport their output. Each accrues a local cache; a caravan network periodically carries it to the Kingdom Stockpile at the hub. This repurposes the existing [[buildings:caravan_post|caravan_post]] building as the hinterland's connective infrastructure:

  • A zone needs a [[buildings:caravan_post|caravan_post]] (and a route to the hub) for its idle goods to flow.
  • The player can optionally route/upgrade caravans to raise throughput or shorten trips.
  • Kept deliberately light — caravans run themselves by default (no per-zone clicking), so active workload stays bounded; routing is an optional optimization layer, not a chore.

Caravan raids/escorts are a possible later hazard hook — noted, not chosen.

Forge board · progressive — replaces the standalone Black Forge zone

RESOLVED The [[zones:forge|Black Forge]] is no longer a standalone node. It becomes a progressive, hazard-free smithing board that opens inside a mine zone (quarry or caves) once the [[buildings:forge|forge]] building is raised at City tier — so "multiple boards per zone" carries the region's mastery content.

Specialization. Master smithing — [[resources:iron_bar|iron_bar]] + [[resources:coke|coke]] → steel/tools, runes. Idle-produces: finished tools / steel / runes (top tier) into the host zone's cache.

Puzzle features. No hazards (mastery board); long, high-value chains.

Unlock gate. event:build_forge at the host mine zone's City tier.

Return hook. The only source of best-in-slot tools/runes for boss fights — a strong reason to keep a mature mine zone in rotation.

Part 4 — How this gets encoded (roadmap)

This page ships the design only. Encoding is deliberately deferred and sequenced as follow-on plans so the content can settle first, tuned later against the Dev Panel.

Encoding sequence (NOT built now) — five follow-on plans
  1. Support-list data. Turn MAP_NODES[*].buildings entries into { id, gate } (from_founding | event:<id>); build UI + a new civicUnlock effect read the gate. (src/features/cartography/data.ts, build UI, src/config/progression/types.ts effects.)
  2. Per-zone specialization + idle-yield config, the Hinterland lifecycle, and the caravan network → Kingdom Stockpile delivery subsystem (repurposing the [[buildings:caravan_post|caravan_post]] building; the largest piece — its own plan).
  3. Zone-scoped + civic triggers in triggers.ts (event.zoneId) and story beats for the flavor hooks (build chapel → nun) in src/story.ts via showBeat.
  4. Progressive board unlocks (fill more boards slots per zone, gated) — including the Forge board in the mine region.
  5. Fairground episodic-demand loop.
