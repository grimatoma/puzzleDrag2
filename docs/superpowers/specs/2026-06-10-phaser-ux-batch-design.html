Phaser Game UX Batch

Five small, high-confidence player-experience improvements to the React+Phaser game: surface the minimum chain length, give tools audible feedback, toast on inventory cap, smooth the hazard vignette, and clamp the chain hover badge on narrow screens.

Design spec · Status: Approved · 2026-06-10 · scope: game entry (/) only — no Dev Panel, Story Editor, or Godot changes

ux clarity-and-feedback juice mobile no-new-mechanics

Overview

A code survey of the game's UX surface found the core drag-chain loop mechanically solid but light on explanation and a few feedback channels. The biggest new-player friction points are rules the game enforces but never states: the minimum chain length is only communicated by a fast brown→orange path-color flip; arming or firing a tool makes no sound; hitting the inventory cap is a tiny ⓘ on float text that vanishes in under a second. Two smaller polish items round out the batch: the hazard vignette repaints with fresh random alpha every 120 ms (a jittery flicker rather than an atmospheric breathe), and the chain hover badge that trails the finger has a vertical clamp but no horizontal one.

TL;DR. Five independent changes, one PR: (1) add minChain to the chain HUD payload and render a "need N more" state in ChainView; (2) add toolArmed/toolFired synth sounds driven by toolPending transitions in useAudio; (3) toast via the existing announce() bridge when a chain's gain is capped, with a per-resource cooldown; (4) replace the hazard vignette's 120 ms random repaint with a slow alpha-tween breathe; (5) horizontally clamp the chain hover badge.

Goals & non-goals

Goals

  • State the min-chain rule where the player is already looking (the chain panel).
  • Close the silent-feedback gaps: tool arm/fire audio, inventory-cap toast.
  • Make hazard atmosphere feel deliberate, not janky.
  • Keep the hover badge fully on-screen on narrow portrait viewports.

Non-goals

  • No new game mechanics, balance changes, or save-shape changes (no SAVE_SCHEMA_VERSION bump).
  • No tutorial-content changes (cross-collect step deferred to a future batch).
  • No keyboard navigation / screen-reader narration work (separate a11y batch).
  • No Godot port changes.

1 · Min-chain indicator in the chain panel

Data flow. GameScene._emitChainUpdate already passes effectiveMinChain (base 3, raised by boss minChain) into buildChainUpdatePayload. Add minChain: number to ChainUpdatePayload and the ChainInfo interface in src/ui/puzzleBoard.tsx — a purely additive payload change.

Presentation. In ChainView, the PanelHeader right slot currently always reads "{resourceLabel} chain". While the live chain is invalid (!valid && count > 0), show "{minChain − count} more to collect" in the warning tone instead; once valid it reverts to the resource label. This puts the rule exactly where the player's eye already is during a drag, costs no new layout, and inherits the panel's existing styling.

Why not a toast or board overlay? The brown path color already signals "not yet" spatially; what's missing is the number. The chain panel is the established home for live chain numbers (count, threshold progress), so the missing number goes there.

2 · Tool armed / fired audio cues

Sounds. Two new entries in SOUNDS (src/audio/index.ts), synthesized like the rest (no binary assets): toolArmed — a short rising two-note blip (confirmation, distinct from chainStart); toolFired — a lower percussive thunk (impact). Exact envelopes tuned during implementation to sit with the existing palette.

Triggers. useAudio is the established home for state-transition sounds. Extend AudioGameState with toolPending and watch transitions: falsy → key plays toolArmed; key → falsy plays toolFired only when that tool's inventory count decreased in the same transition (the reducer consumes the tool on use), so cancelling an armed tool stays silent. Haptics piggyback on the existing settings.hapticsOn gate (short pulse on fire).

3 · Inventory-cap toast

Amended 2026-06-10 (review finding). The originally specced trigger — GameScene.collectPath's overCap — is provably dead code: the registry's inventory mirror is zone-keyed (so inv[res.key] is always undefined), and the cap actually applies to the produced resource, not the tile key. The scene-side approach was reverted in favor of the reducer's real signal.

Trigger. The reducer already detects the true clamp moment: addCappedResourceMut (src/state/helpers.ts) flags seasonStats.capFloaters[key] once per season per resource when a roll-up credit is clamped at the cap — but no UI consumed that signal. A small React hook, useCapToasts(state) (wired in prototype.tsx beside useAudio), watches capFloaters transitions and calls announce() with a warning-tone toast — "Hay Bundle storage full — extra is wasted" — using the item label from the catalog.

Spam control. Inherited from the reducer: capFloaters dedupes per resource per season (reset at CLOSE_SEASON), so no scene-side cooldown is needed. The transition diff is a pure exported helper (newlyCappedKeys(prev, next)) so it's unit-testable without React.

Wording. Only whole roll-up units above the cap are discarded — coins, XP, upgrades, and fractional resourceProgress are all still earned — so the toast says the extra is wasted, not that the chain was lost.

4 · Hazard vignette breathe

Today. _updateHazardAtmosphere runs a 120 ms looped timer; every tick clears the graphics object and repaints full-screen rects with Math.random() alpha — visually a flicker, and needless per-frame work.

Change. Paint the vignette rects once per hazard-state change at fixed alpha, then drive the container's alpha with a single yoyo-looping tween (~1.4 s, Sine.InOut) between a low and high bound. Same colors and intensity range as today; the random jitter becomes a calm breathe. The tween is killed/recreated alongside the existing timer-teardown path. Repaint still happens on resize (the existing layout pass already calls _updateHazardAtmosphere).

5 · Hover badge horizontal clamp

The chain hover badge trails the pointer and clamps y (minY = 26×dpr) but not x: a drag along the left or right board edge can push the badge partially off-canvas — most visible on narrow portrait phones, where the board spans nearly the full viewport. Measure the badge's rendered width and clamp its center to [halfWidth + margin, viewportWidth − halfWidth − margin] in the same place the y clamp lives. Verified against the iPhone-portrait visual profile.

Testing & verification

  • Unit: buildChainUpdatePayload emits minChain and the existing valid behavior is unchanged; cap-toast cooldown helper (allow → suppress within window → allow after); tool-sound trigger logic if extracted, otherwise covered by the payload-shape tests.
  • Visual: npm run test:visual — expected diffs limited to chain-panel scenarios (header text change). Add an invalid-chain scenario to matrix.js; refresh goldens (npm run test:visual:update) only for intentional diffs and commit them in the same PR.
  • Gates: npm run lint, npm run typecheck, npm test before the PR.
  • Manual QA: dev server + ?visual=board-farm-chain-7 (valid-chain header), short drag (invalid header), arm/cancel/fire a tool (audio), ?visual= hazard scenario (breathe), narrow-viewport edge drag (badge clamp).

Risks & edge cases

  • Keeper trials and bosses raise min chain. The indicator must read effectiveMinChain from the payload (it does — single source), never hardcode 3.
  • Tool cancel vs fire. Distinguishing by inventory decrement assumes the reducer consumes the tool atomically with clearing toolPending; verify against USE_TOOL handling before relying on it, fall back to a scene-emitted event if the transition isn't atomic.
  • Toast fatigue. Cooldown is per resource key; a multi-resource capped streak can still show a few toasts — acceptable, they carry different information.
  • Visual goldens are platform-sensitive. Only refresh goldens whose diffs are exactly the intended chain-panel change.
  • Reduced motion. The vignette breathe goes through the scene's tween pipeline (_dur hook), staying consistent with how other juice is treated.
