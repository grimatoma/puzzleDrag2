Progression Redesign

Retire player levels and make zones + per-zone tier ladders the single progression spine. Found a zone → unlock its functionality and buildings; tier up → more plots and the next building batch; a set of civic buildings build across all founded zones. The Almanac stays — decoupled into a mastery/reward track that gates nothing.

Design proposal · Status: Draft for review · 2026-06-22 · scope: Town 1–3 slice

Overview

puzzleDrag2 currently runs two parallel progression spines that compete to express "what unlocks next." This proposal retires one of them — player level / XP — and consolidates all unlocks onto the zone + tier-ladder spine the Direction doc has already locked. The result is a model the player can hold in their head: your zones are your progress.

TL;DR. Player levels become legacy. Founding zones and climbing their tier ladders becomes the one progression concept. The Almanac is kept but decoupled — XP still accrues and still grants its perks, it just stops gating anything. Zones 1–2 (farm → mine) stay a linear tutorial; after zone 2, the map branches so the player's next pick is a real choice.

The problem: two spines

retire Player level / XP

Global state.level / state.xp, driven by the Almanac (src/features/almanac/data.ts). Today it gates biomes, map zones, buildings, recipes, and order quality.

keep Zone tier ladders

Per-zone tiers[] + TIER_UP, founding gates (requiresZoneTier), and the progression-trigger engine. This is the intended spine.

The two overlap: anything a level can gate, a zone tier / founding / building can gate more legibly and with more authorial control. Levels just add a second, vaguer axis the player must also track — and force every new gate's author to pick "level or tier?"

This is already the locked direction

The canonical Direction page (src/balanceManager/content/pages/direction.html) already names the tier ladder as the spine and flags levels as the holdover:

"Advancement = the wagon/expedition, raised when the home zone reaches its City rung — one clear forcing function, not scattered level gates."

"Town 2 is not simply unlocked by a level — you run an expedition to discover it… the forcing function that turns 'what do I do next?' into one obvious answer."

So this proposal doesn't change direction — it finishes the migration the Direction doc already committed to.

Scope. This is a design proposal only — no code changes ship with it. It captures the model, maps every live level gate to its replacement, and sketches a phased implementation for a follow-up PR after sign-off.

The unified model

One spine, three legible mechanisms — all of which already exist in code:

1 · Found a zone

Founding (FOUND_SETTLEMENT) unlocks the zone's biome/board and its base building roster. This is where new functionality enters the game.

2 · Tier up within a zone

TIER_UP spends local resources to add building plots and unlock the next batch of building types (tiers[].unlocks). Sequential progression inside a zone. No turn/yield change — puzzle math is untouched, exactly as today.

3 · Civic / global buildings

A set of buildings buildable across all founded zones that support them, flipped on by global milestones — not tied to any one zone's ladder.

Civic buildings reuse an already-designed mechanism

The "buildable across all zones" idea is the parked supports-list + civic-unlock design in src/balanceManager/content/pages/zones.html (and the runtime hook in src/features/civicEconomy/data.ts). Each zone declares the buildings it can ever host; a global civic milestone flips a gated building's availability everywhere that lists it:

"Civic unlocks are global events that flip a gated building's availability across all founded zones that list it in supports… A mine town that doesn't list chapel is unaffected."

So "sequential building unlocks within a zone" (tier ladder) and "a building built across all zones" (civic + supports) are both expressible with existing concepts — no level needed for either.

The Almanac becomes a mastery track decouple

state.almanac stays: XP still accrues from chains, orders, builds, and quests; almanac.level is still derived; the tier-reward claim track in src/features/almanac/data.ts still hands out its perks — coins, tools, and the structural rewards extraTurn, goldSeal, extraBlueprintSlot, startingExtraScythe. The only change: nothing reads it as a gate. It's a "how mastered am I" reward loop running alongside the zone spine, not a key that opens content.

Why keep it. The reward dopamine and the structural perks are real and worth preserving; the confusion came only from level also being a gate. Cut the gate, keep the rewards.

Zone topology & branching

Zones 1–2 are a fixed teaching sequence so the player learns the two core biomes — farm, then mine — that anchor the introductory loop. After zone 2, the map branches: several zones become foundable at once, each offering distinct functionality and buildings, so the next pick is a strategic choice rather than the only available door.

The branch roster above is illustrative, drawn from existing MAP_NODES in src/features/cartography/data.ts (orchard, mirefen/harbor, caves). Exact roster, costs, and how many branches open at once are tuning for the implementation phase. The mechanism is the point: replace "the next node unlocks at level N" with "after zone 2 you may found any of these, and each gives you something different."

Note. The home → quarry gate already exists as data: requiresZoneTier: { zone: "home", tier: 3 } on the quarry node. The tutorial sequence needs no new gating concept — only the removal of the parallel node.level check (see Gate Migration).

Almanac decoupling — what actually changes

Keep all of state.almanac (xp, derived level, claim track) and every perk it grants. The decoupling is purely about what reads level as a gate:

  • Stop reading state.level / state.xp as a gate anywhere — the rows above are the full list.
  • The legacy mirrors state.level / state.xp (synced from the almanac on commit) can stay to feed the mastery HUD, or be collapsed into state.almanac. Either is fine once nothing gates on them.
  • Structural perks survive untouched — extraTurn, goldSeal, extraBlueprintSlot, startingExtraScythe keep flowing from the claim track.

HUD & UX impact

  • src/ui/Hud.tsx — the Lv N · XP/150 chip re-reads as Almanac mastery, not a gameplay gate. Same widget, reframed copy.
  • src/ui/LevelUpCinematic.tsx — keep as a mastery celebration; reword so it doesn't imply new content unlocked (since it no longer does).
  • Lock-reason strings — "Requires level X" (Town), "Mine unlocks at Level 2" / "Harbor fishing unlocks at Level 3" (biomeAccess), "Level 3 required" (zone cards, recipes) rewritten into zone/tier/building language: e.g. "Found a mining settlement to open the mine board," "Reach City to launch the expedition," "Build the Workshop to unlock this recipe."

Phased implementation (follow-up work)

Not part of this doc — the build plan once the model is approved. Ordered so each phase is independently shippable and testable.

Phase A — Retire level gates

Replace the five live level gates (biome, map, building, recipe, order) with their tier/founding/building equivalents from the migration table. Rewrite lock-reason strings. The bulk of the work; high test coverage required.

Phase B — Decouple the Almanac

Confirm no gate reads state.level; reframe HUD + cinematic copy as mastery. Mostly verification + copy once Phase A lands.

Phase C — Branch after zone 2

Map adjacency + the foundable-set at the post-quarry branch point: present orchard / mirefen / caves (final roster TBD) as simultaneous choices. New design + content.

Phase D — Update canonical docs

Update the Direction wiki page (drop the "until the level gate is retired" hedge) and zones.html. Respect the wiki fact-injection guard tests — use the wiki-content skill.
