Sprite Pipeline

Generate and animate a cohesive family of pixel-art game sprites from two durable inputs — gap-filled, quality-gated, and handed to the Godot v2 tile slot by a separate, on-demand step. Keyframes are source-pluggable stills (hand-authored in Aseprite, with PixelLab as an optional accelerator); animation is always hand-built in Aseprite. Cheap critique reviews bracket the two operations that actually cost: keyframe generation and animation.

What this is

A reusable pipeline for producing a growable family of game sprites/tiles on a locked look. You hand it two durable inputs; it generates only what is missing and animates it, with four cheap critique gates bracketing the two expensive operations so a bad asset is caught on paper — not after you have spent generation credits and frame-by-frame effort.

TL;DR. Two inputs in (a style spec distilled from reference art + a three-file pipeline config), a cohesive animated tile family out. Stages 0–4 carry the work; gates G1–G4 keep cost down by rejecting off-style work before the spend. Keyframes are source-pluggable stills (Aseprite by hand and/or PixelLab, competing in one pool); animation is Aseprite only; Godot consumes the result through a separate update step. The pipeline runs fully without PixelLab.
The big change (read if you knew the old version). Two things were superseded. (1) The old per-set sets/<set>/manifest.json is gone — work is now declared in a three-file pipeline config (pipeline.json + pipeline.history.json + pipeline.schema.json) of hierarchical items[] (a master keyframe + derived children + animations). (2) PixelLab was mandatory; it is now an optional keyframe source, and PixelLab's own animate/v3 is not used for shipped motion — every frame is hand-built in Aseprite.

The consistency contract

Two requirements, where the first is what makes the second possible:

1Keyframes are a real family

Each variant is derived from the same anchor image, never re-rolled from scratch. A child (season / damage / growth) keeps the master's size, position, silhouette, and identity, changing only what the variant names.

2Aseprite tweens between them, by hand

Because from and to share a size / silhouette / anchor, a hand-built tween has matching endpoints to work between — so it can be a real staggered, arced motion instead of a fade.

There are two ways to derive a child (the pipeline is agnostic — see Tool routing):

  • Hand (home-grown, default): in Aseprite, import_image the approved master PNG and edit it into the variant (recolor, add frost/snow, wither, damage). The master pixels are literally the starting point, so size/silhouette/anchor are preserved by construction.
  • PixelLab (optional accelerator): state on the master's objectId — an image-conditioned AI edit. Same guarantee, AI-driven.
Why earlier runs failed — and why it's now fixable. The winter pumpkin came out a smaller body with the stem flipped — a keyframe defect (the child was generated from scratch, not derived). The birch autumn→winter tween "just went top-down" — an animation defect, because its two endpoints were structurally different trees, so the only safe motion was a uniform fade. Derive the child from the master (shared trunk + canopy envelope) and Aseprite can stage a genuine leaf-clump-by-clump fall that reveals the matching branch skeleton beneath. Derive keyframes from a shared anchor, then animate by hand.

The problem it solves

The goal is to mass-produce cohesive, professional pixel-art assets — tiles, items, characters — with consistent quality and a loop you can keep adding to. The craft pieces already existed, but they did not compose into a pipeline. Three gaps stood out:

No physics direction

Aseprite animates well when hand-directed frame by frame, but on its own it does not follow physics — you hand-describe all 16 frames of leaves falling or snow piling. There was no systematized motion-direction layer.

No cohesion anchor

Strong static-art and motion skills existed, but nothing forced every asset onto one palette, light direction, and silhouette — and nothing forced a variant to derive from its sibling rather than re-roll. A set drifts without a single contract to score each member against.

No live review surface

No way to eyeball a whole family at once — gaps included — pick / approve a candidate, and route those decisions back into the work. The backlog was invisible and approvals were hand-edited into JSON.

The fix is a pipeline that takes reference assets defining a look plus a three-file config declaring a cohesive, growable group, generates only what is missing, brackets the expensive steps with cheap critique gates, and exposes a live control viewer where you eyeball every set (gaps included) and select / approve / comment — and those decisions are written straight back into the config.

The four stages

References distil into a style spec once (Stage 0). From there each run plans the set against what is already on disk (Stage 1), generates the missing keyframe stills (Stage 2), storyboards the physics of each motion (Stage 3), and animates those storyboards in Aseprite (Stage 4). The pixel pipeline ends at the produced frames + preview GIF. Packing those frames into the Godot engine is a separate, on-demand step — npm run godot:update-tiles — run by hand, never as part of the pipeline run or the npm build.

Pipeline flow

References (the look) and the config (the work) feed Stage 0/1. Gates G1–G4 sit immediately around the two spend steps. A reject loops back to fix that one step; an accept advances. The pipeline ends at the produced frames + GIF; the live viewer surfaces the whole set, and your select/approve/comment decisions are written straight back into the config to drive the next idempotent pass. Updating Godot (dashed, off the main flow) is the separate, on-demand npm run godot:update-tiles step.

The same flow is a handshake between an orchestrator, a cheap critic, the keyframe source, the Aseprite builder, and the engine. The critic is consulted four times — once on each side of each spend — and never gates a whole set in one pass: each missing asset is scored on its own so per-asset rejects are not lost.

The cost model

Generation (Stage 2) and animation (Stage 4) are the costly steps — credits and frame-by-frame effort. A critique gate sits on each side of each, so a reject costs a prompt, not a build.

Why this shape. The two expensive operations are bracketed: G1 → generate → G2 and G3 → animate → G4. G2 and G3 straddle the generation→animation boundary; G1 and G4 are the outer cheap checks. A miss on a locked spec field (off-palette, wrong light) at G1/G2, or a "slide pretending to be motion" at G3/G4, is a reject — fix and re-run that step only.

One critic, four firing points. The cheap diamonds bracket each red spend box — a reject rewinds one step, never the whole set.

One critique sub-agent, four firing points — each a cheap review immediately before or after an expensive operation. The rubric comes from the relevant craft skill: still quality from pixel-art-craft, motion soundness from pixel-art-animation.

Cost-gated control flow

Spend is gated before every major cost event — the master candidate pack, each derived-child batch, and the animate stage — so a bad asset never burns the next batch of credits. The loop per keyframe:

  1. Generate candidates into a pool. A keyframe accumulates up to settings.candidates (1 | 2 | 4) candidates, each tagged source: hand | pixellab. PixelLab master: one create-object call returns a whole review pack (seed count scales with canvas — 64 at ≤42px, 16 at ≤85px, 4 at ≤170px), --style-conditioned on the family's priors. PixelLab child: separate state calls on the approved master's objectId. Hand: author/edit in Aseprite and drop in the PNG.
  2. LLM self-audit scores the whole group in ONE call. Montage the candidates into a single sheet (scripts/montage.py), Read it once, and return a per-candidate verdict (the llm: pass | fail field) — not one Read per seed. For a PixelLab pack, promote only the audited pick(s) with select-frames; the unpromoted seeds die with the pack. Regenerate only the failed subset.
  3. Approval gate. When humanApproval is on (and autonomous off), pause for sign-off in the pixelGen viewer — the human picks a candidate and the control server writes selected + selectedPath to pipeline.json and the candidate's status to pipeline.history.json. When autonomous is on, the LLM verdict decides and you record it with pipeline-patch.mjs (never hand-edit the JSON — a dropped comma silently breaks the run).
  4. Proceed to the next gated event (derive the children, then animate the idles/transitions). Each gated event is its own batch with its own audit + approval, so the next spend only happens against assets that already passed.
Mode lives in the config, not an env var. To run a session full-auto without committing a settings change, flip it with pipeline-patch.mjs set-mode autonomous and restore set-mode gated before you commit (the committed default keeps the human gate on so the next run isn't silently un-gated). The mode lives in pipeline.json so the viewer and the headless run always agree on it.

The builder → critique loop (and how to batch)

Each gap asset is produced by a builder → critique pair:

  1. Builder does one step for one asset (generate this still / animate this storyboard) and reports what it produced.
  2. Critique scores that output against the relevant rubric (the gate above) and returns accept or reject + reasons.
  3. On reject, re-run the builder with the critique's notes; on accept, advance.
Batching a set = fan out one builder→critique pair per gap asset. Because each asset is independent (its only shared context is the style spec + priors, both on disk), the gaps can be worked in parallel — one sub-agent pair per missing keyframe/idle/transition — then reconciled. The additive-overlay animation technique makes every Aseprite call stateless (each frame addressed by explicit frame_number + layer_name, no selection ops), and each builder owns its own _work/<id>.aseprite — so parallel animators never collide. Concurrency cap ~10–16 agents. Parallel builders must NOT each git commit — they write files and hand back; the orchestrator commits once after reconciling.

Tool routing — what executes what

The hard rules: Stage 2 keyframes can come from hand-authored Aseprite and/or optional PixelLab; Stage 4 animation is Aseprite only. Pillow is review glue and nothing else; it never makes art.

Keyframes → hand (default)

Author/edit in Aseprite — draw a master with the primitives + conformance helpers (quantize_palette, apply_shading, apply_outline), or import_image a sibling/approved master PNG and edit it. No credits. Record --source hand.

Keyframes → PixelLab (optional)

Async object flow via scripts/pixellab.mjs: create-object review pack → select-frames (masters); state --object <id> (children). Check balance first. Never a from-scratch text generation of a child.

Animation → Aseprite only

Every idle and transition authored frame-by-frame via mcp__plugin_pixel-plugin_aseprite__*, following the additive-overlay / two-keyframe erosion recipe. No PixelLab v3, no procedural Pillow frames, ever.

Pillow → review glue only

scripts/montage.py (G2/G4 review sheets) and scripts/gif.py (assemble the preview GIF from exported frames). Pillow never generates art or motion.

Pre-flight: prefer the CLI; load schemas first. Prefer scripts/pixellab.mjs over raw MCP calls — it wraps the whole async create → poll → download loop and keeps base64 frames out of the LLM (hand-emitted base64 corrupts). The Aseprite + PixelLab MCP tools are usually deferred; bulk-load with ToolSearch "aseprite" / ToolSearch "pixellab" or calls fail with InputValidationError.

The two inputs

You hand the pipeline two durable inputs. One fixes the look (reference assets → a style spec); the other declares the work (a three-file pipeline config). Swap the inputs and the same pipeline produces a different look / set.

1 · Reference assets → a style spec

A small set of hero exemplars + a locked palette + an art-direction note, distilled once into <assets>/_style-spec.json (canvas, ramps, light, outline, FPS). This is the cohesion anchor: every still and every animation is scored against it. For an existing game, the shipped tiles are the references.

2 · A three-file pipeline config

pipeline.json (the spec + state — global settings + hierarchical items[]), pipeline.history.json (the candidate/attempt-log sidecar), and pipeline.schema.json (the formal JSON Schema both data files are validated against). Idempotent: re-running diffs the spec against itself + disk and builds only the gaps.

Input 1 — the style spec

The look contract. Extracted at Stage 0 with Aseprite's analyze_reference / get_palette / analyze_palette_harmonies, then transcribed into a JSON the whole pipeline reads. It is the single thing every gate scores against.

Settings supersede the spec. The style spec's canvas is the game's native size (90×90); the actual tile generation size (32px) and FPS live in settings in pipeline.json, which override the style spec's canvas / animation.fps as the pipeline defaults. So the style spec defines the look ramps/light/outline; settings defines the output dimensions.

What references to supply (hero exemplars simple→detailed, a locked palette, ≥1 animation exemplar, an art-direction note, optionally an on-style / off-style pair) and how each spec field is extracted is the subject of references/reference-assets-spec.md.

Input 2 — the three-file pipeline config

The work is declared by three files that live side-by-side in godot/assets/tiles/v2/. They split the old single manifest into three concerns; all three are read, written, and validated through one shared seam, scripts/manifest.mjs, and every script refuses to proceed on data that fails the schema.

FileOwnsRoot shape
pipeline.jsonthe spec + current state — what to build and what's been chosen. The file humans/agents edit.{ settings, items[] }
pipeline.history.jsonthe candidate/attempt log sidecar — every seed ever tried (incl. failures), the full audit trail. Optional (absent → read as {}, degraded viewer mode).{ "<itemId>": { "<keyframeId>": [candidate…] } }
pipeline.schema.jsonthe formal definition — a JSON Schema (draft 2020-12) for both data files. The machine-readable source of truth.{ $defs: { pipelineDoc, historyDoc, … } }

The hierarchy is the derivation graph

Each items[] entry bundles one master keyframe, its children (variants derived from the approved master), and the animations over them. There is no master:true flag and no derivesFrom pointer — a child derives from the item's master purely by position. A child is only eligible to generate once its master is approved (selected !== null) and, for the PixelLab path, its objectId is recorded.

Idempotent, structural gap-fill

On each run the pipeline diffs pipeline.json against itself by shape (with candidate counts merged in from history) and against files on disk, then acts: an unapproved master below its candidate target → generate seeds; a child with no candidates whose master is approved → generate it from the master; an animation whose keyframes are approved and whose status is pending → animate it; any failed candidate → re-seed just that one. Already-approved keyframes and already-generated animations are skipped, so re-running is a cheap gap-fill, not a rebuild. (build_viewer.mjs --plan prints this action list as JSON.)

basePrompt & priors — cohesion at two levels

basePrompt

Optional item-level string prepended to each keyframe's prompt (basePrompt + ", " + prompt). Carries the shared description — subject, family, framing, shadow — so per-keyframe prompts only say what makes that variant distinct. Cohesion at the prompt level.

priors

Relative paths to already-shipped sibling assets, passed as visual context when gap-filling a master (PixelLab --style conditioning, or import_image for the hand path). New members inherit the family's silhouette and palette. Cohesion at the pixel level. (Children don't use priors — they derive from the master.)

Paths & on-disk layout

Every path (keyframe selectedPath, candidate path, animation gif, item priors, styleSpec) is relative to pipeline.json's own dir (godot/assets/tiles/v2/). New generation uses an items/<id>/<key>/NN.png layout; legacy birch art keeps its original sets/<set>/… paths. Both work — path is only a pointer.

Starting a run — how to actually kick it off

You don't hand-author the config. You start a Claude Code session in this repo and name the sprites you want — the sprite-pipeline skill interviews you, writes the config, and shows you a proposal you approve before anything is spent.

Just say what you want. Open Claude Code in the repo root and type something like:
"Use the sprite-pipeline skill: make me 3 new crop tiles — wheat, corn, and a pumpkin — matching the existing tree tiles, each with a gentle idle sway."
Naming the skill and listing the tiles is enough. The skill's intake front door takes over from there. (To grow something that already exists — "add a raining birch variant" — it skips intake and goes straight to gap-fill.)

What happens, step by step

  1. Interview. The skill asks only what it can't infer (one family or several? animated or static? which shipped tiles to match as priors? size / FPS / seed count / human-gate-vs-autonomous) — usually one batched question card.
  2. It writes pipeline.json. A new items[] entry (master + children + draft animations), each keyframe seeded as { id, prompt, selected: null, selectedPath: null }. First run also creates the global settings block. No candidates are authored — those fill in during the build.
  3. It starts the live viewer and builds the proposal. The pixelGen control server comes up and every requested asset shows as a pending placeholder card with its prompt/motion note. This is the proposal — no art, no credits yet.
  4. You review & approve. Tweak prompts or comment in the viewer until it reads right, then say "run it". Only then does the builder→critique loop start spending.
No spend until you approve. Intake produces the config + the proposal view only. Generation (PixelLab credits, if you chose that source) and animation (Aseprite ops) start after your "run it". The full interview + approval protocol is in .claude/skills/sprite-pipeline/references/intake.md.

The intake front door: a few questions become a config + an all-pending pixelGen proposal. Generation only starts after you approve — the same cards then fill with art.

Prerequisites

  • The skill is in this repo (.claude/skills/sprite-pipeline/) — Claude Code auto-discovers it, so just reference it by name.
  • PixelLab only if you want it. The pipeline runs fully on hand-authored Aseprite keyframes. If you do choose the PixelLab source, its MCP server must be connected and you'll spend credits (check with node .claude/skills/sprite-pipeline/scripts/pixellab.mjs balance).
  • Aseprite MCP (the pixel-plugin server) for animation, always.
  • A Godot 4.6 binary on PATH only for the separate npm run godot:update-tiles step after the pipeline finishes — not for the pipeline itself.

The live review viewer loop

The pixelGen viewer is a live control surface, not a static report. Built by scripts/build_viewer.mjs into pixelGen/ and served by the control server scripts/serve_viewer.mjs at http://localhost:8100/pixelGen/, it renders every set's keyframes, candidate seeds, idle GIFs, and transitions on one page — and accepts select / approve / regen / comment decisions that are written straight back into the three-file config.

Start the control server FIRST, at the top of a run — not at the end. Before any spend, launch node .claude/skills/sprite-pipeline/scripts/serve_viewer.mjs (background) and point at http://localhost:8100/pixelGen/. Its build_viewer.mjs --watch child re-emits data.json whenever pipeline.json changes, so as the run records progress the page updates live and you watch the family fill in. The viewer is a progress monitor — running it only at the end defeats the purpose. (Use the control server, not a plain http.server; that's just a snapshot.)

The loop closes through the live viewer: serve → eyeball → select/approve/comment → the control server patches pipeline.json (+ history) → gap-fill re-run fills exactly those items, and the page re-polls.

  • Renders every set — keyframe cards (grouped), candidate seed grids, idle players, and transition players (looping GIFs), with a status badge per asset.
  • Gaps are visible — anything declared in pipeline.json but not yet on disk shows a placeholder card with its id + intended prompt, so the backlog is the page (and the same surface is the intake proposal).
  • Decisions write through — on POST /api/<action> the control server validates and patches the three-file model in place, splitting each patch by what it owns: select/comment → pipeline.json; regen → pipeline.history.json; approve → both. Writes are atomic (temp + rename).
  • Headless equivalent — in an autonomous run the orchestrator uses scripts/pipeline-patch.mjs (record-candidate / approve / reject / animate-done / set-mode) to write the same split without the buttons.
Serve off the game port. The game's service worker hijacks the Vite preview, so the viewer is served on a non-game port. The pixelGen entry in .claude/launch.json runs the control server on :8100 (the page lives at http://localhost:8100/pixelGen/); pixelGen-static is a snapshot-only http.server on the same port, and docs-static serves :8099. The built pixelGen/ dir is a generated artifact — .gitignore covers it; don't commit it.

Generic vs per-project

The process is reusable on any game. Only the style spec, config, generated assets, and review decisions are per-game. The physics of motion is generic — leaves fall the same way in every game — so it ships in the skills, not the instance.

Executor discipline. Stage 2 keyframes = hand-authored Aseprite and/or optional PixelLab, one G2 pool. Stage 4 animation = Aseprite only. Pillow is review-glue only — it never produces art. The craft skills and templates are project-agnostic; copy the skill folders into another repo to carry the pipeline forward, then supply that game's references + config.

Where it lives

Three sibling skills carry the knowledge; the orchestrator calls the other two:

sprite-pipeline

The orchestrator spine — the 4-stage flow, the two inputs, tool routing, the 4-gate cost model, the cost-gated control flow, the builder→critique loop, and the viewer loop. .claude/skills/sprite-pipeline/

pixel-art-craft

The still-image rubric — palette, hue-shifted ramps, light/shading (no pillow-shading), anti-aliasing, outlines. Powers keyframe art and the G1/G2 critiques.

pixel-art-animation

The motion craft — the 12 principles, motion patterns, and the physics-of-motion.md direction layer. Owns the storyboard and the G3/G4 critiques.

Output lands in the game's tile tree: generated assets in godot/assets/tiles/v2/items/<id>/, the built loop in the v2 .tres slot that godot/scenes/Tile.gd resolves (animated SpriteFrames → static PNG → procedural color — newest wins). The locked look for this game is committed at godot/assets/tiles/v2/_style-spec.json.

Scripts quick-reference
ScriptWhat it does
scripts/manifest.mjsThe shared three-file seam every other script imports — load / validate / write the spec, history sidecar, and schema; loadMerged splices candidates back onto keyframes for the projection/plan code. Validate the on-disk docs, never the merged shape.
scripts/build_viewer.mjsReads + schema-validates both data files, emits pixelGen/data.json + copies the viewer/ template. --watch re-emits on change; --plan prints the structural gap-fill action list as JSON.
scripts/serve_viewer.mjsThe pixelGen control server — static-serves the viewer + asset tree, and on POST /api/<action> patches the three-file model in place. Spawns build_viewer.mjs --watch. Default port 8100.
scripts/pipeline-patch.mjsThree-file bookkeeping CLI (the headless counterpart to the viewer's buttons) — record-candidate / approve / reject / animate-done / set-mode / show. Use instead of hand-editing JSON in an autonomous run.
scripts/pixellab.mjsThe PixelLab keyframe client (Stage 2 only). balance; object flow create-object → select-frames → state. All image payloads are files — base64 never passes through the LLM. (An animate/v3 path exists as a clearly-marked escape hatch, not the pipeline path.)
scripts/pixels.mjsPNG opaque-pixel feature map — read a still's non-transparent pixels / diff two stills, so a storyboard can cite real coordinates.
scripts/montage.py · gif.pyPillow review glue only — montage stills/frames for G2/G4 (montage.py); assemble a preview GIF from exported frames (gif.py). Never generate art.
scripts/integrate.mjsThe Godot update engine (a separate step, exposed as tools/update-godot-tiles.mjs / npm run godot:update-tiles) — import → verify sidecars → assemble_tres.gd per idle → verify_sf.gd. --list dry-runs the work list.
scripts/assemble_tres.gdPack frames/<id>/NN.png into a v2 SpriteFrames .tres (one looping idle). Copied into godot/tools/ and run there.
