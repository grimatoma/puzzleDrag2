Sprite Pipeline

Generate and animate a cohesive family of pixel-art game sprites from two durable inputs — gap-filled, quality-gated, and integrated into the Godot v2 tile slot. Cheap critique reviews bracket the two operations that actually cost: generation and animation.

What this is

A reusable pipeline for producing a growable family of game sprites/tiles on a locked look. You hand it two durable inputs; it generates only what is missing and animates it, with four cheap critique gates bracketing the two expensive operations so a bad asset is caught on paper — not after you have spent generation credits and frame-by-frame effort.

TL;DR. Two inputs in (a style spec from reference art + a co-located set manifest), a cohesive animated tile family out. Stage 0–5 carry the work; gates G1–G4 keep cost down by rejecting off-style work before the spend. PixelLab generates stills, Aseprite is the only animation executor, Godot consumes the result.

The problem it solves

The goal is to mass-produce cohesive, professional pixel-art assets — tiles, items, characters — with consistent quality and a loop you can keep adding to. The craft pieces already existed, but they did not compose into a pipeline. Three gaps stood out:

No physics direction

Aseprite animates well when hand-directed frame by frame, but on its own it does not follow physics — you hand-describe all 16 frames of leaves falling or snow piling. There was no systematized motion-direction layer.

No cohesion anchor

Strong static-art and motion skills existed, but nothing forced every asset onto one palette, light direction, and silhouette. A set drifts without a single contract to score each member against.

No review surface

No way to eyeball a whole set at once — gaps included — comment on it, and route those notes back into the work. The backlog was invisible and feedback was hand-typed.

The fix is a pipeline that takes reference assets defining a look plus a co-located set manifest declaring a cohesive, growable group, generates only what is missing, brackets the expensive steps with cheap critique gates, and exposes a review viewer where you eyeball every set (gaps included) and copy comments back to seed the next pass.

The four stages

References distil into a style spec once (Stage 0). From there each run plans the set against what is already on disk (Stage 1), generates the missing base stills (Stage 2), storyboards the physics of each motion (Stage 3), and animates those storyboards (Stage 4). The pixel pipeline ends at the produced frames + preview GIF. Packing those frames into the Godot engine is a separate, on-demand step — npm run godot:update-tiles — run by hand, never as part of the pipeline run or the npm build.

Pipeline flow

References (the look) and the manifest (the work) feed Stage 0/1. Gates G1–G4 sit immediately around the two spend steps. A reject loops back to fix that one step; an accept advances. The pipeline ends at the produced frames + GIF; the viewer then surfaces the whole set, and copied comments edit the manifest to drive the next idempotent pass. Updating Godot (dashed, off the main flow) is the separate, on-demand npm run godot:update-tiles step.

The same flow is described as an order-of-operations handshake between an orchestrator, a cheap critic, the generator, the Aseprite builder, and the engine. The critic is consulted four times — once on each side of each spend — and never gates a whole set in one pass: each missing asset is scored on its own so per-asset rejects are not lost.

The cost model

Generation (Stage 2) and animation (Stage 4) are the costly steps — credits and frame-by-frame effort. A critique gate sits on each side of each, so a reject costs a prompt, not a build.

Why this shape. The two expensive operations are bracketed: G1 → generate → G2 and G3 → animate → G4. G2 and G3 straddle the generation→animation boundary; G1 and G4 are the outer cheap checks. A miss on a locked spec field (off-palette, wrong light) at G1/G2, or a "slide pretending to be motion" at G3/G4, is a reject — fix and re-run that step only.

One critic, four firing points. The cheap diamonds bracket each red spend box — a reject rewinds one step, never the whole set.

The four gates

One critique sub-agent, four firing points — each a cheap review immediately before or after an expensive operation. The rubric comes from the relevant craft skill: still quality from pixel-art-craft, motion soundness from pixel-art-animation.

The reject rule. A miss on a locked spec field (off-palette, wrong light) at G1/G2, or a slide pretending to be motion at G3/G4, is a reject — you fix and re-run that step only, never the whole set. The actual gate prompts live in assets/builder-prompt.md (the per-asset build instruction) and assets/critique-prompt.md (the scoring rubric).

The builder → critique loop (and how to batch)

Each gap asset is produced by a builder → critique pair:

  1. Builder does one step for one asset (generate this still / animate this storyboard) and reports what it produced.
  2. Critique scores that output against the relevant rubric (the gate above) and returns accept or reject + reasons.
  3. On reject, re-run the builder with the critique's notes; on accept, advance.
Batching a set = fan out one builder→critique pair per gap asset. Because each asset is independent (its only shared context is the style spec + priors, both on disk), the gaps can be worked in parallel — one sub-agent pair per missing keyframe/idle/transition — then reconciled. Run the gates per asset; don't batch a whole set through one gate and lose per-asset rejects.

Tool routing — what executes what

The hard rule: Aseprite is the only animation executor. Pillow is review glue and nothing else; it never makes art.

Base stills → PixelLab

Async AI generation (create → poll get_* → download; check credits first; do the object review→select step), or a hand-drawn/edited still. Never Pillow.

All animation → Aseprite

Via the mcp__plugin_pixel-plugin_aseprite__* tools, following the frame-assembly recipe. The only animation executor — no procedural Pillow frame generation, ever.

Pillow → review glue only

scripts/montage.py upscales a still and lays frames into a montage for the G2/G4 eyeball checks. That is its entire sanctioned role.

Godot → assemble & verify

scripts/assemble_tres.gd packs frame PNGs into the v2 SpriteFrames the engine loads as the idle loop.

The two inputs

You hand the pipeline two durable inputs. One fixes the look (reference assets → a style spec); the other declares the work (a co-located, idempotent set manifest). Swap the inputs and the same pipeline produces a different look / set.

1 · Reference assets → a style spec

A small set of hero exemplars + a locked palette + an art-direction note, distilled once into <assets>/_style-spec.json (canvas, ramps, light, outline, project FPS). This is the cohesion anchor: every still and every animation is scored against it. For an existing game, the shipped tiles are the references.

2 · A co-located set manifest

One manifest.json per set, living beside its output, declaring the group's keyframes + per-keyframe idles + transitions. It is idempotent: re-running diffs declared ids against files on disk and builds only the gaps, feeding existing siblings in as priors so new members stay continuous.

Input 1 — the style spec

The look contract. Extracted at Stage 0 with Aseprite's analyze_reference / get_palette / analyze_palette_harmonies, then transcribed into a JSON the whole pipeline reads. It is the single thing every gate scores against. Two values are project-wide and define how the whole family animates: animation.fps and animation.framesDefault.

Worked example — _style-spec.json (canvas · palette ramps · light · outline · animation) 
{

What references to supply (hero exemplars simple→detailed, a locked palette, ≥1 animation exemplar, an art-direction note, optionally an on-style / off-style pair) and how each spec field is extracted is the subject of references/reference-assets-spec.md.

Input 2 — the set manifest

A set is a cohesive, growable group of sprites — every season of one tree, all the fish, every variant of one crop. The manifest lives beside the output it describes at <assets>/sets/<set>/manifest.json and is the durable source of truth for that group.

Idempotent gap-fill

On every run the pipeline reads the declared keyframes/idles/transitions, diffs them against files present under the set directory, generates only the missing artifacts, and feeds the existing siblings in as priors so a new member inherits the set's silhouette, palette, and detail density. Re-running is therefore cheap and non-destructive — a gap-fill, not a rebuild. To deliberately regenerate one asset, delete its files and re-run.

basePrompt & priors — cohesion at two levels

basePrompt

Optional set-level string prepended to each keyframe's prompt (basePrompt + ", " + keyframe.prompt). Carries the shared description — subject, family, framing, shadow — so per-keyframe prompts only say what makes that variant distinct. Cohesion at the prompt level.

priors

Paths to already-generated sibling assets passed as visual context when gap-filling. The new member inherits the family's silhouette and palette rather than drifting. Cohesion at the pixel level.

Worked example — the birch set

This is the canonical manifest: the birch tree set at godot/assets/tiles/v2/sets/birch/manifest.json. It conforms to a style spec two directories up, uses two already-shipped sibling tiles three directories up as priors (the tiles sit in godot/assets/tiles/, one level above the v2/ tree), inherits the project FPS (fps: null), tags each keyframe with "group": "seasons" so the viewer buckets them, and declares four seasonal keyframes plus the idles and seasonal transitions between them.

    { "id": "tile_tree_birch_winter", "group": "seasons", "generator": "pixellab", "prompt": "bare branches, snow on limbs" }
  ],
  "idles": [
    { "for": "tile_tree_birch_autumn", "frames": 8, "motion": "sway + occasional falling leaf" },
    { "for": "tile_tree_birch_winter", "frames": 6, "motion": "bare-branch sway + snow glint" }
  ],
  "transitions": [
    { "from": "tile_tree_birch_summer", "to": "tile_tree_birch_autumn", "frames": 16,
      "physics": "chlorophyll fade->gold; first leaves loosen" },
    { "from": "tile_tree_birch_autumn", "to": "tile_tree_birch_winter", "frames": 20,

}

Reading it: only the missing seasonal keyframes are generated on a given run; each new one is conditioned on tile_tree_oak.png / tile_tree_fir.png plus the already-generated birch siblings. The autumn → winter transition is the richest brief — staggered leaf-fall at terminal velocity, snow accumulating bottom-up — and that physics string is exactly the motion plan the Aseprite animator executes.

Growable. Append a raining keyframe + idle and re-run — only the new entries generate, primed by the existing birch assets. The backlog grows by editing one JSON file. Full field reference: references/manifest-schema.md.

Per-set directory layout

Everything a set produces is written into the same directory, so the manifest and its outputs travel together:

<assets>/sets/<set>/
  manifest.json              # the durable source of truth
  keyframes/<id>.png         # one base still per keyframe (PixelLab output)
  storyboards/<id>.md        # per-asset shot notes / motion plan
  frames/<id>/NN.png         # exploded animation frames (00.png, 01.png, …)
  previews/<id>.gif          # a looping preview GIF per animated id
  <key>.tres                 # built SpriteFrames the engine loads (idle loop)

Generic vs per-project

The process is reusable on any game. Only the style spec, manifests, generated assets, and review comments are per-game. The physics of motion is generic — leaves fall the same way in every game — so it ships in the skills, not the instance.

Starting a run — list tiles → proposal → run

You don't hand-author the manifest. You start a Claude Code session and name the sprites you want ("5 new crop tiles: wheat, corn, pumpkin, …"); the sprite-pipeline skill interviews you, writes the config, and rebuilds the pixelGen viewer as a proposal you review before anything is spent. This is the intake front door — it sits before Stage 1.

The intake front door: a few questions become a manifest + an all-pending pixelGen proposal. Generation only starts after you approve — the same cards then fill with art.

No spend until you approve. Intake produces the manifest + the pixelGen proposal only — it never generates art. You review the proposal (placeholder cards with each tile's prompt/motion note), tweak prompts or comment, and re-build until it reads right. Only when you say "run it" does the builder→critique loop start spending PixelLab credits + Aseprite ops. The full interview + approval protocol lives in .claude/skills/sprite-pipeline/references/intake.md.

Already have a manifest that covers what you want? Skip intake — the run goes straight to Stage 1 gap-fill, and the proposal is just rebuilding pixelGen against the existing manifest (e.g. the committed birch set, whose nine assets all show as pending until its first run).

The review viewer loop

A static review viewer (built by scripts/build_viewer.mjs into pixelGen/, served at http://localhost:8100/pixelGen/) renders the set's keyframes, idle GIFs, and transitions on one page so you can eyeball cohesion across the whole family — the human-facing end of the G4 montage check. It is also the intake proposal surface (all-pending before a run; the same cards fill with art after), where the backlog lives and where corrections re-enter the pipeline.

The loop closes through the viewer: build → montage (G4) → eyeball → comment → copy → paste into Claude Code → idempotent re-run fills exactly those items.

  • Renders every set — keyframe cards (grouped, e.g. the four seasons), idle players, and transition players (looping GIFs), with a status badge per asset.
  • Gaps are visible — anything declared in a manifest but not yet on disk shows a placeholder card with its id + intended prompt, so the backlog is the page.
  • Comment anywhere — per-asset notes and free page-level comments, stored in localStorage.
  • Copy comments — one button concatenates all notes (each tagged with its asset id) into a paste-ready block and copies it to the clipboard. Paste it into Claude Code to drive the next pass; because the manifest is idempotent, it regenerates exactly those items.
Serve off the game port. The game's service worker hijacks the Vite preview, so the viewer is served on a non-game port — the pixelGen entry in .claude/launch.json runs http.server --directory godot/assets/tiles/v2 on :8100, so the page lives at http://localhost:8100/pixelGen/ (with docs-static on :8099). The built pixelGen/ dir is a generated artifact — .gitignore covers it; don't commit it.

Where it lives

Three sibling skills carry the knowledge; the orchestrator calls the other two:

sprite-pipeline

The orchestrator spine — the 5-stage flow, the two inputs, tool routing, the 4-gate cost model, the builder→critique loop, and the viewer loop. .claude/skills/sprite-pipeline/

pixel-art-craft

The still-image rubric — palette, hue-shifted ramps, light/shading (no pillow-shading), anti-aliasing, outlines. Powers keyframe art and the G1/G2 critiques.

pixel-art-animation

The motion craft — the 12 principles, motion patterns, and the new physics-of-motion.md direction layer. Owns the storyboard and the G3/G4 critiques.

Output lands in the game's tile tree: generated assets in godot/assets/tiles/v2/sets/<set>/, the built loop in the v2 .tres slot that godot/scenes/Tile.gd resolves (animated SpriteFrames → static PNG → procedural color — newest wins). The locked look for this game is committed at godot/assets/tiles/v2/_style-spec.json.
