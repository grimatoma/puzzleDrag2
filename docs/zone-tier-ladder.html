Zone Tier Ladder

Settlements grow through per-zone tiers — each rung unlocks more plots and more buildings, rendered as a hand-crafted Phaser tilemap. The home zone reaching City unlocks the Town 2 expedition.

Overview

Today a settlement has a fixed plotCount and a fixed building list. This feature turns each settlement into a growing town: the player upgrades their town centre through a ladder of tiers, and every rung grants more building plots and unlocks more building types. The ladder is authored per zone, so each zone has its own rung count, plot curve, cost curve, and unlock schedule.

TL;DR. Add a per-zone tiers[] ladder + a tier field on each settlement + a TIER_UP action. Each (zone, tier) renders a hand-authored tilemap through the existing Phaser TownScene. Tier-ups grant plots + building unlocks only — no turn/yield change. The home zone hitting its top rung (City) gates the Town 2 expedition.

Where this plugs in

The relevant machinery already exists; this feature mostly redirects data sources rather than building new subsystems.

  • Zones are nodes in src/features/cartography/data.ts, each already carrying its own buildings[], plotCount, entryCost, and board set.
  • The town view (src/ui/Town.tsx) reads plotCount, builds a TownPlan, and hands it to a Phaser scene. Plot count is the single knob that drives layout size (Town.tsx:168).
  • The renderer (src/ui/town/TownScene.ts) is already a Phaser.Scene on a 40×30 tile grid (32px tiles, public/town/tileset.png). It paints ground tiles + a depth-sorted object layer from the TownPlan.
  • Settlements already have per-zone runtime state (founding) in state.settlements[zoneId] — the natural home for a tier field.

Architecture & data flow

The only structural change to rendering is inverting the map source: instead of buildTownPlan generating the plan, the current tier indexes into an authored map registry. Everything downstream of TownPlan is unchanged.

Authored tilemap format

A hand-crafted map is a TownPlan in which the ground is an explicit tile grid (full hand control of terrain/roads/water) plus explicit object lists. TownScene gains one branch: when groundTiles is present, paint straight from it and skip the procedural paintGrass/paintRoads/paintWater/… passes; the object layer (lots, boards, props) is drawn from the authored lists exactly as today.

// src/ui/town/townMaps.ts  (new)
export interface AuthoredTownMap {
  // 30 rows × 40 cols of tileset indices (TILE=32, GRID 40×30 = 1280×960).
  // -1 = leave blank (transparent). Painted directly into the ground layer.
  groundTiles: number[][];
  // Interactive plots. index is STABLE across tiers (tier N adds higher indices).
  lots: { index: number; cx: number; cy: number; w: number; h: number }[];
  boards?: { kind: "farm" | "mine" | "fish"; cx: number; cy: number; w: number; h: number }[];
  props?:  { kind: string; x: number; y: number }[];
  plaza?:  { cx: number; cy: number; rx: number; ry: number };
  well?:   { cx: number; cy: number; r: number };
}

// Keyed by zoneId, indexed by tier. Missing entry → procedural fallback.
export const TOWN_MAPS: Record<string, AuthoredTownMap[]> = { /* … */ };
Why a tile grid, not a full procedural TownPlan. The user wants literal hand-tiling. An explicit groundTiles array is the most direct expression of that and reuses the existing tileset indices (T.GRASS, T.DIRT, T.WATER, decorative variants). The object lists stay coordinate-based because TownScene already places lots/boards/props by coordinate.
Stable lot indices = appended growth for free. Because lot index is preserved between tiers and the renderer maps stored buildings by lot index (Town.tsx:190), a building placed at Hamlet stays exactly put at City — the new tier just adds higher-indexed lots around it.

Tier model

Each cartography node gains an optional tiers?: ZoneTier[]. The array length is the rung count for that zone; nodes with no tiers (events, boss arenas) keep today's single fixed layout and show no upgrade control.

interface ZoneTier {
  id: string;                // "camp" | "hamlet" | "village" | … (zone-authored)
  name: string;              // display: "Mining Camp", "Boomtown", …
  plots: number;             // TOTAL plots at this rung (must match authored map lots)
  unlocks: BuildingId[];     // buildings newly buildable at this rung (cumulative)
  upgradeCost?: ResourceCost;// cost to REACH this rung; rung 0 has none (founding)
}
  • Buildings available at the current rung = union of unlocks for all rungs ≤ current. The node's existing buildings[] becomes the validated superset (a unit test asserts union(tiers.unlocks) === buildings), so every system that reads node.buildings keeps working.
  • Plots come from tiers[tier].plots, replacing the static plotCount read at Town.tsx:168. The authored map for that rung must expose exactly that many lots (test-enforced).
  • Cost reuses the existing resource-cost shape that FOUND_SETTLEMENT already validates and deducts.

State & action

One new field, one new action — both mirroring the proven founding flow.

State

state.settlements[zoneId].tier: number (default 0). Already-founded zones default to rung 0. Persisted with the rest of the settlement record.

Action TIER_UP

Payload { zoneId }. Look up tiers[tier+1].upgradeCost → verify the player can pay → deduct → tier++. No-op at max rung or if unaffordable.

As shipped: TIER_UP landed as a plain case "TIER_UP" in the root reducer (src/state.ts), right beside FOUND_SETTLEMENT — not via the slice system, so the slice-registration footgun never applied. Coins are deducted from the global pool; the resource cost is deducted from the zone inventory. At max rung or when unaffordable it returns state unchanged.

Upgrade UI & the Town 2 gate

Hearth upgrade control

The town-centre building (Hearth) gains an "Upgrade to {nextTier.name} — {cost}" control, reusing the existing FoundSettlement banner pattern. Disabled-with-reason when unaffordable; hidden at max rung.

City → Town 2

As shipped: generalised into a reusable node field requiresZoneTier: { zone, tier } (quarry uses { zone: "home", tier: 2 }). A zoneTierGateReason() helper drives a locked-with-reason node on the cartography map (src/features/cartography/index.tsx), mirroring the requiresHearthTokens pattern but data-driven for any future gate.

Per-zone ladders tier costs tuned

Tuned to a moderate run-budget. The upgrade-cost column has been re-tuned (down ~3–5× from the shipped first-pass) so the ladder fits a moderate pacing target — grow home → City in ~15–25 board runs, derived from the income-per-run model (~10 chains/run on the 6×6 board → a few of each staple resource + coins/run). Two structural changes back the costs: chain coin yield was raised ~2× (was floor(N·value/2), now floor(N·value)) so coins keep pace with the breadth of coin sinks, and the income knob is now decoupled (TILES_PER_RESOURCE vs UPGRADE_THRESHOLDS) so resource yield tunes without touching board-tier pacing. Plots and unlocks are unchanged; costs stay resource-led with a coin component sized to the doubled income. The wider sweep (building/tool/recipe/quest costs to the same budget) is the continuing work — see What's next.

Other nodes

Every other cartography node keeps a single fixed layout (no tiers → no upgrade control), rendering via the procedural fallback until/unless a ladder is authored for it later.

Building → tier philosophy

Early rungs

Survival & staples: food production (Mill, Granary, Bakery), first housing, the Inn. Gets a new settlement self-sufficient.

Mid rungs

Comfort & production: Brewery, Apiary, Workshop, Chapel, defensive Watchtower. The town becomes a real economy.

Late rungs

Industry & prestige: Forge, Portal, Observatory, ClockTower, advanced housing. Gates the high-end and the next town.

What's next

Primary next work — comprehensive economy balance pass. The tier-upgrade costs here are one knob in a larger economy (tools, buildings, recipes, founding, workers, boons, decorations, quests, dailies). The next effort tunes the whole economy to a run-budget model — deciding how many puzzle-board runs each milestone should take (moderate pacing, ~15–25 runs to grow home → City) and deriving every cost from income-per-run, so numbers are logical and story-aligned rather than picked at random. Grounded in the Puzzle Craft 2 reference.
  • Decouple the income knob — today a single constant (UPGRADE_THRESHOLDS) sets both the board-tier-upgrade rate and the tiles-per-resource income rate. Split out a dedicated tiles-per-resource constant so income tunes independently (a small, test-covered change in src/state.ts).
  • Account for board dynamics — the model must fold in per-season drop rates (seasonDrops) and how built buildings/workers/tiles shift drop pools and yields (pool_weight, bonus_yield, threshold_reduce, turn_budget_bonus).
  • Update the wikis — document the tuned economy + run-budget rationale in the Dev Panel wiki (src/balanceManager/content/pages/balance.html & direction.html); the infobox/concept grids auto-refresh from config.
  • Single source of truth — remove the empty src/config/balance.json override layer so the canonical configs are the only definition (preserving the shared /story/-editor sanitizers).
  • Re-baseline visual goldens — town DOM legitimately changed (upgrade banner + tier plot counts); needs a canonical host.
  • Town-map art polish — functional Claude-tiled grids now; a human / PixelLab visual pass later.
  • Extend ladders to more zones — only home + quarry are tiered today; other nodes stay single-tier until a ladder earns its keep.
  • Authoring tooling — a visual Town Map Editor (à la /story/) remains deferred; revisit if hand-authored maps multiply.

Appendix

Tileset index reference (from TownScene.ts)

The authored groundTiles use the same Tuxemon tileset indices the renderer already references (0-based, 24 cols, 32px, extruded margin=1 spacing=2):

GRASS        26          // pure flat grass (base fill)
GRASS_ALT    50,51,76,77,98,99   // subtle grass variants
GRASS_FLOWER 120,121      // poppy + clover (opaque)
DIRT         35           // clean tan path (roads / plaza / fields)
WATER        250          // deep solid water
// Multi-tile baked props: PINE, TREE2, FOUNTAIN, BUSH, SIGN, ROCK, ORE

Grid: GRID_COLS=40, GRID_ROWS=30, TILE=32 → design space 1280×960.

Note: graphify toolchain (historical)

A repo hook mandates graphify query/explain/path before reading source. When this design was written the graphify CLI wasn't on PATH, so it was grounded by reading source directly (the documented fallback). A local shim has since been installed, so the hook now functions.
