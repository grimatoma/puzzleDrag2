Progression Feed & Configurable Triggers

A config-driven trigger/effect engine — triggers become data-defined logical checks instead of a closed enum — and the wiki "Progression" timeline that simply reads it. One source of truth, two consumers, kept honest by status flags and parity tests.

Status: Draft for review · Updated 2026-06-03 · Scope: engine schema + wiki view (content authoring & story prose are out of scope)

This is an implementation spec, not the direction SSOT. For what the game is building, see the in-app wiki Direction page (/b/#/page/direction). This doc covers the trigger/effect engine + the generated Timeline view that renders it.
Trigger engine Progression view Wiki / Dev Panel Schema-first

Overview

The game already has a notion of "things that happen" — triggers — but only the story system uses them, and they live in a closed enum (StoryTriggerType) with a hand-maintained switch. Every other unlock (zones, buildings, tools, tiles, recipes) is gated by unrelated mechanisms (level checks, stationBuilt, discovery methods, costs).

This spec proposes two layers:

A · The engine

Replace the closed enum with configurable triggers: a trigger is a data object with a when predicate (composable all/any/not over a fact registry) plus effects. Flags become one kind of fact; gating, story, and progression all become "triggers + effects."

B · The view

A new "Progression" wiki tab that reads that same config and renders it as a reporter-style, two-tier live feed — each event and what it unlocked — ordered by the milestone spine, with zones as parallel tags. Status-flagged, deep-linked, CI-validated against live config so it cannot drift.

TL;DR. One config of { when, effects, classification, status } triggers becomes the single source of truth. The view reads it now; the engine migrates onto it incrementally behind parity tests, so we ship the timeline early without risking the story system.

Architecture: one config, two consumers

The pivotal idea: the trigger config the engine evaluates at runtime is the same config the view renders. They are reconciled by a status field (is this actually wired today?) and, during migration, by parity tests against the current behaviour.

Goals & non-goals

Goals

  • A data-defined trigger schema (facts + predicates + effects) that's open-ended — new triggers are config, not code.
  • A generated progression view in the wiki that reads it, honest by status flag, navigable, and drift-proof.
  • A parity-safe path to migrate the live story engine — and later the other gates — onto the new model.
  • Make the current half-wired gaps (dead mine_unlocked flag, UI-only token gates, coop→chickens) visible and explicitly planned.

Non-goals (here)

  • Story content authoring — the dialogue/beats themselves (your stage two).
  • An in-game chronicle/feed for the player's actual run (separate feature; the existing #/chronicle view is untouched).
  • Rebalancing values — the view enables balancing; it doesn't change numbers.
  • A big-bang rewrite — the migration is incremental and reversible until parity is green.

Layer A · The configurable-trigger engine

Today conditionMatches (src/story.ts:441) is a switch over 14 enum variants, reading from three places: the current event, an inventory totals snapshot, and flags. The redesign generalises exactly those three inputs into a fact registry and makes the check data.

Facts

A fact is a named getter over game state (plus the current event context). Facts are parameterised families — resource.<key>.total, building.<id>.built, flag.<id> — resolved by a small matcher. This is the whole vocabulary of "things that happen."

State facts vs event context. Most facts are state (queryable any time). A few current cases are genuinely edge-triggered (craft_made, order_fulfilled, session_start). We support both: predicates read state facts plus an optional event context for the tick — a faithful generalisation of today's conditionMatches(t, event, totals, flags) signature. Where it's cheap, we add a cumulative counter (e.g. craft.<key>.count) so the check becomes pure state and survives reloads.

The logical check (when)

A composable predicate tree. Leaves read a fact and compare; branches combine. This is the "actually configurable trigger with a logical check" from the brief.

type Op = "eq"|"ne"|"gte"|"lte"|"gt"|"lt"|"truthy";

// A leaf reads one fact; op defaults to "truthy" (so { fact } means "is set").
type Leaf = { fact: string; op?: Op; value?: JsonValue };

type Cond =
  | Leaf
  | { all: Cond[] }      // AND
  | { any: Cond[] }      // OR
  | { not: Cond };       // NOT

// Example — "the Mine opens":
const when: Cond = { all: [
  { fact: "resource.supplies.total", op: "gte", value: 3 },
  { any: [
    { fact: "building.pit_props.built" },
    { fact: "level", op: "gte", value: 2 }   // the reserved soft-gate
  ]}
]};

The evaluator is a ~30-line pure function evaluate(cond, ctx) → boolean with no game knowledge beyond the fact registry — trivially unit-testable.

Effects

When a trigger fires it applies effects. This generalises today's two overlapping shapes — BeatSideEffects (src/story.ts:65) and ChoiceOutcome (:74) — into one discriminated list, and adds the gating unlocks so availability is an effect too.

type Effect =
  | { kind: "setFlag";       flag: string | string[] }
  | { kind: "clearFlag";     flag: string | string[] }
  | { kind: "unlockZone";    zone: ZoneId }
  | { kind: "unlockBuilding";building: BuildingId }
  | { kind: "discoverTile";  tile: TileKey }
  | { kind: "grant";         resources?: Inventory; coins?: number }
  | { kind: "advanceAct";    to: number }
  | { kind: "spawnNpc";      npc: NpcId }
  | { kind: "spawnBoss";     boss: BossId }
  | { kind: "bondDelta";     npc: NpcId; amount: number }
  | { kind: "showBeat";      beat: StoryBeatId };  // narrative attaches here
Story becomes one effect kind. "Trigger fires → show a story beat" is just { kind:"showBeat" }. Narrative stops being a separate engine and becomes content hung off the same triggers — which is exactly the seam your stage-two story work needs.

The trigger object

One object carries the runtime check, its effects, and the classification the view reads. Authoring is mostly this last group.

interface ProgTrigger {
  id: string;
  label?: string;
  when: Cond;                 // the configurable logical check
  effects?: Effect[];
  once?: boolean;            // default true: fires once when `when` first holds
  repeat?: { cooldown: number };

  // ── classification the view reads ──
  milestone?: boolean;       // part of the ordered spine
  requires?: string[];        // prior trigger ids → encodes spine order as DATA
  zone?: ZoneId;             // parallel-track tag (non-linear)
  phase?: string;            // derived from spine, or explicit
  status?: WikiStatus;       // WIRED | PARTIAL | PLANNED | STUB | DOC-ONLY
  blurb?: string;            // reporter flavour / future story hook
}
Ordering as data. The main arc's sequence today is enforced imperatively by nextPendingBeat (src/story.ts:407). Here it's just requires: [prevId] inside when — order emerges from predicates. Zones carry no order at all; they're a zone tag, honouring "zones aren't linear."

Evaluation & edge-firing

Replace the ~8 scattered evaluateAndApplyStoryBeat calls (state.ts:575, 878, 883, 1001, 1007, 1014, 1029; keeperTrials.ts:184) with one evaluateTriggers(state, eventCtx) step. A non-repeat trigger fires once when its when first holds — tracked by a persisted fired set (the principled version of today's _fired_<beatId> flags).

Cascading (an effect makes another trigger's when true in the same tick) is handled by iterating to a fixpoint, matching today's beat→flag→beat cascade in storyEffects.ts. To keep it cheap, triggers are indexed by the facts/events they depend on, so a tick only re-checks the affected few.

On "remove flags". Layer B · The progression feed

A new top-level "Progression" tab in the wiki (the /b/ Dev Panel + Game Wiki). It renders the trigger config as the reporter-style feed validated in the brainstorm mockups: a vertical timeline whose spine is the milestone triggers (ordered via requires), with each non-milestone trigger bucketed by milestone-phase and grouped by category under it (two-tier), and zone as a colour/tag + filter rather than an ordering.

Tier 1 — milestones

Full "event" cards: arrivals, zone foundings, key story beats. Ordered top-to-bottom. The only sequenced layer.

Tier 2 — unlocks

Grouped ➜ Unlocked rows (tiles / buildings / tools / recipes / workers / effects) — collapsible, not individually sequenced.

Facets

Zone tag + filter (non-linear), category filter, and a WIRED/PARTIAL/PLANNED status chip per row.

Visual reference: the two interactive mockups shared in the brainstorm (livefeed + the layered/lanes comparison).

Where the feed's data comes from

The feed reads the trigger config and enriches terse effects by deriving detail from the same maps the wiki already imports (concepts.ts:13-28). A founding trigger says unlockZone: quarry; the view expands that into the zone's tiles/resources/buildings/workers by reading ZONES + BIOMES — so authoring stays small and the detail can't go stale.

Feed needsDerives from
A zone's unlock bundleZONES, BIOMES, BUILDINGS (filtered by biome), TYPE_WORKERS
A building's effectBUILDINGS[].abilities, ABILITIES
A recipe's gateRECIPES[].station
A tile's discoveryTILE_DISCOVERY_METHODS
Authoring burden is small. Most events are derived from existing config; the authored layer is classification (milestone/zone/requires), a status flag where code-truth isn't obvious, and a handful of PLANNED entries that don't exist in code yet (coop, chickens).

Anti-drift validation

A unit test (the honesty guarantee) asserts:

  • every effect's referenced key (zone/building/tile/resource/recipe) resolves in the live maps — or the trigger is explicitly PLANNED;
  • every when leaf references a known fact family;
  • the requires graph is acyclic (the spine is a DAG);
  • every zone tag is a real ZoneId.
Rename a building or ship the coop, and the feed either updates or fails CI — it can't silently lie. This is the payoff of reading config rather than hand-writing prose.

Parity-safe migration

  1. Build the new evaluator beside the old conditionMatches — nothing wired yet.
  2. Translate every existing beat trigger to when form (the table in Engine → Subsuming is the spec).
  3. Parity test: for sampled (state, event) pairs across all beats, assert conditionMatches(old) === evaluate(translate(old)). Fuzz a little.
  4. Flip the runtime to the new evaluator; keep the old as dead code until parity is green in CI.
  5. Delete StoryTriggerType + the switch; demote flags. Repeat the same beside-then-flip dance per non-story gate in Phase 3.
Touchpoints beyond the reducer. sanitizeTrigger (config/applyOverrides.ts:593), the Story Tree Editor (/story/) trigger picker, and balance.json presentation patches all reference the old trigger vocabulary. The redesign must update the editor's picker to author when trees. Budget for this — it's the least-fun part.

Save schema & persistence

Persisted story state today: state.story.{act, flags, choiceLog, repeatCooldowns} (persistence.ts:34). The new model adds a fired set and optional cumulative counters (craft.*.count, order.fulfilled). Bump SAVE_SCHEMA_VERSION 43→44 (constants.ts:244).

Per repo convention, forward migrations aren't maintained — a version bump simply discards old saves and the player starts fresh. That's acceptable here and removes migration complexity.

Scope boundaries

In: the engine schema (facts/conditions/effects/triggers), the wiki feed reading it, the initial config (full spine + early-game fine detail + a few PLANNED), anti-drift validation, and the parity-safe story-engine migration.

Deferred but explicitly planned by this work: wiring the missing mechanics (coop→chickens, resource-gated zones, the three dead gates) — each is a PLANNED row that turns WIRED as it ships. Out entirely: story content authoring (stage two) and an in-game player chronicle.
