Reshaping the Board

Three directions for the puzzle's core surface — variable board size & shape, parallel boards, and a board-of-boards. What each could be, the decisions that define them, and where to start.

Status: Exploration · 2026-06-03 · not a commitment — a map of the option space

board design variety difficulty progression unconstrained by today's code

Overview

The board has always been a fixed 6×6 grid. This doc explores treating that grid as a design surface instead of a constant — three concepts that each change the shape of play in a different way.

This is a vision study, not a build plan. It deliberately reasons about what these concepts should be, and treats the current code as a launch pad we'll extend freely — not as a set of limits to design around. Where the engine already bends, that's a head start; where it's pinned to the number six, that's just future work, noted neutrally.

TL;DR. All three are open to us. The engine already flexes further than it looks (it sizes and centers the board from the live grid shape). The smallest, most tactile win is variable board size — and it's the shared key that quietly unlocks both parallel boards and the meta-board.

The shared key. Every one of these wants the board's dimensions to be data rather than a hard-coded constant. Decouple size once and parallel boards and the meta-board both inherit it — so the three aren't really independent choices, they're a sequence with a common first step.

The board today — what we build on

One question — "would changing the board break the game?" — is really two, and they have different answers.

Will the code cope?

Mostly, yes — already. The renderer derives tile size from the viewport and auto-centers the board every layout, and several systems read the live grid shape rather than the constant. The parts still pinned to "6" are mechanical to generalize.

Will the balance hold?

Not on its own — but that's a knob we set, not a wall. Reward scales with board area, so a bigger board pays out more and a smaller one less. The fix is to let size drive the balance, which we design in (next section).

Make the two numbers data-driven and the bottom lane follows mechanically; the top lane already flexes. That's why size is the smallest of the three changes — and the common groundwork the other two build on.

Where the single board lives in state (for the parallel / meta concepts)

Today the game holds exactly one active board: state.grid is a single Tile[][] (src/types/state.ts). Zones share that one board and swap in/out via per-biome saved snapshots (savedField); the React→Phaser registry bridge (src/prototype.tsx) and the CHAIN_COLLECTED action both assume there's one grid. None of that is a barrier — it's simply the shape we'd generalize into "the board(s) currently in play" for concepts 2 and 3.

Balance is an input, not an assumption

The one real coupling to "36 cells" is the economy — and it's worth understanding precisely, because it turns the scary version of the question ("won't a bigger board break everything?") into a design knob.

Upgrades come from chain length against a per-resource threshold:

upgrades = floor(chainLength / threshold)   // utils.ts · UPGRADE_THRESHOLDS

More cells mean disproportionately longer possible chains, so upgrades-per-move rise faster than area. The team already tunes to this — the tile-pool weighting even carries the comment "so the board doesn't become chain-impossible at 6×6 = 36 cells." So the relationship is known; it's just currently expressed as a fixed assumption.

Design principle. Treat board area as an input to the balance knobs. Derive per-resource thresholds and per-zone turn budgets from cell count, so "upgrades per turn" stays roughly constant while the feel of the board changes. Then size becomes a clean dial — bigger feels generous, smaller feels tense — without inflating or starving the economy. Shape (dead cells / holes) is a softer lever: it changes how chains route far more than how much they pay.

1 · Variable board size & shape

Serves variety · difficulty

Untether the grid from 6×6 and the board itself becomes part of a zone's identity — each place hands you a different puzzle to read. This is the most direct, most felt change, and the one that lays groundwork for everything else.

Different size per zone

The meadow is a roomy 7×7; a cramped cavern is 5×6; a harbor is wide and shallow. Size becomes a signature, not a setting.

Non-square & irregular

Rectangles, L-shapes, carved holes, themed silhouettes — a pond-shaped fish board, a terraced hillside. Dead cells reroute chains and create natural choke points.

Size as a difficulty dial

Tighter boards mean fewer options and harder reads. A boss can shrink the board for a phase; an expedition tier can carve cells away.

Boards that grow

A run opens on a small board and expands as you progress — the space literally unfolds, pairing nicely with seasonal pacing.

The defining decision. How far should balance auto-scale with area? Two honest options: (a) derive thresholds & turn budgets from cell count so any size is automatically fair (great for irregular/procedural boards), or (b) author balance per named board for hand-crafted pacing. A hybrid — auto-scale as the default, hand-tune the showcase boards — is likely the sweet spot.
What we'd touch

Promote COLS/ROWS from constants to per-board data; have the handful of GameScene loops and the initial-grid builder read those values; the rest of the renderer already adapts. Irregular shapes add one idea — a per-cell "playable" mask — that the collapse and chain code would honor. Neutral, contained, and the foundation the next two reuse.

2 · Parallel boards

Serves variety · difficulty

Instead of one board at a time, run two (or more) side by side. The game stops being "solve this board" and becomes "where do I spend my attention?" — a fundamentally different pressure.

Split economies

Each board feeds its own resources — farm beside mine — and you keep two ledgers balanced at once.

Shared clock, separate boards

One turn budget spent across both. Every move on the left is a move you didn't make on the right — pure opportunity cost.

Chains across the seam

Let adjacency bridge the boards' shared edge so a chain can leap between them — high spectacle, big design swing. Or keep them strictly isolated for a cleaner read.

Draft & rotate

More boards than you can tend; you rotate focus while idle boards drift — hazards creep, tides turn — so neglect has a cost.

The two decisions that set the feel. (1) Do chains cross boards? Isolated is dramatically cleaner to read and reason about; crossing is spectacle but reshapes adjacency, collapse, and where upgrades land. (2) Shared or per-board turn budget? Shared = constant triage and tension; per-board = two calmer puzzles in parallel. These two answers, more than anything technical, decide whether this is a relaxing double-feature or a white-knuckle juggle.
What we'd touch

Generalize the single state.grid into a small collection of boards, and tag each move (the CHAIN_COLLECTED path and the registry bridge) with which board it hit; give each board its own turn tracking. Rendering is either multiple Phaser scenes or one scene drawing several grids. All natural extensions of today's shape — not blockers.

3 · Meta-board — a board of boards

Serves progression · variety

A board made of boards. Individual runs stop being the whole game and become moves in a larger one. This is the strongest path to a true progression layer — a game above the game.

There's one fork that defines everything else, so it's worth deciding first:

A · Aggregative incremental

You finish small boards and their outcomes place tiles on a higher-level board. Clear the farm plot and it drops a "harvest" tile onto your estate grid; line those up for estate-wide upgrades and seasonal meta-objectives.

Fits how zones already snapshot and persist — a meta-layer above runs, buildable a piece at a time. The runs you have stay the runs you have; the meta-board is new strategic connective tissue.

B · Simultaneous spectacle

A literal grid of small boards you play as one surface — chains may span sub-board borders, and a collapse can ripple across the whole composite.

The most spectacular version and the biggest swing: it reworks input hit-testing, the collapse pipeline, and where upgrades land. A "puzzle of puzzles" rather than a strategic map.

The defining decision. Aggregative vs. simultaneous is the whole ballgame — a strategic layer over runs (A) vs. a single giant nested puzzle (B). They share almost no implementation, so this is the one choice to lock before anything else. For the "new progression layer" goal specifically, A is the more natural fit and the lower-risk start; B is the headline-grabbing stretch.
What we'd touch

A new meta-layer of state sitting above the per-board grid. The aggregative reading mostly reuses the existing snapshot/persistence model and adds a higher-level grid that consumes run results. The simultaneous reading additionally reworks input, collapse, and upgrade placement to span sub-boards. Both presume concept 1's data-driven dimensions underneath.

Where I'd start

Prototype variable board size first. It's the smallest, most tactile change, it instantly demonstrates both variety and a difficulty dial, and — critically — it's the shared groundwork the other two are built on. You get a visible result quickly and de-risk everything downstream.

From there the path forks with your goal:

Want pressure?

Build toward parallel boards — a second live board on the same clock is the most direct way to raise the ceiling on engagement.

Want a long arc?

Build toward the aggregative meta-board — a strategic estate grid that runs feed into, layered on the snapshot model you already have.

Want spectacle?

The simultaneous meta-board is the showpiece — worth a contained spike once size is data-driven, to see how cross-board chains actually feel.

Sizing them is a separate exercise once a direction is picked; this study exists to make that pick well-informed, not to commit to it. Nothing here is gated by today's code — only by which experience we want to chase first.
