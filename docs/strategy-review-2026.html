The one-paragraph version

A game that is already authored, not generic

The core drag-to-chain loop feels good, the craft and quest surfaces are genuinely well-made, and there's a real warm parchment/ink design language with a zero-config seasonal-art pipeline that is quietly a superpower. What's missing is a silhouette: a generic title and premise, a town that doesn't move, and a seasonal-art system that's 96% unused. Fix three things and it crosses from "nice" to "memorable" — and none of them touch your balance.

Would I download this? — Yes, and keep it. But I wouldn't tell a friend yet.

It already looks better than most cozy-puzzle shovelware. The gap is identity and aliveness, both fixable with machinery you've already built: (1) make the town breathe, (2) make the seasons visible, (3) give it a story silhouette.

Your lead question

Pixel art vs SVG — the size argument runs backwards

The instinct is "SVG is tiny, pixel art is heavy — going pixel will bloat the game." The measurements say that's not the real tradeoff, and the real win isn't size at all — it's that the town is currently dead.

DimensionProcedural / SVG (today)Pixel art (the seasonal pipeline)
Building art code~4,700 LOC → ~46 KB gzipnear-zero code (drop a folder)
Share of JS bundle~3% of Phaser (Phaser = 6.9 MB raw / 855 KB gzip)static assets, off the JS critical path
All pixel assets today—3.9 MB total (77 tiles + town)
"Feels alive" animationbaked away CSS anims are lost when the SVG is rasterized to a town texturenative spritesheets — what the pipeline already makes

The real finding: the town doesn't move, and that's a wiring gap — not an art gap.

SMOKE_BUILDINGS is defined in config.ts and wired to nothing — TownScene has zero tweens, zero smoke. The SVG buildings are rasterized to a flat static texture (bakeSvgTexture), so their flicker/wiggle is destroyed. Meanwhile MapScene is fully alive (rising smoke puffs, ember-glow, halo pulses). Your "little things doing nice animations" instinct is exactly right — the map already does it; the town just never got wired up.

Recommendation — hybrid, in this order

  1. Wake the town with procedural ambient (zero new art). Wire SMOKE_BUILDINGS → a makeSmokeColumn-style emitter copied from MapScene.
  2. Pixel-art ~6 hero buildings (hearth, mill, bakery, forge, inn, home-tiers) as static PixelLab sprites dropped into the existing texture slot.
  3. Add 4–8-frame idle spritesheets for those heroes (chimney smoke, forge glow, mill sails) using the board's strip format — this is where pixel art's advantage actually lands.
  4. Keep SVG for the building-picker DOM preview (vector animates for free there), all HUD/menu icons, and any not-yet-arted building (perfect 46 KB placeholders).
  5. Leave the map procedural — it already reads as intentional and animated; reskinning 12 nodes is low ROI.

Don't pixel-art all 30 buildings × 4 seasons: towns don't change with seasons, and that's weeks of generation for no payoff.

Day/night vs seasons

Two clocks, and the contrast is a feature

The board already compresses a full 4-season arc into one run (a "year in a harvest"). The town/map are the persistent now. Asking whether they conflict is the right question — and the answer is they shouldn't, because they should run on different axes of time.

Give the town/map a slow day/night; keep the board's per-run season cycle. Day/night is cheap (a tint overlay + lamp glows at dusk — MapScene's ember-glow is already 80% of it). The only way to create dissonance is to make both show "season" at once, so don't: town = time of day, board = time of year. No conflict, and the rhythm difference is a selling point.

New visual systems

Where new visuals would pay off

  • Ambient world animation — smoke, forge glow, mill sails, the odd critter, swaying laundry. The single biggest "feels alive" lever; the map proves the pattern works.
  • Day/night lighting on town/map — a cheap tint + lamp bloom at dusk (see the two-clocks section); pairs with the existing docs/projects/18-town-2d-lighting.md brief.
  • The seasonal lifecycle, made visible — the transition animations exist for 3 tiles; surfacing the seedling→ripe→rotten arc ties directly into the Ripeness system (Systems tab) so the art and the mechanic reinforce each other.
  • Weather flourishes — a rain/snow overlay keyed to the board season, reusing the season-transition cinematic brief (17-season-flip-cinematic.md).

The rule that keeps balance safe

What "breaking the balance" means

The economy is rate-limited as coins/turn and resources/turn (each board action spends a turn; runs are turn-budgeted; chains pay floor(N × value) coins and roll capped resources). The bright line:

A system is balance-safe if it doesn't raise coins-per-turn or resources-per-turn above the current curve.

The cleanest safe designs spend the same turn differently (skill expression, not extra yield), or pay out in a parallel currency (embers / coreIngots / runes) that already exists and feeds only the boons / meta layer — never coins or capped resources. Anything that routes through a tunable constant can be re-balanced by the existing AI playtest harness.

Ranked new systems

Add depth without touching the curve

1 · Board-altering boons SHIP FIRST

The boons system already has a typed effect registry + reducer hook, but every shipped boon is economy-only. Add board-shape effects (min_chain_relax, wildcard_tile_chance, cross_collect_radius) bought with embers/coreIngots.

Safe because: parallel currency, balance-isolated by design; infra half-built (brief #04). No schema bump.

2 · Ripeness as a board modifier SHIP FIRST

Seedling→ripe→rotten timing, switched on per-season or as a boss modifier, paying in rune progress.

Safe because: changes which turn pays best, not how much; rot is a net income brake. Maps onto seasons + the seasonal art.

3 · Cross-Pollinate hybrids DELIGHT

Routing to hug another family grafts ✦ hybrids — but credit a capped parallel "rare bloom" resource at a harness-tuned value, not the prototype's ×3 coin spike.

Safe because: the whole thing sits on one knob (value + cap) the harness already audits.

4 · Streak / combo meter FEEL

Consecutive non-dud chains build a visible meter that feeds rune/almanac XP and a flourish — never coins. Pure feel + parallel progression.

5 · Enclosure reaping SET-PIECE

Loop Harvest's fence-and-reap, quarantined to rare "Festival Field" boards so the quadratic bonus is contained and rate-tunable.

6 · Depth stacks (mine only) BIOME

Deep Layers in the mine biome: richer ore traded against flood/gas hazards that lock cells. Net income is risk-gated, not inflated.

Is the story compelling?

Thin-to-generic, with two good bones

Verdict: don't keep it as-is — but it's a 1–2 day reskin. The render path is 100% data-driven; only ~6 of 24 beats hard-reference a resource key; the fiction is a few hundred lines of strings across ~6 files (no schema, no actions, no reducers).

Keep these

  • The "banked, not snuffed" hook — your rising smoke wakes dormant sister-holds. Gives map expansion emotional logic most farm games lack.
  • The silent lantern-bearing Hollow Folk + "what is named remains, what is forgotten is reclaimed" — a distinctive, dignified register.
  • Mira's Letter — genuinely well-written; proves the branching system works end to end.

What's weak

  • "Hearthwood Vale / relight the ancestral hearth" is maximally generic cozy-fantasy — Stardew/Fae Farm/Sun Haven all own it. Zero differentiation at the title.
  • The marketed keeper Coexist/Drive-Out choice is vaporware (parked + dead Frostmaw code). You're advertising depth you don't ship.
  • The melancholy/elegiac prose fights the candy-bright board art the project keeps drifting toward.

Four directions, grounded → cookie-crazy

Pivot options

A · "Emberwake" — sharpen, don't pivot

Same world, rename it, cut the melancholy ~30%, and actually ship the keeper fork. Lowest cost; doesn't solve "no silhouette."

B · "Hollow & Harvest" MY PICK

Lean all the way into the Hollow Folk: chaining = "name and keep," reclaiming tiles from a slow tide of forgetting; the town is a visible bulwark of kept names. Turns the existing-but-vestigial machinery (named settlements, choice-log, Charter) into the actual point.

C · "Spirits in the Pantry" — cozy-whimsical

Cranky household gods you coax back with snacks; each keeper is a spirit you befriend or evict. Best art-fit for the bright board direction; broadly marketable; loses the one classy note.

D · "Cinder & Cosmos" — cookie-crazy

The hearth is a dying star-seed; each sister-hold is a different physics (tidal, frozen, clockwork). Retro-justifies the wildly-varied zone atlas the team already wants. Highest ceiling, biggest tonal gamble.

Recommendation — ship Option B.

It's the highest-distinctiveness option that's also the smallest conceptual leap (it's already in the bible, just buried as flavor). It fixes the honesty problem — the fiction and the systems finally agree — and it differentiates hard at the title. Same ~6 string files, no schema/action/reducer changes; the name_settlement prompt + choice-log are pre-built hooks. Fall back to C for maximum market-safety; reserve D for if the per-biome-physics zone atlas becomes the headline feature.

Where UX fails

The recurring breakdowns

From the June UI review (full ranked ledger in ui-review-20260618/): the design system is strong but discipline slips in predictable ways.

  • Token drift — ~150 ad-hoc color utilities + raw hex fork the design system (heaviest on board chrome, Settings/Menu, cartography).
  • Sub-44px hit targets on the project's own --tap-min controls; sub-12px noun type against its own ≥12px floor.
  • The board tool rack reads as a wall of gray; two town banners overlap top-center; quest cards float in a narrow column at desktop (~60% empty width).
  • Inventory grid shows no item names (icon-only — bad on touch); Sell/Buy are tiny ghost pills.
  • Localized generic-AI tells — off-palette Settings/Menu + purple/blue cartography buttons (not the sanctioned arcane violet).

Never monetize coins/embers/gems/yield/thresholds/run-length. No energy timers, no loot boxes, no FOMO countdowns on the daily streak. Every paid item is one-time or fixed-content — no gacha, no consumables.

Efficiency

Runtime & build wins

DONE Top runtime win — the per-dispatch re-aggregation is fixed.

The inventoryCap effect in prototype.tsx ran currentCap() → computeAggregatedAbilities (loops all workers + buildings + discovered tiles) on every action — including every chain tile during a drag — because its effect dep was the whole gameState. Its dep array is now narrowed to [built, mapCurrent, workers, tileCollection.discovered, tileCollection.activeByCategory] — the exact inputs currentCap reads, so behavior is unchanged. Still open: scope the other ~31 whole-object registry-effect deps the same way.

  • Build ships un-minified JS DONE — vite.config.js had minify:false + sourcemap:true → 11.8 MB raw JS. Now gated on the Vite command: minify on + sourcemaps off for prod (vite build), while dev (vite serve) stays unminified so the Dev-Panel Animations-Demo bridge and other dev tooling remain readable. ~2–3× smaller raw JS and faster mobile parse.
  • 3 live WebGL contexts on the board+town path — well-managed (sleep/wake), no crash risk, flagged only for mobile memory awareness.
  • Texture churn & reducer cloning are fine — guarded re-bake throttled to 20 fps on a 6×6 board; spread-based reducer, no deep clones.

Test coverage

Deep on logic, shallow on clicks

Good enough to refactor reducer/economy/save safely; not good enough for the UI-click or Phaser-input layer. Unit tier is deep (3,701 tests, every slice + save migrations covered). But 19 of 20 e2e specs dispatch through a React-fiber bridge instead of clicking — so a button that stops calling its handler passes the whole suite. Goldens are mobile-thin (no iphone-landscape baselines; daily-streak has zero browser coverage and is suppressed in the seed).

Add — highest value first

  1. town-build-flow.spec — enter farm → chain → close season → click Build → place → assert built grew & debited.
  2. daily-streak.spec — boot without suppression, click claim, assert reward + date advance. (A launch-time blocker with zero coverage.)
  3. found-zone.spec — travel → click Found → biome pick → TIER_UP. The core progression loop has no e2e.
  4. orders-fulfill-ui.spec & run-summary-to-spend.spec — guard the HUD/economy display seam.
  5. Goldens: a daily-streak-modal scenario + re-baseline at iphone-landscape (on Linux — not regenerable on this Windows host).
